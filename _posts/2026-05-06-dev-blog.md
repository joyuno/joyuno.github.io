오늘은 포트폴리오 리팩토링부터 사주 앱 비즈니스 검토, 트레이딩 봇 디버깅까지 여러 프로젝트를 오가며 Claude를 활용한 하루였다. 단순 코드 생성보다는 세션 컨텍스트를 미리 쌓아두거나 반복 지시를 피하는 프롬프팅 전략이 효과를 봤다.

## claude.md로 프로젝트 컨텍스트 미리 쌓기
포트폴리오 프로젝트에서 각 프로젝트별로 강조할 내용과 경로를 요약한 md 파일들을 만들고, 이를 claude.md가 참조하도록 세팅했다. 취준생 포지션에 맞춰 소개글과 직무를 동적으로 변형해달라고 요청하면서 "현재 업무에 맞게 소개글 변형 + 직무 적합성 제안"처럼 구체적 태스크를 나열하니, AboutSection.tsx나 ProjectsSection.tsx 같은 컴포넌트 수정이 한 번에 잘 맞아떨어졌다. 이전에는 세션마다 프로젝트 배경을 재설명하느라 컨텍스트가 새나갔는데, claude.md를 베이스 디렉토리에 박아두니 응답이 2배 일관성 있게 나왔다. PDF UI/UX 기획안 위치도 명시적으로 주니 HeroSection.tsx 편집 시 "handoff 폴더의 최신 PDF 기준"으로 정확히 반영됐다. 배운 점: AI 세션 전에 "메모리 파일"로 컨텍스트를 영속화하면 장기 프로젝트에서 삽질이 줄었다.

## 반복 지시 피하기: 옵시디언 cap 설정 사례
옵시디언 정리 프로젝트에서 git trend와 arxiv 스코어링 로직을 조정할 때, "cap을 100으로 고정 말고 변수별로 100점 만점 분배, 무신사/토스는 90 cap"이라고 처음 요청했는데 Claude가 이전 지시를 잊고 고정 cap으로 제안했다. 두세 번 반복하니 "개같은 cap 설정 하지 말고 변수 조절"처럼 강하게 재강조해야 겨우 README.md와 arxiv_collector.py를 제대로 수정했다. 한 번에 모든 변수(cap 100, 무신사90 등)를 나열했지만 AI가 우선순위를 놓쳤던 탓. 다음부턴 "1. cap 제거, 2. 변수별 가중치 예시 제시"처럼 번호 매긴 단계 프롬프트로 바꿔보자. 안 통한 점: 감정적 표현("개같은")은 무시당하고, 숫자 중심 재지시가 먹혔다.

## 비즈니스 로직 검토: 사주 앱 비용·IAP 분석
사주 프로젝트에서 서버 인프라 비용을 MAU 1000 기준으로 "minipc나 워크스테이션 스펙(현재 시스템 프롬프트 + Claude Sonnet API + Gamma 2b 파인튜닝)"이라고 구체 스펙 주며 물었더니, 적자 여부와 pass 도입 대안(문자 인증 전환, IAP 비용 고려)을 논리적으로 분석해줬다. "궁합 상세 API 호출 비용 vs 채팅 오픈 타이밍" 모순도 잘 짚었고, Info.plist나 chat_list_screen.dart 수정 지침도 나왔다. 프롬프트에서 "1. pass 월 비용 vs 문자 인증, 2. IAP 도입 고려사항, 3. 약관 필수 부분 생성" 순차 태스크로 하니 응답이 체계적이었다. 잘 통한 점: 기술 스펙 + 비즈니스 시나리오(MAU, 소비자 플로우)를 섞어주니 AI가 "인간 검토처럼" 균형 잡힌 피드백 줌. 다만 PDF 핸드오프 v3 갱신은 "이전 PDF IAP 기준" 컨텍스트가 필요해 직접 비교해야 했다.

## 트레이딩 봇 디버깅: 세션 summary와 세밀 질문
bit-scalp과 ficeptterminal에서 매크로 스코어(RANGING 0.60, HIGH_VOLATILITY 0.80)와 청산가 로직을 물었을 때, "명목 34.4가 왜 3.44달러? SL 2.5%인데 청산가 왜 참고?"처럼 로그 스니펫을 직접 붙여주니 risk_manager.py와 executor.py 수정이 정확했다. ficeptterminal은 이전 세션 컨텍스트가 끊겨 summary를 먼저 주고 이어갔더니 quantflow_runner.py walk-forward 결과 분석이 매끄러웠다. 안 통한 점: 가용잔고 90% 리스크 조정은 "이미 ranging/strength 필터 있으니 90%도 빼자" 제안이 AI에서 안 나와 직접 orchestrator.py 손봤다. 노하우: 로그 테이블 붙이고 "가용잔고 리프레쉬 확인"처럼 실행 순서 질문으로 AI를 "디버거 도구"로 썼다.
