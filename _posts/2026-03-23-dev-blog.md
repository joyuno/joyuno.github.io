# 오늘의 TIL: ML 엔진 OOM 싸움과 데이터 파이프라인 디버깅의 하루

오늘은 `otel_project`에서 ML 엔진과 ClickHouse의 메모리 최적화 작업을 본격적으로 시작한 날이다. 아침부터 신호탄은 ML 엔진과 ClickHouse가 모두 **signal 137(OOM Kill)**로 종료되었다는 리포트였다. Collector 로그에는 ClickHouse 쓰기 타임아웃이 328회나 기록되어 있었고, ML 엔진은 모든 레이어에서 "데이터 없음"을 반복하고 있었다. 32GB RAM 호스트에서 메모리 전쟁이 시작된 것이다.

## ClickHouse 메모리 설정 최적화

**문제**: ClickHouse 컨테이너가 OOM으로 자꾸 죽는다. 기본 설정이 호스트 메모리를 너무 많이 잡아먹는 것 같았다.

**시도**: `deploy/docker/clickhouse-memory.xml`을 새로 작성했다. 서버 메모리를 7GB, mark cache를 2GB, uncompressed cache를 1GB로 제한하는 설정을 추가했다. 동시에 `docker-compose.yml`에서 ClickHouse 서비스에 `mem_limit: 8g`를 명시적으로 설정했다.

**해결**: 
```xml
<max_server_memory_usage>7000000000</max_server_memory_usage>
<mark_cache_size>2000000000</mark_cache_size>
<uncompressed_cache_size>1000000000</uncompressed_cache_size>
```
이렇게 설정하니 ClickHouse가 안정적으로 뜨기 시작했다. 메모리 사용량이 8GB 이하로 제한되면서 호스트의 다른 서비스들과 공존할 수 있게 됐다.

**배운 점**: ClickHouse는 기본적으로 호스트 메모리의 절반을 사용하려고 하는데, 이게 컨테이너 환경에서는 독이 될 수 있다. 명시적인 메모리 제한이 필수적이다.

## ML 엔진 파라미터 튜닝과 이상탐지 민감도 조정

**문제**: ML 엔진의 L1 이상탐지가 너무 민감해서 정상 데이터를 노이즈로 잡고 있었다. 또 스케줄러가 60분 주기로 동작하는 것 같아 실시간성이 떨어졌다.

**시도**: 먼저 `config.py`에서 `SCHEDULE_INTERVAL_MIN`을 "5"로 설정해 5분 주기로 변경했다. 그런데 이상탐지 결과가 여전히 만족스럽지 않아서 `layer1.py`의 파라미터들을 들여다봤다.

**해결**:
1. `NOISE_SCORE_FLOOR`를 0.65에서 0.75로 올려서 노이즈 필터링을 더 강화했다.
2. `ANOMALY_SCORE_THRESHOLD`도 0.65로 조정해 이상 판정 기준을 높였다.
3. `L1_MAX_SAMPLES`를 30000에서 15000으로 줄여 메모리 사용량을 절반으로 감소시켰다.

**배운 점**: ML 파라미터 튜닝은 "찍먹"보다는 체계적인 접근이 필요하다. 로그를 보면서 어떤 파라미터가 어떤 효과를 내는지 이해하는 시간이 필요했다. 특히 `ANOMALY_SCORE_THRESHOLD`가 낮으면 민감도는 높아지지만 false positive가 늘어난다는 걸 직접 보게 됐다.

## 데이터 파이프라인 단절 진단

**문제**: ML 엔진이 계속 "데이터 없음" 상태에서 대기 중이었다. 설정을 바꿔도 결과가 나오지 않았다.

**시도**: Docker 컨테이너 상태부터 확인했다. `docker ps`로 보니 ClickHouse 컨테이너가 꺼져 있었다! ML 엔진 컨테이너는 실행 중이지만 실제로는 아무 일도 하지 못하고 있었다.

**해결**:
1. ClickHouse 컨테이너를 재시작하고 연결 테스트
2. ML 엔진 코드에서 DB 쿼리 부분 디버깅
3. ClickHouse 초기화 스크립트와 실제 테이블명 매칭 확인

알고 보니 **테이블명 불일치**가 문제였다. ML 엔진 코드에서는 `anomalies` 테이블을 쿼리하는데, 실제 ClickHouse에는 `anomaly_results` 테이블만 있었다. Collector가 데이터를 쓰는 테이블과 ML 엔진이 읽으려는 테이블이 달랐던 것이다.

**배운 점**: 데이터 파이프라인 디버깅은 "끝에서부터 거꾸로" 해야 한다. ML 엔진 에러 → 쿼리 확인 → 테이블 존재 확인 → Collector 데이터 흐름 순으로 추적하니 금방 원인을 찾을 수 있었다.

## 크로스 레이어 통합과 인과관계 분석 추가

**문제**: 기존 알림 시스템이 단순히 이상 감지 결과만 전달해서, 실제 장애 원인을 파악하기 어려웠다.

**시도**: `cross_layer.py`에 인과관계 분석 모듈을 통합했다. `_find_root_cause_traces()` 함수를 새로 작성해서, 이상이 감지되면 관련된 트레이스들을 추적하고 근본 원인을 찾도록 했다.

**해결**:
```python
def _find_root_cause_traces(self, anomaly_window_start, anomaly_window_end):
    # 서비스 맵 기반으로 인과관계 체인 구축
    causal_chain = build_causal_chain(anomaly_window_start, anomaly_window_end)
    root_cause_traces = correlate_logs(causal_chain)
    return json.dumps(root_cause_traces)
```
Go 백엔드의 `notification_model.go`와 `notification_store.go`도 수정해서 `root_cause_traces`와 `root_cause_summary` 필드를 추가했다.

**배운 점**: 단순한 "무언가 잘못됐다"는 알림보다 "무엇이, 왜, 어떻게 잘못됐는지"를 알려주는 것이 훨씬 가치 있다. 인과관계 분석을 추가하니 알림의 actionable함이 크게 향상됐다.

## 프론트엔드 빌드 에러와의 우연한 조우

**문제**: 중간에 `Dockerfile.frontend` 빌드가 실패했다. `npm run build`에서 exit code가 제대로 안 나왔다.

**시도**: 로컬에서는 `npm run build`가 잘 되는데, Docker 빌드에서만 실패했다. 빌드 캐시 문제인가 싶어 Docker 빌드 캐시를 지우고 다시 시도했다.

**해결**: 알고 보니 일시적인 네트워크 문제나 의존성 패키지 서버 이슈였다. 몇 분 뒤 다시 시도하니 정상적으로 빌드됐다.

**배운 점**: Docker 빌드 실패는 항상 코드 문제만은 아니다. 인프라 이슈, 네트워크, 캐시 문제 등을 의심해봐야 한다. 그리고... 가끔은 그냥 기다려보는 것도 해결책이다.
