1. Feast란?

Feast(Feature Store)는 머신러닝 피처를 관리, 서빙, 검색하기 위한 오픈소스 플랫폼입니다. ML 시스템에서 피처 데이터의 일관성, 재현성, 접근성을 보장하는 데 중점을 둡니다.

📦 피처 레지스트리

피처 정의, 데이터 소스, 엔티티, 온라인/오프라인 스토어를 중앙에서 관리합니다. Git 기반 버전 관리를 지원합니다.

🔄 오프라인 서빙

모델 학습/평가를 위해 대규모 히스토리 데이터를 배치 형태로 제공합니다. Spark, BigQuery 등과 통합됩니다.

⚡ 온라인 서빙

실시간 예측을 위해 초저지연 피처 조회를 제공합니다. Redis, DynamoDB 등을 백엔드로 사용합니다.

2. 왜 써야 하는가?

기존 ML 파이프라인 대비 Feast가 해결하는 문제점을 비교합니다.

비교 항목 전통적 방식 (직접 구현) Feast 사용 시
피처 일관성 학습/서빙 시 피처 계산 로직이 달라 훈련-서빙 편향 발생 가능성 높음 단일 소스로 정의되어 온라인/오프라인 일관성 보장
피처 검색 & 재사용 팀/프로젝트 간 피처 공유 어려움, 중복 계산 발생 중앙 레지스트리를 통한 피처 검색 및 재사용 용이
운영 오버헤드 피처 파이프라인, 서빙 인프라 각각 구축 및 유지보수 필요 표준화된 아키텍처와 커넥터 제공으로 운영 부담 감소
실시간 서빙 낮은 지연 시간 요구 시 별도 키-값 저장소 설계 필요 내장 온라인 스토어 지원으로 실시간 피처 조회 간소화

3. 핵심 개념

Feast 아키텍처의 주요 구성 요소와 데이터 흐름입니다.

+-------------------------------------------------------------------+ | Feature Definitions (YAML/Repo) | | +---------------------------+ | | | Entity | Feature View | | | | Data Source | Registry | | +---------------------------^---------------------------------------+ | (Apply) +-------------------------------------------------------------------+ | FEAST CORE | | +---------------------+------------------+ | | | | | | | Offline Store Online Store Registry | | (BigQuery, Snowflake) (Redis, DynamoDB) (File, GCS) | +--------------^---------------------^-----------------------------+ | | +------v-----+ +-------v------+ | Training | | Serving | | (Batch) | | (Real-time) | +------------+ +--------------+

핵심 용어

  • Entity: 피처의 기본 키 (예: user_id, driver_id).
  • Feature View: 하나 이상의 피처와 데이터 소스, 엔티티를 그룹화한 논리적 단위.
  • Data Source: 원본 데이터 위치 (파일, 데이터 웨어하우스, 스트림).
  • Offline Store: 히스토리 데이터 저장소 (학습용).
  • Online Store: 실시간 서빙을 위한 저지연 저장소.
  • Registry: 피처 정의와 메타데이터를 저장하는 중앙 저장소.

4. 퀵스타트

Step 1: 설치

# 기본 설치 (파일 기반 레지스트리 + 로컬 온라인 스토어)
pip install feast

# 추가 제공자 설치 (예: GCP, AWS, Snowflake)
pip install feast[gcp]
# 또는
pip install feast[aws]

Step 2: 프로젝트 초기화

# feast 프로젝트 생성
feast init my_feature_repo
cd my_feature_repo

# 디렉토리 구조 확인
ls -la
# feature_repo/
#   ├── feature_store.yaml  # Feast 설정
#   ├── example.py          # 예제 코드
#   └── data/               # 예제 데이터

Step 3: 피처 정의 (example.py 수정)

from datetime import timedelta
from feast import Entity, FeatureView, Field, FileSource
from feast.types import Float32, Int64
import pandas as pd

# 1. 엔티티 정의
driver = Entity(name="driver", join_keys=["driver_id"])

# 2. 데이터 소스 정의 (예: Parquet 파일)
driver_stats_source = FileSource(
    path="data/driver_stats.parquet",
    timestamp_field="event_timestamp",
    created_timestamp_column="created",
)

# 3. 피처 뷰 정의
driver_stats_fv = FeatureView(
    name="driver_hourly_stats",
    entities=[driver],
    ttl=timedelta(days=1),
    schema=[
        Field(name="conv_rate", dtype=Float32),
        Field(name="acc_rate", dtype=Float32),
        Field(name="avg_daily_trips", dtype=Int64),
    ],
    online=True,
    source=driver_stats_source,
    tags={"team": "driver_matching"},
)

Step 4: 적용 및 머티리얼라이즈

# 레지스트리에 피처 정의 적용
feast apply

# 오프라인 스토어에서 온라인 스토어로 데이터 적재 (머티리얼라이즈)
feast materialize-incremental $(date -u +"%Y-%m-%dT%H:%M:%S")

Step 5: 피처 조회

from feast import FeatureStore
import pandas as pd

store = FeatureStore(repo_path=".")

# 오프라인 조회 (학습 데이터 생성)
training_df = store.get_historical_features(
    entity_df=pd.DataFrame({
        "driver_id": [1001, 1002, 1003],
        "event_timestamp": pd.to_datetime(["2024-01-01", "2024-01-01", "2024-01-01"])
    }),
    features=["driver_hourly_stats:conv_rate", "driver_hourly_stats:avg_daily_trips"]
).to_df()

# 온라인 조회 (실시간 예측)
online_features = store.get_online_features(
    features=["driver_hourly_stats:conv_rate", "driver_hourly_stats:acc_rate"],
    entity_rows=[{"driver_id": 1001}]
).to_dict()

5. 핵심 설정 레퍼런스

feature_store.yaml의 주요 구성 옵션입니다.

설정 키 설명 예시 값
project 프로젝트 이름 (네임스페이스) my_ml_project
registry 레지스트리 저장소 위치 (파일, GCS, S3) gs://my-bucket/registry.db
provider 클라우드 제공자 (local, gcp, aws) gcp
online_store 온라인 스토어 타입과 설정 type: redis
connection_string: "localhost:6379"
offline_store 오프라인 스토어 타입과 설정 type: bigquery
dataset: feast_dataset
entity_key_serialization_version 엔티티 키 직렬화 버전 (호환성) 2

6. 실무 팁 & 트러블슈팅

✅ 실무 팁

  • 피처 그룹화: 동일한 데이터 소스와 엔티티를 공유하는 피처는 하나의 FeatureView로 묶어 관리 효율성을 높이세요.
  • TTL 설정: 온라인 스토어의 저장 비용과 데이터 신선도를 고려해 적절한 TTL(Time To Live)을 설정하세요.
  • CI/CD 통합: feast apply를 CI 파이프라인에 통합해 피처 정의 변경을 자동으로 적용하세요.
  • 모니터링: 온라인 스토어의 지연 시간, 적재 지표, 피처 누락 비율을 모니터링하세요.

⚠️ 자주 겪는 문제 & 해결책

  • 문제: feast apply 시 "Registry at path X is not initialized" 오류.
    해결: feast init으로 초기화했는지 확인하고, registry 경로가 올바른지 feature_store.yaml에서 점검하세요.
  • 문제: 온라인 조회 시 피처 값이 null로 반환됨.
    해결: 1) feast materialize로 데이터가 온라인 스토어에 적재되었는지 확인. 2) FeatureView의 online=True 설정 확인. 3) 엔티티 키가 정확히 매칭되는지 확인.
  • 문제: 대규모 히스토리 데이터 조회 시 메모리 부족.
    해결: Spark 또는 BigQuery 오프라인 스토어를 사용하고, 배치 단위로 데이터를 처리하세요.
  • 문제: 피처 피벗(Pivot) 시 성능 저하.
    해결: 와이드 포맷(wide format) 데이터 소스를 사용하거나, Feast의 변환 기능 대신 원본 데이터를 피벗된 형태로 준비하세요.
