1. Helm이란?

Helm은 Kubernetes용 패키지 매니저입니다. 복잡한 Kubernetes 애플리케이션을 정의, 설치, 업그레이드하는 프로세스를 단순화하는 "차트(Charts)"라는 패키지 형식을 제공합니다.

📦 패키지 매니저

apt, yum, pip와 유사하게 Kubernetes 애플리케이션(Deployment, Service, ConfigMap 등)을 하나의 패키지(Chart)로 묶어 배포, 버전 관리, 의존성 해결을 가능하게 합니다.

🛠 템플릿 엔진

Go 템플릿을 사용하여 동적인 Kubernetes 매니페스트를 생성합니다. 환경별(dev/stage/prod) 값(value) 주입을 통해 재사용성을 극대화합니다.

🔄 릴리스 관리자

Kubernetes에 배포된 애플리케이션의 릴리스(설치 인스턴스)를 추적하고, 롤백, 업그레이드, 삭제를 관리합니다. helm list로 모든 릴리스를 확인할 수 있습니다.

핵심: Helm은 "Kubernetes 애플리케이션의 생명주기 관리"를 표준화하고 자동화하는 도구입니다.

2. 왜 써야 하는가?

기존의 kubectl apply 방식과 Helm을 비교해보면 생산성 차이가 명확합니다.

비교 항목 kubectl apply (Raw YAML) Helm
재사용성 환경별 YAML 복사/수정 필요. 중복 코드 발생. 단일 Chart 템플릿 + values.yaml 파일로 모든 환경 관리.
배포 관리 배포 히스토리 추적 어려움. 롤백이 수동. 릴리스 버전 자동 기록. helm rollback으로 한 줄 롤백.
의존성 수동으로 순서대로 apply. (e.g., ConfigMap → Deployment) Chart.yaml에 의존성 정의. Helm이 순서대로 설치/업데이트.
공유 및 배포 Git 저장소에 YAML 파일을 올리는 것이 일반적. 공식/사설 Chart 저장소(Repository)로 패키지 공유 및 버전 관리.
템플릿 기능 Kustomize, envsubst 등 별도 도구 필요. 내장된 강력한 템플릿 엔진 제공.
주의: 간단한 단일 Pod나 학습용 배포에는 오버헤드가 될 수 있습니다. 복잡성과 팀의 운영 규모에 맞게 도입을 결정하세요.

3. 핵심 개념

Helm의 아키텍처와 핵심 구성 요소를 이해해야 실무 적용이 수월합니다.

+-------------------+ +----------------------+ +-------------------+ | Helm Client |----->| Tiller (v2) / |----->| Kubernetes | | (helm CLI) | | Helm Library (v3) | | API Server | +-------------------+ +----------------------+ +-------------------+ | | | v v v [차트 저장소 조회] [템플릿 렌더링] [리소스 생성/관리] [명령어 실행] [릴리스 관리] (Deploy, SVC, 등)

주요 용어

  • Chart: Helm 패키지. 애플리케이션을 실행하는 데 필요한 모든 리소스 정의의 집합.
  • Repository (Repo): Chart가 저장되고 공유되는 곳.
  • Release: Kubernetes 클러스터에 배포된 Chart의 인스턴스. 하나의 Chart로 여러 Release 생성 가능.
  • Values: Chart에 전달되는 구성 값. 템플릿을 구체화합니다. (values.yaml, --set)
  • Template: Kubernetes 매니페스트를 생성하는 Go 템플릿 파일들 (templates/ 디렉토리).

Helm v3 이후 Tiller가 제거되어 아키텍처가 단순화되었으며, 보안과 운영이 개선되었습니다. 본 가이드는 v3 기준입니다.

4. 퀵스타트: 5분 안에 첫 Chart 배포하기

Step 1: 설치

macOS (Homebrew) / Linux (스크립트) / Windows (Chocolatey)

# macOS
brew install helm

# Linux
curl https://raw.githubusercontent.com/helm/helm/main/scripts/get-helm-3 | bash

# 설치 확인
helm version --short

Step 2: 기본 저장소 추가

# 공식 안정화(stable) 저장소 추가 (최신 v3에서는 bitnami를 권장)
helm repo add bitnami https://charts.bitnami.com/bitnami
helm repo update  # 저장소 메타데이터 갱신

# 저장소 목록 확인
helm repo list

Step 3: 첫 번째 Chart 검색 및 설치

# nginx 차트 검색
helm search repo bitnami/nginx

# nginx 설치 (my-nginx라는 릴리스 이름으로)
helm install my-nginx bitnami/nginx

# 릴리스 상태 확인
helm list
kubectl get pods,svc -l app.kubernetes.io/instance=my-nginx

Step 4: 나만의 간단한 Chart 생성하기

# Chart 스캐폴딩 생성
helm create my-first-app
cd my-first-app
tree .  # 구조 확인

# templates/ 디렉토리의 YAML 파일들을 values.yaml 값으로 렌더링 테스트
helm template . --debug

# 로컬에 설치 (--dry-run으로 시뮬레이션 가능)
helm install my-release . --dry-run
helm install my-release . --namespace=default --create-namespace

Step 5: Python 애플리케이션을 위한 Chart 예제 (values.yaml)

values.yaml

# my-first-app/values.yaml
replicaCount: 3

image:
  repository: my-registry/python-api
  tag: "v1.2.0"
  pullPolicy: IfNotPresent

service:
  type: ClusterIP
  port: 8000

resources:
  limits:
    cpu: 500m
    memory: 512Mi
  requests:
    cpu: 200m
    memory: 256Mi

env:
  - name: DEBUG
    value: "false"
  - name: DATABASE_URL
    valueFrom:
      secretKeyRef:
        name: db-secret
        key: url

템플릿 예시 (templates/deployment.yaml 일부)

# my-first-app/templates/deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: {{ .Release.Name }}-deployment
spec:
  replicas: {{ .Values.replicaCount }}
  selector:
    matchLabels:
      app: {{ .Release.Name }}
  template:
    metadata:
      labels:
        app: {{ .Release.Name }}
    spec:
      containers:
      - name: {{ .Chart.Name }}
        image: "{{ .Values.image.repository }}:{{ .Values.image.tag }}"
        ports:
        - containerPort: {{ .Values.service.port }}
        env:
        {{- toYaml .Values.env | nindent 8 }}
        resources:
          {{- toYaml .Values.resources | nindent 10 }}

5. 핵심 설정 레퍼런스 (values.yaml)

경로 기본값/예시 설명
replicaCount 1 Pod 복제본 수
image.repository nginx 컨테이너 이미지 레포지토리
image.tag "latest" 이미지 태그 (고정 버전 사용 권장)
image.pullPolicy IfNotPresent Always, IfNotPresent, Never
service.type ClusterIP ClusterIP, NodePort, LoadBalancer
service.port 80 서비스 노출 포트
ingress.enabled false Ingress 리소스 생성 여부
resources.limits {} CPU/Memory 상한
nodeSelector {} Pod를 특정 노드에 스케줄링
tolerations [] 테인트(taint) 허용 설정
affinity {} Pod/Node 어피니티/안티-어피니티 규칙

6. 실무 팁 & 트러블슈팅

🚀 실무 팁

1. Values 파일 전략

values.yaml을 기본으로, 환경별 values-{env}.yaml 파일을 만들어 --values 또는 -f 플래그로 오버라이드하세요.

helm install -f values-production.yaml .

2. 릴리스 이름 관리

자동 생성(--generate-name)보다는 CI/CD 파이프라인에서 고유한 릴리스 이름(예: app-name-${GIT_SHA})을 사용하여 추적성을 높이세요.

3. 의존성(Subchart) 관리

공통 컴포넌트(로그 수집기, 모니터링 에이전트)는 Subchart로 분리하고 Chart.yaml의 dependencies: 섹션에서 관리하세요.

helm dependency update .

🔧 자주 겪는 문제 & 해결책

문제 원인 해결 방법
Error: rendered manifests contain a resource that already exists 동일한 이름의 Kubernetes 리소스가 이미 존재. 기존 리소스 삭제 또는 helm upgrade 사용. --atomic 플래그로 실패 시 자동 롤백.
Error: unable to build kubernetes objects from release manifest 템플릿 렌더링 오류 (형식 오류, 값 누락). helm template . --debug로 렌더링 결과를 먼저 검증하세요.
설치 후 Pod가 CrashLoopBackOff 상태 values.yaml의 설정 오류 (예: 잘못된 환경 변수, 리소스 부족). kubectl describe pod와 kubectl logs로 원인 확인. values.yaml의 image.tag 등을 점검.
Error: release named "xxx" already exists 동일한 릴리스 이름으로 이미 설치됨. helm list --all로 확인 후, helm uninstall xxx로 삭제하거나 다른 이름으로 설치.
템플릿 함수({{ .Values.foo }})가 작동 안 함 값이 정의되지 않았거나 자료형 불일치. 템플릿에 기본값 설정: {{ .Values.foo | default "bar" }}. required 함수로 필수값 강제.
업그레이드가 너무 오래 걸리거나 멈춤 리소스의 readinessProbe 실패 또는 리소스 할당 대기. --timeout 플래그로 시간 제한 증가. kubectl get events로 클러스터 이벤트 확인.
디버깅 필수 명령어:
helm get manifest <RELEASE_NAME> : 배포된 실제 매니페스트 확인
helm get values <RELEASE_NAME> : 해당 릴리스에 적용된 values 확인
helm history <RELEASE_NAME> : 릴리스 변경 이력 확인
