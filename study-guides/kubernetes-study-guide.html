1. Kubernetes란?

Kubernetes(쿠버네티스, K8s)는 컨테이너화된 애플리케이션의 배포, 스케일링, 관리를 자동화하는 오픈소스 플랫폼입니다. Google이 설계했으며 현재 CNCF(Cloud Native Computing Foundation)가 관리합니다.

자동화된 배포와 롤링 업데이트

애플리케이션을 선언적으로 정의하고, 원하는 상태를 유지하며, 무중단 업데이트를 지원합니다.

서비스 디스커버리와 로드 밸런싱

컨테이너에 DNS 이름이나 자체 IP 주소를 부여하고, 트래픽을 분산시킵니다.

자동 복구(self-healing)

실패한 컨테이너를 재시작하고, 노드 장애 시 컨테이너를 다른 노드로 재배치합니다.

2. 왜 써야 하는가?

기존 배포 방식과의 비교를 통해 Kubernetes의 가치를 확인해 보세요.

기준 전통적 VM 단순 컨테이너 (Docker) Kubernetes
스케일링 수동, 느림 부분적 자동화 자동, 수평/수직
가용성 수동 장애 조치 제한적 내장된 고가용성
리소스 효율 낮음 (OS 오버헤드) 높음 매우 높음 (빈 패킹)
배포 복잡도 높음 중간 초기 높음, 이후 낮음
생태계 제한적 풍부 매우 풍부 (Helm, Operators 등)

3. 핵심 개념

Kubernetes 클러스터의 기본 아키텍처를 ASCII 다이어그램으로 표현했습니다.

[ 사용자 / kubectl / API ] | v +-----------------------+ | Control Plane | | (Master Node(s)) | | | | - API Server | | - Scheduler | | - Controller Manager | | - etcd (클러스터 DB) | +-----------------------+ | | (관리 명령) v +---------------------------------------------------+ | Worker Nodes | | +----------+ +----------+ +----------+ | | | Pod | | Pod | | Pod | | | | [Container] | [Container] | [Container] | | +----------+ +----------+ +----------+ | | | kubelet | | kubelet | | kubelet | | | | kube-proxy| | kube-proxy| | kube-proxy| | | +----------+ +----------+ +----------+ | | | +---------------------------------------------------+
핵심 오브젝트: Pod(컨테이너 그룹), Deployment(배포 관리), Service(네트워크 엔드포인트), ConfigMap/Secret(설정), Namespace(가상 클러스터).

4. 퀵스타트

4.1 로컬 개발 환경 설치 (Minikube)

# Minikube 설치 (macOS 예시)
brew install minikube
# 클러스터 시작
minikube start --driver=docker
# 상태 확인
kubectl cluster-info

4.2 첫 번째 애플리케이션 배포

간단한 Python 웹 애플리케이션을 배포해 봅니다.

app.py (로컬에 생성):

from flask import Flask
app = Flask(__name__)

@app.route('/')
def hello():
    return "Hello from Kubernetes Pod!"

if __name__ == '__main__':
    app.run(host='0.0.0.0', port=8080)

Dockerfile:

FROM python:3.9-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install -r requirements.txt
COPY app.py .
CMD ["python", "app.py"]

requirements.txt:

Flask==2.3.2

Kubernetes Deployment YAML (deploy.yaml):

apiVersion: apps/v1
kind: Deployment
metadata:
  name: python-webapp
spec:
  replicas: 2
  selector:
    matchLabels:
      app: python-webapp
  template:
    metadata:
      labels:
        app: python-webapp
    spec:
      containers:
      - name: web
        image: python-webapp:latest
        ports:
        - containerPort: 8080
---
apiVersion: v1
kind: Service
metadata:
  name: python-webapp-service
spec:
  selector:
    app: python-webapp
  ports:
    - protocol: TCP
      port: 80
      targetPort: 8080
  type: LoadBalancer

배포 명령어:

# 이미지 빌드 및 Minikube에 로드
docker build -t python-webapp .
minikube image load python-webapp:latest
# 배포 적용
kubectl apply -f deploy.yaml
# 상태 확인
kubectl get pods,svc

5. 핵심 설정 레퍼런스

설정 항목 키 (YAML) 기본값 / 예시 설명
리소스 제한 resources.limits cpu: "500m", memory: "512Mi" 컨테이너가 사용할 수 있는 최대 리소스
레플리카 수 spec.replicas 3 유지할 Pod의 개수
헬스체크 livenessProbe httpGet: /health, 초기지연: 30초 컨테이너 재시작 여부 판단
서비스 타입 spec.type ClusterIP, NodePort, LoadBalancer 서비스 노출 방식
볼륨 마운트 volumeMounts mountPath: "/data" 컨테이너 내 스토리지 경로
환경 변수 env - name: DB_HOST value: "db" 컨테이너에 주입할 환경 변수

6. 실무 팁 & 트러블슈팅

팁 1: kubectl alias 설정
alias k=kubectl을 ~/.bashrc 또는 ~/.zshrc에 추가하여 명령어를 단축하세요.
문제: Pod가 Pending 상태로 멈춤
원인: 리소스 부족 또는 노드 선택기 불일치.
해결: kubectl describe pod <pod-name>으로 이벤트 확인. kubectl get nodes로 노드 상태 점검.
팁 2: ConfigMap으로 설정 분리
환경별 설정(개발/스테이징/운영)을 ConfigMap으로 관리하고 Pod에 마운트하세요. 코드와 설정을 분리할 수 있습니다.
문제: 서비스에 연결할 수 없음
원인: 레이블 셀렉터 불일치 또는 네트워크 정책.
해결: kubectl get svc <service-name> -o yaml로 selector 레이블 확인. Pod의 레이블과 일치하는지 확인하세요.
팁 3: 네임스페이스 활용
팀/환경별로 네임스페이스를 분리하여 리소스를 격리하고 관리 효율성을 높이세요.
