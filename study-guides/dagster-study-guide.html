1. Dagster란?

Dagster는 데이터 애플리케이션을 구축, 실행, 모니터링하기 위한 클라우드 네이티브 오케스트레이션 플랫폼입니다. 코드 중심의 개발 경험과 데이터 자산(Asset)을 일급 객체로 관리하는 철학이 특징입니다.

1. 데이터 자산 중심 (Asset-Based)

테이블, 파일, 모델 등 데이터 자산을 정의하고, 그 의존성과 라이프사이클을 명시적으로 관리합니다. "이 파이프라인이 무엇을 만드는가?"에 집중합니다.

2. 개발자 경험 최적화

로컬 테스트, 타입 검증, 구성 관리, 디버깅 도구가 뛰어나 개발 및 유지보수 효율성을 높입니다. 강력한 UI/CLI를 제공합니다.

3. 운영 견고성 (Operational Robustness)

재시도, 알림, 리소스 관리, 데이터 품질 검증(Expectation)을 내장하여 프로덕션 환경에서의 안정성을 보장합니다.

2. 왜 써야 하는가?

기존 오케스트레이션 도구와의 비교를 통해 Dagster의 차별점을 확인합니다.

도구 패러다임 주요 강점 Dagster와 비교
Airflow Task-Based (DAG) 성숙한 생태계, 다양한 연산자 Airflow는 "작업 실행"에 초점, Dagster는 "데이터 자산 생성"에 초점. Dagster의 개발 경험과 타입 안정성이 우수.
Prefect Task-Based (동적 DAG) Pythonic API, 유연한 동적 워크플로 Prefect는 작업 흐름 자체에 강점. Dagster는 데이터 자산 메타데이터, 데이터 계보, UI 통합에서 더 깊은 통찰 제공.
Luigi Task-Based (의존성) 간단한 설계, Spotify 검증 Luigi는 단순함이 장점이지만, 모니터링, 구성, 자산 관리 기능이 제한적. Dagster는 현대적 요구사항을 충족.
Dagster Asset-Based 통합 데이터 플랫폼 경험, 뛰어난 개발/운영 도구, 데이터 계보 및 품질 내장 -
💡 결론: 복잡한 데이터 플랫폼을 구축하고, 데이터 자산의 가시성과 신뢰성을 높이며, 팀의 개발 생산성을 향상시키고 싶다면 Dagster가 강력한 선택지입니다.

3. 핵심 개념

Dagster 아키텍처의 핵심 구성 요소와 데이터 흐름을 이해합니다.

+---------------------+ +----------------------+ +-------------------+ | User Code | | Dagster Runtime | | External Systems| | |------| |------| | | - Assets | | - Dagster Daemon | | - Data Warehouse | | - Ops | | - Run Coordinator | | - Object Storage | | - Jobs | | - Scheduler | | - Compute (K8s) | | - Resources | | - Event Log | | | +---------------------+ +----------------------+ +-------------------+ ^ ^ ^ | | | | +----------------------+ | +-------------------| Dagster UI/CLI |------------+ | | | - Asset Catalog | | - Run Viewer | | - Launchpad | +----------------------+

주요 용어

  • Asset: 데이터 자산(테이블, 파일, ML 모델). 정의와 의존성을 명시.
  • Op: 계산의 기본 단위. 순수 함수처럼 작성.
  • Job: 실행 가능한 Op/Asset의 그래프.
  • Resource: 외부 서비스(DB, API)에 대한 연결을 추상화.
  • I/O Manager: Asset의 로드/저장 방식을 관리.
  • Sensor / Schedule: Job 실행을 트리거.

4. 퀵스타트

로컬 환경에서 첫 번째 Dagster 프로젝트를 설정하고 실행해봅니다.

Step 1: 설치
# pip를 사용한 설치 (가상환경 권장)
pip install dagster dagster-webserver

# 또는 Poetry 사용 시
poetry add dagster dagster-webserver
Step 2: 프로젝트 구조 생성
mkdir my_first_dagster
cd my_first_dagster
touch my_first_dagster.py
Step 3: 첫 번째 Asset 정의 (my_first_dagster.py)
from dagster import asset, Definitions
import pandas as pd

@asset
def raw_data() -> pd.DataFrame:
    """CSV 파일에서 원본 데이터를 로드합니다."""
    return pd.read_csv("https://raw.githubusercontent.com/datasciencedojo/datasets/master/titanic.csv")

@asset
def processed_data(raw_data: pd.DataFrame) -> pd.DataFrame:
    """원본 데이터를 처리합니다: 결측치 제거, 컬럼 선택."""
    df = raw_data.copy()
    df = df[['Survived', 'Pclass', 'Sex', 'Age', 'Fare']].dropna()
    df['Sex'] = df['Sex'].map({'male': 0, 'female': 1})
    return df

@asset
def survival_report(processed_data: pd.DataFrame) -> str:
    """생존률 분석 리포트를 생성합니다."""
    survival_rate = processed_data['Survived'].mean() * 100
    return f"탐승객 생존률: {survival_rate:.2f}%"

defs = Definitions(
    assets=[raw_data, processed_data, survival_report]
)
Step 4: Dagster UI 서버 실행
dagster dev -f my_first_dagster.py

명령어 실행 후, 브라우저에서 http://localhost:3000으로 접속하여 UI를 확인하고 Asset을 Materialize 해보세요.

✅ 성공: 이제 로컬에서 Dagster 파이프라인을 실행하고 UI에서 자산 계보를 확인할 수 있습니다.

5. 핵심 설정 레퍼런스

실무에서 자주 사용하는 주요 설정을 표로 정리합니다.

구성 요소 설정 키 (YAML 예시) 목적
Resource snowflake_io_manager Snowflake 연결 및 데이터 로드/저장 관리
I/O Manager parquet_io_manager Asset을 Parquet 파일로 S3/로컬에 저장
Executor multiprocess_executor 병렬 실행을 위한 멀티프로세스 설정
Logger json_console_logger 로그를 JSON 형식으로 출력 (ELK 수집용)
Schedule daily_3am_schedule 매일 오전 3시에 Job 실행
Sensor s3_file_sensor S3 버킷에 새 파일 도착 시 Job 트리거
⚠️ 주의: 프로덕션 배포 시에는 dagster.yaml 파일을 통해 중앙 설정을 관리하고, Secrets는 환경 변수나 전문 관리 도구를 사용하세요.

6. 실무 팁 & 트러블슈팅

자주 겪는 문제와 해결책

문제 1: Asset이 UI에 보이지 않음

원인: Definitions 객체에 Asset이 등록되지 않았거나, 모듈 경로가 잘못됨.

해결: defs = Definitions(assets=[...])를 확인하고, dagster dev 명령어에 정확한 파일 경로를 지정.

문제 2: 외부 리소스(DB) 연결 실패

원인: Resource 구성 오류 또는 네트워크 문제.

해결: dagster instance migrate로 스키마 업데이트. Resource의 @resource 데코레이터와 구성 YAML을 점검.

문제 3: 병렬 실행 시 성능 저하

원인: Executor 설정 미비 또는 리소스 경합.

해결: multiprocess_executor 사용 시 max_concurrent 값을 조정. 각 Op에 태그를 달아 리소스 제약을 명시.

문제 4: 대용량 데이터 메모리 부족

원인: 단일 Op에서 모든 데이터를 메모리에 로드.

해결: 청크 처리(Chunking) 구현 또는 Spark/DuckDB I/O Manager 사용으로 Out-of-Core 처리.

실무 팁

  • 태그 활용: Op/Asset에 tags={"team": "ml", "domain": "recommendation"}을 추가하여 UI에서 필터링 및 모니터링.
  • 구성 분리: ConfigSchema를 사용해 런타임 매개변수(예: 날짜, 임계값)를 코드와 분리.
  • 로컬 테스트: materialize_to_memory 또는 execute_in_process로 유닛 테스트 작성.
  • CI/CD 통합: GitHub Actions에서 dagster job execute를 이용한 파이프라인 검증 단계 추가.
