1. Apache Beam란?

Apache Beam은 배치와 스트리밍 데이터 처리를 통합한 프로그래밍 모델입니다. 단일 파이프라인 코드로 다양한 실행 엔진(런너)에서 동작할 수 있도록 설계되었습니다.

📦 통합 처리 모델

한 번의 코드 작성으로 배치와 스트리밍 데이터를 모두 처리할 수 있습니다. "Write once, run anywhere" 철학을 실현합니다.

🚀 멀티 런너 지원

Apache Flink, Google Dataflow, Apache Spark, Samza, Nemo 등 다양한 실행 엔진에서 동일한 파이프라인을 실행할 수 있습니다.

🔧 풍부한 커넥터

Kafka, Pub/Sub, BigQuery, Snowflake, JDBC, 파일 시스템 등 다양한 데이터 소스/싱크를 지원합니다.

2. 왜 써야 하는가?

기존 솔루션 대비 Apache Beam의 장점과 적합한 사용 사례를 비교합니다.

솔루션 주요 특징 Beam 대비 차이점
Apache Spark 배치 중심, 마이크로 배치 스트리밍, 메모리 기반 처리 Beam은 진정한 통합 모델, 스트리밍 퍼스트 설계, 런너 중 하나로 Spark 지원
Apache Flink 스트리밍 퍼스트, 저지연 처리, 상태 관리 강점 Beam은 추상화 레이어, Flink는 런너 중 하나. Beam은 벤더 중립성 제공
Airflow/Luigi 워크플로 오케스트레이션, 태스크 의존성 관리 Beam은 데이터 변환 처리에 특화, 오케스트레이터와 함께 사용 가능
직접 구현 완전한 제어권, 특정 요구사항 최적화 Beam은 유지보수 비용 감소, 확장성, 런타임 최적화 제공
✅ Beam 선택이 적합한 경우: 멀티 클라우드/하이브리드 환경, 배치/스트리밍 통합 필요, 실행 엔진 변경 가능성, 장기적인 파이프라인 유지보수성 중요.

3. 핵심 개념

Beam 프로그래밍 모델의 기본 구성 요소입니다.

+--------------------------------------------------------------------------------+ | Apache Beam Model | | | | [Data Sources] → [Pipeline] → [PCollection] → [Transforms] → [Data Sinks] | | ↑ ↑ ↑ ↑ ↑ | | (I/O Connectors) (Runner) (Parallel Data) (Map, Filter, (I/O Connectors) | GroupBy, etc.) | | | | +--------------------------+ +-------------------------------------------+ | | | Batch Source | | Windowing Strategies | | | | - File-based (CSV, JSON) | | - Fixed Windows - Sliding Windows | | | | - BigQuery, JDBC | | - Session Windows - Global Window | | | +--------------------------+ +-------------------------------------------+ | | | | +--------------------------+ +-------------------------------------------+ | | | Streaming Source | | Triggering Mechanisms | | | | - Kafka, Pub/Sub | | - Processing Time - Watermark | | | | - Kinesis, RabbitMQ | | - Element Count - Composite Triggers | | | +--------------------------+ +-------------------------------------------+ | +--------------------------------------------------------------------------------+
  • Pipeline: 모든 데이터 처리 작업의 컨테이너. 시작점입니다.
  • PCollection: 파이프라인에서 처리되는 분산 데이터 컬렉션.
  • PTransform: PCollection에 적용되는 변환 작업 (Map, Filter, GroupByKey, CoGroupByKey, Combine, Flatten, Partition).
  • Runner: 파이프라인을 실행하는 엔진 (DirectRunner, DataflowRunner, FlinkRunner, SparkRunner 등).
  • Windowing: 무한 스트림 데이터를 유한한 논리적 청크로 분할하는 방법.
  • Trigger: 윈도우의 결과를 언제 내보낼지 결정하는 메커니즘.

4. 퀵스타트

Python 환경에서 Apache Beam을 시작하는 단계별 가이드입니다.

Step 1: 설치

# 기본 설치 (최신 버전)
pip install apache-beam

# GCP Dataflow 러너 사용 시
pip install 'apache-beam[gcp]'

# 특정 러너와 함께 설치
pip install 'apache-beam[flink]'
pip install 'apache-beam[spark]'

Step 2: 첫 번째 파이프라인 작성 (wordcount.py)

import apache_beam as beam
from apache_beam.options.pipeline_options import PipelineOptions

class WordCountOptions(PipelineOptions):
    @classmethod
    def _add_argparse_args(cls, parser):
        parser.add_argument('--input', default='gs://dataflow-samples/shakespeare/kinglear.txt')
        parser.add_argument('--output', required=True)

def run():
    options = WordCountOptions()
    
    with beam.Pipeline(options=options) as p:
        # 파이프라인 정의
        lines = p | 'Read' >> beam.io.ReadFromText(options.input)
        
        counts = (
            lines
            | 'Split' >> beam.FlatMap(lambda x: x.split())
            | 'PairWithOne' >> beam.Map(lambda word: (word, 1))
            | 'GroupAndSum' >> beam.CombinePerKey(sum)
            | 'Format' >> beam.Map(lambda word_count: f'{word_count[0]}: {word_count[1]}')
        )
        
        counts | 'Write' >> beam.io.WriteToText(options.output)

if __name__ == '__main__':
    run()

Step 3: 로컬 실행 (DirectRunner)

# 로컬 파일로 테스트
python wordcount.py --output ./output/result

Step 4: Dataflow에서 실행 (GCP)

# GCP 프로젝트 설정
export PROJECT_ID=your-project-id
export BUCKET_NAME=your-bucket-name
export REGION=asia-northeast3

# Dataflow 실행
python wordcount.py \
  --runner DataflowRunner \
  --project $PROJECT_ID \
  --region $REGION \
  --temp_location gs://$BUCKET_NAME/tmp \
  --output gs://$BUCKET_NAME/output/results \
  --job_name wordcount-job-$(date +%Y%m%d-%H%M%S)

5. 핵심 설정 레퍼런스

파이프라인 실행을 제어하는 주요 옵션들입니다.

옵션 기본값 설명 예시
--runner DirectRunner 실행 엔진 지정 DataflowRunner, FlinkRunner, SparkRunner
--streaming false 스트리밍 모드 활성화 --streaming (플래그만)
--project - GCP 프로젝트 ID (Dataflow용) --project=my-data-project
--region us-central1 리소스가 생성될 지역 --region=asia-northeast3
--temp_location - 임시 파일 저장소 (GCS 경로) --temp_location=gs://bucket/tmp
--staging_location - 스테이징 파일 저장소 --staging_location=gs://bucket/staging
--num_workers 자동 작업자 VM 수 (Dataflow) --num_workers=5
--max_num_workers 자동 최대 작업자 수 --max_num_workers=20
--machine_type n1-standard-4 작업자 머신 타입 --machine_type=n2-standard-8
--disk_size_gb 25 작업자 디스크 크기(GB) --disk_size_gb=100
--experiments - 실험적 기능 활성화 --experiments=use_runner_v2,shuffle_mode=service

6. 실무 팁 & 트러블슈팅

💡 실무 팁

1. 적절한 러너 선택

개발/테스트: DirectRunner
프로덕션 스트리밍 (GCP): DataflowRunner
온프레미스/하이브리드: FlinkRunner
기존 Spark 인프라: SparkRunner

2. 상태 관리 최적화

Stateful 처리 시 타이머와 상태 크기를 고려하세요. 너무 큰 상태는 성능 저하를 일으킵니다. 상태 TTL(Time-To-Live) 설정으로 메모리 관리.

3. 스케일링 전략

Dataflow의 자동 스케일링을 신뢰하되, --max_num_workers로 상한선을 설정하세요. 처리량이 많은 작업은 더 큰 머신 타입을 고려.

⚠️ 자주 겪는 문제 & 해결책

문제 1: "ModuleNotFoundError: No module named 'apache_beam'"
원인: Beam 패키지가 설치되지 않았거나 가상환경 문제.
해결: pip install apache-beam 재설치, 가상환경 활성화 확인.
문제 2: Dataflow 작업이 "QUEUED" 상태에서 멈춤
원인: 할당량 부족, 서비스 계정 권한 문제.
해결: GCP 콘솔에서 할당량 확인, 서비스 계정에 Dataflow Admin, Storage Admin 역할 부여.
문제 3: 스트리밍 작업에서 지연 증가
원인: 핫 키(hot key), 부적절한 윈도우/트리거 설정.
해결: 키 분포 확인, 윈도우 크기 조정, Combine 전에 Reshuffle 적용.
문제 4: "The job is being drained" 후 실패
원인: 작업자 장애, OOM(메모리 부족).
해결: 로그에서 스택 트레이스 확인, --disk_size_gb 증가, --machine_type 업그레이드.
디버깅 명령어:
# Dataflow 작업 로그 보기 gcloud dataflow jobs list --region=REGION gcloud dataflow jobs describe JOB_ID --region=REGION gcloud logging read "resource.type=dataflow_step AND resource.labels.job_id=JOB_ID" --limit=50
