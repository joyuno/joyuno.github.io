1. Delta Lake란?

Delta Lake는 데이터 레이크 위에 ACID 트랜잭션, 확장 가능한 메타데이터 처리, 통합 배치/스트리밍 데이터 처리를 제공하는 오픈 소스 스토리지 레이어입니다. Parquet 파일을 기반으로 하며, 트랜잭션 로그(Delta Log)를 통해 데이터의 신뢰성과 일관성을 보장합니다.

ACID 트랜잭션

다중 작업 읽기/쓰기 시 데이터 무결성을 보장합니다. 동시성 제어로 충돌을 방지하고, 롤백이 가능하여 안전한 데이터 수정이 가능합니다.

스키마 진화 & 시행

자동 또는 명시적 스키마 진화를 지원하며, 스키마 시행으로 잘못된 데이터가 쓰이는 것을 방지합니다. Merge, Update, Delete 연산을 지원합니다.

타임 트래블

데이터 버전 관리를 통해 특정 시점(타임스탬프 또는 버전 번호)의 데이터를 쿼리할 수 있습니다. 데이터 롤백과 감사 추적에 유용합니다.

2. 왜 써야 하는가?

기존 데이터 레이크의 문제점(데이터 불일치, 쓰기 충돌, 낮은 품질)을 해결하면서도 데이터 웨어하우스 수준의 관리 기능을 제공합니다.

기능 기존 Parquet/ORC Delta Lake
ACID 보장 ❌ 없음 (파일 덮어쓰기) ✅ 완전 지원 (트랜잭션 로그)
Update/Delete/Merge ❌ 불가 (전체 재생성 필요) ✅ 네이티브 지원
스키마 관리 ⚠️ 수동 관리 ✅ 자동 진화 & 시행
데이터 버전 관리 ❌ 없음 ✅ 타임 트래블 (버전 히스토리)
성능 최적화 ⚠️ 제한적 (수동 최적화) ✅ Z-Ordering, 소규모 파일 컴팩션

3. 핵심 개념

Delta Lake의 핵심은 트랜잭션 로그(Delta Log)와 이를 기반으로 한 ACID 보장입니다. 모든 데이터 변경사항은 순서대로 로그에 기록되고, 이를 통해 일관된 뷰를 제공합니다.

┌─────────────────────────────────────────────────────────────┐ │ 애플리케이션 / Spark │ └───────────────────────────┬─────────────────────────────────┘ │ ┌───────────────────────────▼─────────────────────────────────┐ │ Delta Lake Layer │ │ ┌─────────────┐ ┌─────────────┐ ┌──────────────────┐ │ │ │ ACID │ │ Schema │ │ Time Travel │ │ │ │ Transactions│ │ Enforcement │ │ (Versioning) │ │ │ └─────────────┘ └─────────────┘ └──────────────────┘ │ │ ▲ │ └────────────────────┼────────────────────────────────────────┘ │ Delta Log (JSON/Parquet) ┌────────────────────▼────────────────────────────────────────┐ │ Cloud/Object Storage │ │ ┌──────────────────────────────────────────────────────┐ │ │ │ /delta_table/ │ │ │ │ ├── _delta_log/ ← 트랜잭션 로그 (버전 관리) │ │ │ │ │ ├── 00000000000000000000.json │ │ │ │ │ ├── 00000000000000000001.json │ │ │ │ │ └── ... │ │ │ │ ├── part-00001-xxx.parquet ← 데이터 파일 │ │ │ │ └── part-00002-yyy.parquet │ │ │ └──────────────────────────────────────────────────────┘ │ └─────────────────────────────────────────────────────────────┘
💡 핵심 포인트: Delta Log는 모든 변경사항의 단일 소스입니다. 읽기 작업 시 현재 스냅샷을 빠르게 구성하고, 쓰기 작업 시 직렬화 가능한 격리를 보장합니다.

4. 퀵스타트

4.1 설치 및 설정 (PySpark)

# PySpark 셸 또는 노트북에서
from pyspark.sql import SparkSession

# Delta Lake 패키지 포함하여 SparkSession 생성
spark = SparkSession.builder \
    .appName("DeltaQuickstart") \
    .config("spark.sql.extensions", "io.delta.sql.DeltaSparkSessionExtension") \
    .config("spark.sql.catalog.spark_catalog", "org.apache.spark.sql.delta.catalog.DeltaCatalog") \
    .getOrCreate()

# 또는 기존 SparkSession에 패키지 추가
# spark = SparkSession.builder.config("spark.jars.packages", "io.delta:delta-core_2.12:2.4.0").getOrCreate()

4.2 첫 Delta 테이블 생성

# 샘플 데이터 생성
data = [(1, "Alice", 3000, "Engineering"),
        (2, "Bob", 4000, "Sales"),
        (3, "Charlie", 3500, "Engineering")]
df = spark.createDataFrame(data, ["id", "name", "salary", "department"])

# Delta 테이블로 저장 (로컬 또는 S3/ADLS 경로)
delta_path = "/tmp/delta/employees"
df.write.format("delta").mode("overwrite").save(delta_path)

print(f"Delta 테이블 생성 완료: {delta_path}")

4.3 읽기, 업데이트, 타임트래블

# 읽기
df_read = spark.read.format("delta").load(delta_path)
df_read.show()

# 업데이트 (SQL 방식)
from delta.tables import *
deltaTable = DeltaTable.forPath(spark, delta_path)
deltaTable.update(
    condition = "department = 'Engineering'",
    set = {"salary": "salary + 500"}
)

# 버전 히스토리 확인
history = deltaTable.history()
history.select("version", "timestamp", "operation", "operationParameters").show()

# 타임트래블 (이전 버전 읽기)
df_v0 = spark.read.format("delta").option("versionAsOf", 0).load(delta_path)
df_v0.show()

5. 핵심 설정 레퍼런스

설정 기본값 설명 실무 권장값
spark.databricks.delta.properties.defaults.autoOptimize.autoCompact false 자동 소규모 파일 컴팩션 true (스트리밍 시나리오)
spark.databricks.delta.properties.defaults.autoOptimize.optimizeWrite false 쓰기 시 자동 최적화 (파일 크기 조정) true
delta.dataSkippingNumIndexedCols 32 데이터 스킵핑을 위한 인덱스 컬럼 수 10-20 (자주 필터링하는 컬럼)
delta.logRetentionDuration 30 days Delta Log 보관 기간 (타임트래블 가능 기간) 7-90 days (규정 준수에 따라)
delta.deletedFileRetentionDuration 1 week 삭제된 파일 보관 기간 (VACUUM 전) 7 days (안전한 롤백을 위해)
spark.sql.parquet.compression.codec snappy Parquet 압축 코덱 zstd (더 나은 압축률)
⚠️ 주의: VACUUM 명령은 deletedFileRetentionDuration 이전의 데이터를 물리적으로 삭제합니다. 타임트래블이 필요한 경우 보관 기간을 충분히 설정하세요.

6. 실무 팁 & 트러블슈팅

자주 겪는 문제와 해결책

문제 1: 소규모 파일 문제 (Small Files Problem)

증상: 수많은 작은 Parquet 파일로 인해 읽기 성능 저하 및 메타데이터 오버헤드 발생.

해결:

# 주기적 OPTIMIZE 실행
spark.sql("OPTIMIZE delta.`/path/to/table`")

# Z-Ordering으로 데이터 클러스터링 (자주 함께 조회되는 컬럼)
spark.sql("OPTIMIZE delta.`/path/to/table` ZORDER BY (date, user_id)")

# 쓰기 시 자동 최적화 활성화
spark.conf.set("spark.databricks.delta.properties.defaults.autoOptimize.optimizeWrite", "true")

문제 2: 스키마 불일치 (Schema Mismatch)

증상: 새 데이터의 스키마가 기존 테이블과 달라 쓰기 실패.

해결:

# 스키마 자동 진화 활성화 (쓰기 시)
df.write.format("delta").mode("append") \
    .option("mergeSchema", "true") \
    .save("/delta/table")

# 또는 기존 테이블에 컬럼 추가
spark.sql("ALTER TABLE delta.`/path/to/table` ADD COLUMNS (new_col STRING)")

문제 3: 동시 쓰기 충돌 (Concurrent Write Conflicts)

증상: 다수의 작업이 동시에 쓰기 시도 시 충돌 발생.

해결:

# Optimistic Concurrency Control 활용
# MERGE 구문을 사용한 업서트(upsert) 패턴 적용
deltaTable.alias("target").merge(
    sourceData.alias("source"),
    "target.id = source.id"
).whenMatchedUpdateAll() \
 .whenNotMatchedInsertAll() \
 .execute()

# 쓰기 재시도 로직 구현 (예: 3회 재시도)

문제 4: 메타데이터 성능 저하

증상: 수만 개 이상의 파티션 또는 장기 운영 시 메타데이터 처리 느려짐.

해결:

# 파티션 수 줄이기 (날짜별 → 월별)
# CHECKPOINT 주기 조정 (기본 10커밋)
spark.conf.set("spark.databricks.delta.checkpoint.partSize", 50)

# 주기적으로 로그 정리 (주의: 타임트래블 기간 단축)
spark.sql("VACUUM delta.`/path/to/table` RETAIN 168 HOURS")  # 7일만 보관
