1. Prometheus + Grafana란?

Prometheus는 오픈소스 시스템 모니터링 및 알람 툴킷입니다. Grafana는 메트릭 데이터를 시각화하는 분석 플랫폼입니다. 두 도구를 결합하면 메트릭 수집 → 저장 → 분석 → 시각화 → 알람의 완전한 모니터링 파이프라인을 구축할 수 있습니다.

메트릭 수집

Pull 모델 기반. 대상 시스템의 HTTP 엔드포인트를 스크래핑하여 메트릭을 수집합니다.

시계열 저장

수집된 데이터를 자체 시계열 데이터베이스에 저장하며, 효율적인 쿼리 언어(PromQL)를 제공합니다.

시각화 & 알람

Grafana를 통해 다채로운 대시보드 생성, Prometheus Alertmanager를 통해 알람 규칙 정의 및 전송.

2. 왜 써야 하는가?

다른 모니터링 솔루션과 비교한 Prometheus+Grafana의 강점.

솔루션 모델 데이터 모델 주요 강점 주요 단점
Prometheus Pull 다차원 시계열 높은 자율성, 강력한 쿼리 언어, Kubernetes 네이티브 장기 저장 부족, 샘플링만 가능
Graphite Push 계층적 시계열 간단한 아키텍처, 장기 데이터 다차원 라벨 지원 미약, 확장성 문제
InfluxDB Push/Pull 시계열 + 태그 고성능 쓰기, SQL-like 쿼리 클러스터 관리 복잡, 라이선스 제한
Datadog/New Relic Push (Agent) 다양한 메트릭 통합 플랫폼, AI 기반 분석 비용 높음, 벤더 종속
결론: 오픈소스, 커뮤니티 활발, Kubernetes 환경과의 자연스러운 통합, 강력한 쿼리 언어로 인해 현대 데브옵스 환경의 사실상 표준입니다.

3. 핵심 개념

아키텍처 다이어그램

+-------------------+ +-------------------+ +-------------------+ | Target Systems | | Prometheus | | Grafana | | (Apps, Nodes) |<-----| (Server + TSDB) |<-----| (Dashboard) | | | | | | | | /metrics 엔드포인트 | | Scrape / Store | | Query / Visualize | +-------------------+ +-------------------+ +-------------------+ ^ ^ ^ | | | +-------------------+ +-------------------+ +-------------------+ | Exporters | | Alertmanager | | Data Sources | | (node_exporter) | | (Alerts Routing)| | (Prom, Loki) | +-------------------+ +-------------------+ +-------------------+

핵심 용어

  • 메트릭(Metric): 측정값 (예: http_requests_total).
  • 라벨(Label): 메트릭의 차원 (예: method="POST", status="200").
  • 샘플(Sample): 특정 시간의 메트릭 값 (타임스탬프 + 값).
  • 스크래핑(Scraping): Prometheus가 대상의 /metrics 엔드포인트에서 데이터를 가져오는 과정.
  • 익스포터(Exporter): 비-Prometheus 시스템의 메트릭을 Prometheus 형식으로 변환하는 도구.
  • PromQL: Prometheus Query Language. 시계열 데이터를 쿼리하는 강력한 언어.
  • 알람 규칙(Alerting Rule): 특정 조건을 만족하면 알람을 생성하는 Prometheus 규칙.

4. 퀵스타트

단계 1: Prometheus 설치 (Docker)

docker run -d \
  -p 9090:9090 \
  -v /path/to/prometheus.yml:/etc/prometheus/prometheus.yml \
  --name prometheus \
  prom/prometheus

단계 2: 기본 설정 파일 (prometheus.yml)

global:
  scrape_interval:15s

scrape_configs:
  - job_name: 'prometheus'
    static_configs:
      - targets: ['localhost:9090']

  - job_name: 'node_exporter'
    static_configs:
      - targets: ['localhost:9100']

단계 3: Node Exporter 실행 (시스템 메트릭)

docker run -d \
  -p 9100:9100 \
  --name node_exporter \
  prom/node-exporter

단계 4: Grafana 설치 (Docker)

docker run -d \
  -p 3000:3000 \
  --name grafana \
  grafana/grafana

단계 5: Grafana 설정

1. 브라우저에서 http://localhost:3000 접속 (초기 로그인: admin/admin).

2. "Configuration → Data Sources → Add data source → Prometheus" 선택.

3. URL에 http://localhost:9090 입력 후 Save & Test.

단계 6: Python 애플리케이션 메트릭 노출 예제

from prometheus_client import start_http_server, Counter, Gauge
import random
import time

REQUEST_COUNT = Counter('app_requests_total', 'Total app requests')
CPU_USAGE = Gauge('app_cpu_usage', 'Simulated CPU usage')

def process_request():
    REQUEST_COUNT.inc()
    CPU_USAGE.set(random.uniform(0.5, 1.5))

if __name__ == '__main__':
    start_http_server(8000)  # /metrics 엔드포인트를 8000포트에서 노출
    while True:
        process_request()
        time.sleep(2)

위 애플리케이션을 실행 후, prometheus.yml에 새로운 job을 추가:

  - job_name: 'python_app'
    static_configs:
      - targets: ['localhost:8000']
확인: Prometheus 웹 UI (http://localhost:9090)에서 "Status → Targets" 페이지를 확인하여 모든 대상이 UP 상태인지 확인하세요.

5. 핵심 설정 레퍼런스

구성 요소 설정 키 설명 예시 값
Global scrape_interval 스크래핑 간격 15s
Global evaluation_interval 알람 규칙 평가 간격 30s
Scrape Config job_name 작업 식별자 'kubernetes-pods'
Scrape Config metrics_path 메트릭 엔드포인트 경로 /metrics
Scrape Config static_configs / targets 스크래핑 대상 목록 ['10.0.1.1:9100', '10.0.1.2:9100']
Alerting rule_files 알람 규칙 파일 경로 ['/etc/prometheus/alerts.yml']
Alerting alertmanagers Alertmanager 대상 static_configs: targets: ['localhost:9093']
Storage retention 데이터 보존 기간 15d

6. 실무 팁 & 트러블슈팅

자주 겪는 문제

문제 1: Prometheus 대상이 DOWN 상태

원인: 네트워크 문제, 대상 서비스 중단, 스크래핑 경로 오류.

해결: Prometheus UI의 Targets 페이지에서 에러 메시지 확인. 대상의 /metrics 엔드포인트 직접 접속 테스트. firewall/network policy 확인.

문제 2: 메트릭이 Grafana에 표시되지 않음

원인: 데이터 소스 연결 오류, 시간 범위 설정 문제, PromQL 쿼리 오류.

해결: Grafana 데이터 소스 설정의 "Save & Test" 확인. 대시보드 시간 범위를 확장. Prometheus UI에서 직접 같은 쿼리 실행하여 데이터 존재 확인.

문제 3: 높은 메모리 사용량

원인: 너무 많은 메트릭/라벨 수집, 긴 보존 기간, WAL 블록 축적.

해결: 스크래핑 간격 증가, 불필요한 메트릭 필터링(relabel_configs), 보존 기간 조정.

실무 팁

  • 라벨 설계: 라벨은 메트릭의 차원을 정의합니다. 과도한 라벨(high cardinality)은 성능을 급격히 저하시킵니다. 필요한 차원만 정의하세요.
  • 알람 설계: 알람은 상태(Alerting, OK, Pending)를 가지며, 중요한 알람만 정의하고 알람 그룹핑을 활용하세요.
  • Grafana 대시보드: 재사용 가능한 대시보드를 템플릿 변수(Variables)와 함께 설계하여 팀 내 공유하세요.
  • 장기 저장: Prometheus는 장기 저장에 적합하지 않습니다. Thanos 또는 Cortex를 사용하여 장기 저장 및 글로벌 뷰를 구축하세요.
  • Kubernetes 통합: Prometheus Operator를 사용하여 Kubernetes 환경에서의 배포와 관리를 간소화하세요.
