1. Kafka Streams란?

Apache Kafka의 클라이언트 라이브러리로, Kafka 클러스터를 입력 소스와 출력 대상으로 사용하여 스트림 처리 애플리케이션을 구축할 수 있게 해주는 경량 프레임워크입니다. 별도의 처리 클러스터 없이 표준 Java/Scala 애플리케이션으로 실행됩니다.

1. 상태 저장 처리

윈도우링, 집계, 조인 연산을 위한 내부 상태 저장소(RocksDB)를 제공합니다. 키별 상태를 로컬 디스크에 유지하여 고성능 처리와 장애 복구를 가능하게 합니다.

2. Exactly-Once 의미론

트랜잭션 프로듀서/컨슈머와 통합되어 메시지 처리의 정확히 한 번(exactly-once) 전달을 보장합니다. 데이터 중복이나 유실 없이 안전한 스트림 처리가 가능합니다.

3. 탄력적 확장

애플리케이션 인스턴스를 추가하면 Kafka의 컨슈머 그룹 프로토콜을 통해 파티션 할당이 자동으로 재조정됩니다. 상태는 신규 인스턴스로 마이그레이션되어 무중단 확장이 가능합니다.

2. 왜 써야 하는가? (기존 대안과 비교)

스트림 처리에는 여러 프레임워크가 존재합니다. Kafka Streams는 다음과 같은 이유로 선택됩니다.

솔루션 장점 단점 / Kafka Streams 비교 포인트 적합한 Use Case
Kafka Streams 설치/운영 간소화, Exactly-Once, 상태 관리 내장, Kafka와 긴밀 통합 Java/Scala에 제한됨, 복잡한 DAG 표현이 상대적 제약 Kafka 중심의 실시간 ETL, 실시간 집계, 이벤트 기반 마이크로서비스
Apache Flink 고급 창 및 상태 관리, 배치/스트림 통합, 복잡한 DAG 지원 별도 클러스터 운영 필요, 설정/관리 복잡도 높음 초저지연 복잡 이벤트 처리(CEP), 대규모 상태 처리
Apache Spark Streaming 배치 처리와의 통일성, 머신러닝 라이브러리(MLlib) 활용 마이크로배치 방식으로 인한 지연, Exactly-Once 보장 복잡 배치와 스트림 로직 통합, 기존 Spark 환경 활용
직접 Consumer/Producer 구현 완전한 유연성 상태 관리, 재처리, Exactly-Once 등 모든 것을 직접 구현해야 함 매우 단순한 라우팅 또는 변환 작업

결론: 언제 선택하는가?

이미 Kafka 생태계를 사용 중이고, 운영 오버헤드를 최소화하면서도 상태 저장 처리와 Exactly-Once 보장이 필요한 실시간 데이터 파이프라인 또는 이벤트 기반 애플리케이션을 구축할 때 Kafka Streams가 이상적입니다.

3. 핵심 개념

아키텍처 개요

Kafka Streams 애플리케이션은 토폴로지(처리 논리)를 정의하고, 이를 태스크로 분해하여 실행합니다. 각 태스크는 입력 토픽의 특정 파티션 세트를 처리하며 로컬 상태 저장소를 가집니다.

Application Instance 1 (StreamThread-1) Application Instance 2 (StreamThread-1) +-----------------------------------+ +-----------------------------------+ | Topology: | | Topology: | | Source (Topic A) | | Source (Topic A) | | | | | | | | Processor (Map/Filter/Aggregate) | | Processor (Map/Filter/Aggregate) | | | | | | | | Sink (Topic B) | | Sink (Topic B) | +-----------------------------------+ +-----------------------------------+ | Tasks: | | Tasks: | | - Task 0 (Partitions A-0, A-2) | | - Task 1 (Partitions A-1, A-3) | | State Store: RocksDB-0 | | State Store: RocksDB-1 | +------------------+----------------+ +------------------+----------------+ | | +------------------+-----------------------+ | +-------v--------+ | Kafka Cluster | | - Topic A | | - Topic B | | - __consumer_offsets | | - __transaction_state | +-----------------+

주요 용어

  • KStream: 레코드의 무한한 스트림을 나타냅니다. (키-값 쌍의 연속)
  • KTable: 변경 로그 스트림에서 유도된 테이블(물질화된 뷰)입니다. 키의 최신 값만 유지합니다.
  • GlobalKTable: 모든 태스크에 완전히 복제되는 KTable입니다. 조인 시 브로드캐스트 용도.
  • Processor API: 저수준 API로, 상태 저장소 접근, 커스텀 로직 구현에 사용.
  • DSL (Domain Specific Language): 고수준 API로, map, filter, aggregate, join 등의 연산자 제공.
  • State Store: 키-값, 창, 세션 저장소 등. 기본적으로 RocksDB 사용.

4.1. 의존성 추가 (Python - kafka-python 활용 예시)

Kafka Streams는 Java/Scala 라이브러리이지만, Python 환경에서는 Confluent의 kafka-python과 Faust 같은 라이브러리로 유사한 패턴을 구현할 수 있습니다. 여기서는 개념 설명을 위해 Python 유사 코드를 제시합니다.

pom.xml (Java 예시) / requirements.txt (Python 유사 환경)

    <groupId>org.apache.kafka</groupId>
    <artifactId>kafka-streams</artifactId>
    <version>3.6.0</version>
</dependency>
# Python (Faust를 사용한 스트림 처리 예시)
faust
confluent-kafka

4.4. 실행 및 테스트 CLI 명령어

Kafka 토픽 생성 및 데이터 주입
# Kafka 토픽 생성 (Confluent CLI 또는 kafka-topics.sh)
kafka-topics.sh --create --topic text-input-topic \
  --bootstrap-server localhost:9092 --partitions 2 --replication-factor 1
kafka-topics.sh --create --topic word-count-output-topic \
  --bootstrap-server localhost:9092 --partitions 2 --replication-factor 1

# 테스트 데이터 프로듀스
echo "hello world kafka streams" | kafka-console-producer.sh \
  --topic text-input-topic --bootstrap-server localhost:9092
echo "streams processing is powerful" | kafka-console-producer.sh \
  --topic text-input-topic --bootstrap-server localhost:9092

# 결과 컨슈밍
kafka-console-consumer.sh --topic word-count-output-topic \
  --from-beginning --bootstrap-server localhost:9092 \
  --formatter kafka.tools.DefaultMessageFormatter \
  --property print.key=true --property print.value=true

5. 핵심 설정 레퍼런스

설정 기본값 설명 실무 권장값
application.id (필수) 애플리케이션 식별자. 컨슈머 그룹 ID 및 상태 저장소 디렉토리 이름으로 사용. 비즈니스 의미를 담은 명칭 (e.g., fraud-detection-service)
bootstrap.servers (필수) 연결할 Kafka 브로커 목록. 프로덕션은 3개 이상 브로커 콤마 구분 (e.g., broker1:9092,broker2:9092)
num.stream.threads 1 애플리케이션 인스턴스 내 처리 스레드 수. 입력 토픽의 총 파티션 수를 고려하여 설정 (일반적으로 파티션 수 ≤ 스레드 수)
state.dir /tmp/kafka-streams 로컬 상태 저장소(RocksDB) 디렉토리. 전용 SSD 경로 지정 (e.g., /data/kafka-streams-state)
processing.guarantee at_least_once 처리 보장 수준. Exactly-Once가 필요하면 exactly_once_v2로 설정 (성능 일부 트레이드오프)
commit.interval.ms 30000 오프셋 커밋 간격(ms). Exactly-Once 사용 시 기본값 유지. 저지연 필요시 줄일 수 있으나 부하 증가.
default.deserialization.exception.handler LogAndFail 역직렬화 실패 처리 핸들러. 데이터 품질 이슈 많을 시 LogAndContinue로 변경하여 파이프라인 중단 방지

6. 실무 팁 & 트러블슈팅

자주 겪는 문제 + 해결책

문제 1: 상태 저장소 디렉토리 접근 오류 또는 디스크 부족

증상: 애플리케이션 시작 실패 또는 태스크 마이그레이션 실패.

해결:

  • state.dir을 용량이 충분한 전용 디스크로 설정.
  • 다른 애플리케이션과 디렉토리 공유하지 않기.
  • 정기적으로 애플리케이션을 안전히 종료 후 상태 디렉토리 백업/정리 (운영 중 삭제 금지).

문제 2: 재처리(Reprocessing) 시 예상과 다른 결과

증상: 오프셋을 처음부터 재처리하면 기존 결과와 다른 값이 출력됨.

해결:

  • 입력 데이터에 키가 없거나 변경될 수 있는지 확인. KTable은 동일 키의 최신 값만 유지.
  • 상태 저장소를 초기화하고 재처리: 애플리케이션 종료 후 state.dir 삭제 (주의: 운영 데이터 손실 가능).
  • 변경 불가능한(immutable) 키 설계를 고려.

문제 3: 처리 지연(Lag) 증가

증상: 출력이 느리거나 컨슈머 랙이 계속 증가.

해결:

  • 병렬성 확인: num.stream.threads를 입력 토픽 파티션 수 이상으로 증가.
  • 로직 최적화: 상태 저장소 접근(조회/업데이트)이 병목인지 프로파일링.
  • 브로커/네트워크: 브로커 리소스, 네트워크 지연 확인.
  • Exactly-Once 비활성화 테스트: processing.guarantee를 at_least_once로 변경하여 성능 비교.

팁 1: 운영 모니터링 필수 항목

  • 메트릭: consumer lag, process-rate, commit-rate, state-store metrics (hit-ratio, size).
  • 로그: rebalance 로그, 역직렬화 예외 로그 주시.
  • 상태 저장소 디스크 사용량: 지속적 모니터링.

팁 2: 애플리케이션 롤링 업그레이드

  1. 새 버전 JAR 준비.
  2. 인스턴스 하나씩 순차적으로 종료(kill -15) → 기다림(상태 마이그레이션) → 새 버전 시작.
  3. Kafka의 리밸런싱 프로토콜이 상태를 새 인스턴스로 자동 전송.
  4. 모든 인스턴스 교체 완료 후 검증.
