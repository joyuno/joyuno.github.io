1. Triton Inference Server란?

Triton Inference Server는 NVIDIA에서 개발한 오픈소스 추론 서빙 솔루션입니다. TensorFlow, PyTorch, ONNX Runtime, TensorRT 등 다양한 프레임워크와 백엔드에서 훈련된 모델을 통합적으로 서빙할 수 있는 플랫폼을 제공합니다. 클라우드, 데이터센터, 엣지 및 임베디드 환경에서 고성능 추론을 위한 설계되었습니다.

다중 프레임워크 지원

단일 서버에서 TensorFlow, PyTorch, ONNX, TensorRT, OpenVINO, Python, DALI 등 다양한 백엔드를 동시에 실행할 수 있습니다. 모델 재훈련 시 프레임워크 전환 비용이 최소화됩니다.

고성능 추론

동시성 모델(Concurrent Model Execution), 동적 배칭(Dynamic Batching), 모델 병렬화(Model Parallelism)를 통해 GPU 활용도를 극대화하고 처리량(Throughput)을 높입니다.

생산 환경 준비

gRPC/HTTP 엔드포인트, 상태 점검, 모델 버전 관리, 모니터링(Metrics), Kubernetes 통합 등 MLOps 파이프라인에 바로 적용 가능한 기능을 내장하고 있습니다.

2. 왜 써야 하는가?

기존의 단순한 모델 서빙 방식(TF Serving, TorchServe 단독)이나 직접 구현한 API 서버와 비교했을 때 Triton의 차별화된 가치를 확인할 수 있습니다.

솔루션 다중 프레임워크 동적 배칭 모델 분석기 K8s 네이티브 적합 시나리오
Triton ✅ ✅ ✅ ✅ 복잡한 멀티프레임워크 환경, 고성능 추론
TensorFlow Serving ❌ (TF Only) ⭕ (제한적) ❌ ⭕ TensorFlow 모델 전용 환경
TorchServe ❌ (PyTorch Only) ✅ ❌ ✅ PyTorch 모델 전용 환경
직접 구현 (FastAPI/Flask) ✅ ❌ ❌ ⭕ 단순 PoC, 제한된 트래픽

결론: 여러 프레임워크의 모델을 통합 관리하고, GPU 자원을 최대한 활용해야 하는 프로덕션 환경에서는 Triton이 명확한 이점을 가집니다.

3. 핵심 개념

Triton의 아키텍처는 모델, 스케줄러, 백엔드로 구성된 모듈식 설계를 따릅니다. 클라이언트 요청은 스케줄러를 통해 적절한 백엔드로 라우팅되고, 백엔드는 실제 모델을 실행합니다.

┌─────────────────────────────────────────────────────────────────────┐ │ Triton Inference Server │ ├─────────────────┬─────────────────┬─────────────────┬──────────────┤ │ HTTP/gRPC │ Metrics │ Health/Ready │ Logging │ │ Endpoints │ Endpoint │ Endpoints │ │ └─────────────────┴────────┬────────┴─────────────────┴──────────────┘ │ ┌──────────────▼──────────────┐ │ Inference Scheduler │ │ ┌────────────────────────┐ │ │ │ Dynamic Batching │ │ │ │ Sequence Batcher │ │ │ │ Ensemble Scheduler │ │ │ └────────────────────────┘ │ └──────────────┬──────────────┘ │ ┌──────────────────┼──────────────────┐ │ │ │ ┌───────▼──────┐ ┌────────▼───────┐ ┌───────▼──────┐ │ Backend │ │ Backend │ │ Backend │ │ TensorRT │ │ PyTorch │ │ ONNX │ │ │ │ │ │ Runtime │ └───────┬──────┘ └────────┬───────┘ └───────┬──────┘ │ │ │ ┌───────▼──────┐ ┌────────▼───────┐ ┌───────▼──────┐ │ Model │ │ Model │ │ Model │ │ Repository │ │ Repository │ │ Repository │ │ (/models) │ │ (/models) │ │ (/models) │ └──────────────┘ └────────────────┘ └──────────────┘

주요 컴포넌트

  • 모델 저장소(Model Repository): 서빙할 모델과 설정 파일이 저장되는 디렉토리 구조.
  • 백엔드(Backend): 특정 프레임워크(TensorRT, PyTorch 등)를 실행하는 런타임 엔진.
  • 스케줄러(Scheduler): 요청을 배치하고 백엔드로 전달하는 정책 관리자 (동적 배칭, 시퀀스 배칭).
  • 엔드포인트: HTTP/REST (포트 8000), gRPC (포트 8001), Metrics (포트 8002)를 제공.

4. 퀵스타트

Docker를 이용한 가장 빠른 시작 방법입니다. 5분 내에 첫 추론을 실행해봅니다.

Step 1: Docker 컨테이너 실행

CLI
# 최신 Triton 서버 컨테이너 실행
docker run --gpus=all --rm -p 8000:8000 -p 8001:8001 -p 8002:8002 \
  -v /path/to/your/model/repository:/models \
  nvcr.io/nvidia/tritonserver:24.01-py3 \
  tritonserver --model-repository=/models

`/path/to/your/model/repository`를 실제 모델 저장소 경로로 변경하세요.

Step 2: 모델 저장소 준비 (예: ONNX 모델)

디렉토리 구조
model_repository/
└── resnet50_onnx
    ├── 1
    │   └── model.onnx
    └── config.pbtxt
config.pbtxt (최소 구성)
name: "resnet50_onnx"
platform: "onnxruntime_onnx"
max_batch_size: 8
input [
  {
    name: "input"
    data_type: TYPE_FP32
    dims: [ 3, 224, 224 ]
  }
]
output [
  {
    name: "output"
    data_type: TYPE_FP32
    dims: [ 1000 ]
  }
]

Step 3: Python 클라이언트로 추론 요청

Python (tritonclient 사용)
import tritonclient.http as httpclient
import numpy as np

# 클라이언트 연결
client = httpclient.InferenceServerClient(url="localhost:8000")

# 입력 데이터 준비 (예: 더미 이미지)
input_data = np.random.randn(1, 3, 224, 224).astype(np.float32)

# 요청 생성
inputs = [httpclient.InferInput("input", input_data.shape, "FP32")]
inputs[0].set_data_from_numpy(input_data)
outputs = [httpclient.InferRequestedOutput("output")]

# 추론 실행
response = client.infer(model_name="resnet50_onnx", inputs=inputs, outputs=outputs)

# 결과 확인
result = response.as_numpy("output")
print(f"추론 결과 shape: {result.shape}")
print(f"예측 클래스: {np.argmax(result)}")

`pip install tritonclient[http] numpy`로 클라이언트 라이브러리를 설치하세요.

5. 핵심 설정 레퍼런스

모델 설정 파일(`config.pbtxt`)에서 자주 사용하는 주요 옵션들입니다.

옵션 기본값 설명 예시
platform - (필수) 사용할 백엔드 플랫폼 "onnxruntime_onnx", "pytorch_libtorch"
max_batch_size 0 (배치 비활성화) 동적 배칭 시 최대 배치 크기 32
instance_group count:1, KIND:GPU 모델 인스턴스 배치 방식 { count: 2, kind: KIND_GPU }
dynamic_batching 비활성화 동적 배칭 설정 { preferred_batch_size: [4,8], max_queue_delay_microseconds: 1000 }
optimization - 실행 최적화 설정 { cuda { graphs: true } }
model_warmup - 서버 시작 시 모델 워밍업 [ { batch_size: 1, inputs: {...} } ]

동적 배칭 설정 예시: 지연 시간을 희생하여 처리량을 극대화할 때 유용합니다. `max_queue_delay_microseconds`를 조절하여 latency-throughput 트레이드오프를 관리하세요.

6. 실무 팁 & 트러블슈팅

자주 겪는 문제

문제 1: 모델 로드 실패 (UNSUPPORTED)

증상: "Unsupported model platform ..." 에러.

원인: `config.pbtxt`의 `platform` 값이 잘못되었거나 해당 백엔드가 컨테이너에 포함되지 않음.

해결: 1. 공식 문서로 정확한 platform 문자열 확인.
2. 필요한 백엔드가 포함된 컨테이너 태그 사용 (예: `-py3` 태그는 Python 백엔드 포함).

문제 2: GPU 메모리 부족 (OOM)

증상: CUDA out of memory 에러.

원인: 너무 많은 모델 인스턴스 또는 큰 `max_batch_size` 설정.

해결: 1. `instance_group`의 `count`를 줄이기.
2. `max_batch_size`를 낮추기.
3. `optimization { cuda { graphs: true } }`로 메모리 재사용 활성화.

문제 3: 높은 지연 시간(Latency)

증상: 개별 요청 처리 시간이 너무 김.

원인: 동적 배칭 대기 시간, 작은 배치 크기, CPU-GPU 데이터 전송 병목.

해결: 1. `dynamic_batching`의 `max_queue_delay_microseconds`를 줄이기.
2. 클라이언트에서 배치 요청을 보내기.
3. 입력 데이터를 GPU 메모리에 유지(Zero-copy).

성능 최적화 팁

  • 모델 분석기 사용: `perf_analyzer` 도구로 병목 지점을 찾고 최적의 배치 크기를 결정하세요.
  • TensorRT 백엔드: 가능하면 모델을 TensorRT로 변환하여 최대 성능을 끌어내세요.
  • 모니터링: Prometheus 메트릭(포트 8002)을 수집하여 리소스 사용량과 처리량을 지속적으로 관찰하세요.
  • Kubernetes 배포: NVIDIA Kubernetes Device Plugin과 함께 사용하여 GPU 리소스 관리를 효율화하세요.

주의: 프로덕션 배포 전 반드시 부하 테스트를 수행하세요. Triton의 동적 배칭은 트래픽 패턴에 민감하게 반응합니다.
