1. Fivetran란?

Fivetran은 완전 관리형(Managed) ELT(Extract, Load, Transform) 데이터 통합 플랫폼입니다. SaaS, 데이터베이스, 파일 시스템 등 다양한 소스의 데이터를 자동으로 추출하여 데이터 웨어하우스나 데이터 레이크에 표준화된 형식으로 로드하는 서비스를 제공합니다.

완전 관리형

인프라 관리, 커넥터 유지보수, 스키마 변경 감지 및 동기화, API 변경 대응 등을 Fivetran이 전담합니다. 사용자는 설정과 모니터링에만 집중.

자동 스키마 관리

소스 시스템의 컬럼 추가/삭제/변경을 자동으로 감지하고 대상 데이터 웨어하우스의 스키마를 동기화합니다. 수동 DDL 작업 불필요.

변경 데이터 캡처(CDC)

로그 기반 CDC를 지원하여 소스 시스템의 부하를 최소화하면서도 실시간에 가까운 데이터 동기화를 가능하게 합니다.

2. 왜 써야 하는가?

기존의 자체 구축 파이프라인 또는 오픈소스 도구와 비교했을 때 Fivetran의 핵심 가치는 운영 부담의 제거와 시간 대비 가치에 있습니다.

비교 항목 Fivetran 자체 구축 (Airflow + 커스텀 코드) 오픈소스 (Airbyte, Singer)
초기 구축 시간 몇 시간 ~ 몇 일 몇 주 ~ 몇 달 몇 일 ~ 몇 주
운영 부담 매우 낮음 (관리형) 매우 높음 (모든 인프라/코드 관리) 중간 ~ 높음 (셀프 호스팅 관리)
커넥터 유지보수 Fivetran 담당 팀 내부 담당 커뮤니티/팀 내부 담당
스키마 변경 대응 자동 동기화 수동 대응 필요 커넥터에 따라 다름
비용 구조 사용량 기반 (월간 활성 행수) 인프라 비용 + 인건비 인프라 비용 + (인건비)
💡 선택 가이드: 데이터 팀의 인원이 적고, 빠른 시간 내에 안정적인 데이터 수집 파이프라인을 구축해야 하며, 예산이 허용된다면 Fivetran은 탁월한 선택입니다. 반면, 커스터마이징이 매우 많거나, 비용에 매우 민감한 경우 다른 대안을 고려하세요.

3. 핵심 개념

Fivetran의 아키텍처는 사용자 대신 데이터를 이동시키는 "관리형 파이프라인"입니다.

┌─────────────────┐ ┌─────────────────┐ ┌─────────────────┐ │ │ │ │ │ │ │ Source │ │ Fivetran │ │ Destination │ │ (e.g., SaaS, │────▶ Cloud │────▶ (e.g., │ │ Database, │ │ Infrastructure│ │ Snowflake, │ │ File) │ │ │ │ BigQuery, │ │ │ │ │ │ Redshift) │ └─────────────────┘ └─────────────────┘ └─────────────────┘ │ │ │ │ │ │ ┌───────▼─────────────────────────▼─────────────────────────▼───────┐ │ │ │ User Configuration & Monitoring │ │ (Fivetran Dashboard / API) │ │ │ └───────────────────────────────────────────────────────────────────┘

주요 컴포넌트

  • 커넥터(Connector): 특정 소스(예: Salesforce, PostgreSQL) 또는 대상(예: Snowflake)과의 연결을 담당하는 구성 요소.
  • 동기화(Sync): 소스에서 데이터를 추출하여 대상에 로드하는 단일 작업. 초기 동기화(Historical)와 증분 동기화(Incremental)로 구분.
  • 스키마(Schema): 대상 데이터 웨어하우스에서 생성되는 테이블 구조. 소스의 테이블/오브젝트와 매핑.
  • 워크스페이스(Workspace): 커넥터, 사용자, 설정을 그룹화하는 조직 단위. 일반적으로 회사 또는 부서별로 생성.
  • 로그(Logs): 각 동기화 작업의 상태, 지연 시간, 처리된 행 수 등 상세 정보.

4. 퀵스타트

Fivetran 대시보드와 API를 사용한 첫 번째 커넥터 설정부터 데이터 확인까지의 과정입니다.

단계 1: 계정 생성 및 대상 설정

Fivetran 대시보드에 로그인 후, 데이터를 로드할 대상(예: Snowflake)을 먼저 설정합니다.

단계 2: 커넥터 생성 (PostgreSQL 예제)

대시보드에서 "Connectors" → "Add Connector" → "PostgreSQL" 선택 후 소스 정보 입력.

# 필요한 경우 소스 DB에 복제 권한 부여 (CDC용)
ALTER USER fivetran_user REPLICATION;

단계 3: Python SDK를 이용한 커넥터 관리

# pip install fivetran_api
import os
from fivetran_api import FivetranAPI

# API 키 설정 (환경 변수 권장)
api_key = os.getenv('FIVETRAN_API_KEY')
api_secret = os.getenv('FIVETRAN_API_SECRET')
client = FivetranAPI(api_key, api_secret)

# 커넥터 목록 조회
connectors = client.list_connectors(group_id='your_workspace_id')
for connector in connectors['data']['items']:
    print(f"ID: {connector['id']}, Service: {connector['service']}, Status: {connector['status']['setup_state']}")

# 특정 커넥터 동기화 트리거
sync_response = client.force_sync_connector(connector_id='your_connector_id')
print(f"Sync triggered: {sync_response['code']}")

단계 4: CLI를 이용한 상태 확인

# Fivetran CLI 설치 및 설정 (공식 문서 참고)
# 커넥터 상태 확인
fivetran connector status --id your_connector_id

# 동기화 로그 조회
fivetran connector logs --id your_connector_id --limit 5

단계 5: 데이터 확인 (Snowflake SQL 예제)

-- Fivetran이 생성한 스키마에서 데이터 확인
SELECT *
FROM your_database.fivetran_schema.your_source_table
LIMIT 10;

-- 동기화 상태 메타데이터 확인 (사용 가능한 경우)
SELECT *
FROM fivetran_audit.database.sync_log
ORDER BY event_time DESC;

5. 핵심 설정 레퍼런스

설정 항목 설명 권장값/팁
Sync Frequency 동기화 실행 빈도 실시간성 요구사항에 따라 5분 ~ 24시간. 초기에는 30분으로 시작.
Historical Sync 초기 전체 데이터 로드 대용량 소스는 기간을 나눠서(paused) 실행하여 대상 DB 부하 관리.
Schema Change Handling 소스 스키마 변경 시 동작 기본값 "AUTO" 권장. 중요한 테이블은 "NOTIFY"로 먼저 알림 받기.
Destination Prefix 대상 스키마 이름 접두사 소스 구분을 위해 사용 (예: src_, stg_).
Data Time Travel 대상이 지원할 경우 히스토리 데이터 보관 Snowflake Time Travel, BigQuery 변경 내역과 연동 권장.
Network Policies Fivetran IP 화이트리스트 소스/대상 방화벽에 반드시 Fivetran IP 범위 추가 필요.

6. 실무 팁 & 트러블슈팅

⚠️ 자주 겪는 문제 1: 동기화 지연(Sync Lag)

증상: 대시보드에서 Sync Status가 "Delayed" 또는 로그에서 "Behind by X hours" 표시.

원인 및 해결:

  • 소스 시스템 부하: 소스 DB의 CPU/IO 부하 확인. 복제 슬롯 부족 시 확장.
  • 대상 웨어하우스 성능: Snowflake/BigQuery의 웨어하우스 크기 업스케일 고려.
  • 너무 큰 트랜잭션: 소스에서 장시간 실행되는 트랜잭션은 CDC를 블록할 수 있음.

임시 조치: 커넥터를 pause 후 resume하여 재시작 (마지막 체크포인트부터 재개).

⚠️ 자주 겪는 문제 2: 중복 데이터

증상: 대상 테이블에 동일한 PK를 가진 행이 여러 개 존재.

원인: 대부분 소스에서의 논리적 삭제 후 재생성, 또는 커넥터의 재설정(reset) 후 증분 키 문제.

해결:

  • Fivetran이 제공하는 _fivetran_deleted 또는 _fivetran_synced 컬럼을 사용해 중복 제거 뷰 생성.
  • 소스 시스템의 키 변경 패턴을 이해하고, 필요시 커넥터를 완전 리셋 후 재동기화 (Historical Sync).
💡 실무 팁 1: 비용 최적화

Fivetran 비용은 월간 활성 행(Monthly Active Rows, MAR)로 책정됩니다.

  • 필요한 테이블만 동기화: 커넥터 설정에서 불필요한 테이블은 제외.
  • 증분 동기화 최적화: 가능한 모든 테이블이 증분 동기화 방식으로 설정되었는지 확인.
  • 자주 변경되지 않는 데이터: 동기화 빈도를 낮춤 (예: 일 1회).
💡 실무 팁 2: 모니터링 자동화

API와 웹훅을 활용해 모니터링을 구축하세요.

# 간단한 동기화 실패 감지 스크립트 예제
import requests
import smtplib

def check_connector_status(api_key, api_secret, connector_id):
    url = f"https://api.fivetran.com/v1/connectors/{connector_id}"
    auth = (api_key, api_secret)
    resp = requests.get(url, auth=auth).json()
    status = resp['data']['status']['setup_state']
    sync_state = resp['data']['status']['sync_state']
    
    if sync_state == 'failed' or status == 'broken':
        # 알림 발송 (Slack, Email, PagerDuty 등)
        send_alert(f"Fivetran Connector {connector_id} 실패: {status}/{sync_state}")
    return resp
🚨 주의: 커넥터 리셋(Reset)

커넥터 리셋은 대상의 모든 데이터를 삭제하고 처음부터 전체 동기화를 다시 시작합니다. 매우 신중하게 진행해야 합니다. 반드시 리셋 전에 대상 데이터를 백업하거나, 중요한 다운스트림 프로세스가 중단되도록 조치하세요.
