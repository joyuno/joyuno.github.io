1. Amazon Redshift란?

Amazon Redshift는 AWS에서 제공하는 완전관리형 페타바이트 규모의 클라우드 데이터 웨어하우스 서비스입니다. Massively Parallel Processing(MPP) 아키텍처를 기반으로 하여, 복잡한 분석 쿼리를 고속으로 처리하도록 설계되었습니다.

핵심 가치: 기존 온프레미스 데이터 웨어하우스 대비 1/10 비용, 확장성, 완전 관리형 운영, SQL 및 BI 도구와의 완벽한 호환성.

3대 핵심 능력

분산 처리 (MPP)

쿼리를 여러 노드에서 병렬 실행하여 성능을 극대화합니다. 리더 노드가 쿼리를 계획하고, 컴퓨트 노드가 데이터를 분산 저장/처리합니다.

컬럼형 저장

데이터를 행(Row)이 아닌 컬럼(Column) 단위로 저장합니다. 분석 쿼리에서 필요한 컬럼만 스캔하여 I/O를 획기적으로 줄이고 압축 효율을 높입니다.

완전 관리형

하드웨어 프로비저닝, 패치, 백업, 모니터링 등 운영 부담을 AWS가 담당합니다. 사용자는 비즈니스 로직과 쿼리 최적화에 집중할 수 있습니다.

2. 왜 써야 하는가? (기존 대안과 비교)

다양한 데이터 저장소와의 비교를 통해 Redshift의 적절한 사용 사례를 이해합니다.

서비스/제품 주요 용도 Redshift 대비 장단점 선택 가이드
Amazon Redshift 대규모 배치 분석, BI 리포트, 복잡한 조인/집계 장점: MPP로 최고 분석 성능, SQL 표준 준수
단점: 실시간 처리 부적합, 운영비용		 TB~PB급 데이터, 주간/월간 리포트, 역사적 트렌드 분석
Amazon Athena S3 데이터에 대한 임시 쿼리 서버리스, 쿼리 당 비용. Redshift보다 느리고 복잡한 쿼리 부적합 스키마 온 리드, 탐색적 분석, 비정형 쿼리
Snowflake 데이터 웨어하우스 스토리지-컴퓨트 분리 아키텍처, 멀티 클라우드. Redshift와 기능 경쟁 클라우드 중립성 필요, 동적 스케일링 중요 시
PostgreSQL / Aurora OLTP, 트랜잭션 처리 낮은 레이턴시, ACID 보장. 분석 쿼리 성능은 Redshift에 비해 현저히 낮음 주문 처리, 사용자 인증 등 실시간 CRUD 작업
BigQuery 서버리스 데이터 웨어하우스 완전 서버리스, 관리 부담 없음. GCP 생태계. Redshift와 유사한 분석 성능 AWS 생태계가 아닐 때, 관리 최소화 필요 시

3. 핵심 개념 (아키텍처)

Redshift 클러스터의 핵심 구성 요소와 데이터 흐름을 이해합니다.

┌─────────────────────────────────────────────────────────────┐ │ Client / BI Tools │ │ (Tableau, Looker, etc.) │ └──────────────────────────┬──────────────────────────────────┘ │ (JDBC/ODBC) ┌──────────────────────────▼──────────────────────────────────┐ │ Leader Node │ │ ┌────────────────────────────────────────────────────┐ │ │ │ Query Planner & Coordinator │ │ │ │ - Parses SQL │ │ │ │ - Creates Distributed Execution Plan │ │ │ │ - Aggregates Results from Compute Nodes │ │ │ └────────────────────────────────────────────────────┘ │ └──────────────────────────┬──────────────────────────────────┘ │ (Distributed Plan) ┌──────────────────────────▼──────────────────────────────────┐ │ Compute Node(s) │ │ ┌────────────┐ ┌────────────┐ ┌────────────┐ │ │ │ Slice 1 │ │ Slice 2 │ │ Slice N │ │ │ │ ┌────────┐ │ │ ┌────────┐ │ │ ┌────────┐ │ │ │ │ │ Local │ │ │ │ Local │ │ │ │ Local │ │ │ │ │ │ Storage│ │ │ │ Storage│ │ │ │ Storage│ │ │ │ │ └────────┘ │ │ └────────┘ │ │ └────────┘ │ │ │ └────────────┘ └────────────┘ └────────────┘ │ │ (Columnar Storage) (Columnar Storage) (Columnar Storage) │ └─────────────────────────────────────────────────────────────┘ │ ┌──────────────────────────▼──────────────────────────────────┐ │ Persistent Storage │ │ (S3 for RA3, Managed Storage) │ └─────────────────────────────────────────────────────────────┘
  • 리더 노드(Leader Node): 클라이언트 연결을 관리하고 쿼리 실행 계획을 수립하며 최종 결과를 집계합니다.
  • 컴퓨트 노드(Compute Node): 실제 데이터 저장과 쿼리 실행을 담당합니다. 노드 내 슬라이스(Slice)로 세분화되어 병렬 처리됩니다.
  • 슬라이스(Slice): 컴퓨트 노드의 논리적 파티션. 각 슬라이스는 CPU 코어와 메모리 할당을 받아 데이터의 일부를 처리합니다.
  • 컬럼형 저장(Columnar Storage): 데이터를 컬럼 단위로 압축 저장하여 분석 쿼리 성능을 향상시킵니다.

4. 퀵스타트: 설치 → 설정 → 첫 예제

Python 환경에서 Redshift에 연결하고 첫 번째 파이프라인을 구축하는 단계별 가이드입니다.

Step 1: 클러스터 생성 및 엔드포인트 확인 (AWS Console/CLI)

AWS CLI 명령어
# 클러스터 생성 (예시, 파라미터는 상황에 맞게 조정)
aws redshift create-cluster \
    --cluster-identifier my-redshift-cluster \
    --node-type dc2.large \
    --master-username adminuser \
    --master-user-password  \
    --cluster-type single-node \
    --publicly-accessible true

# 클러스터 상태 및 엔드포인트 확인
aws redshift describe-clusters --cluster-identifier my-redshift-cluster \
    --query 'Clusters[0].[ClusterStatus, Endpoint.Address, Endpoint.Port]'

Step 2: Python 드라이버 설치 및 연결 설정

Python (psycopg2 설치 및 연결)
pip install psycopg2-binary boto3
import psycopg2
import boto3
from botocore.exceptions import ClientError

def get_redshift_connection():
    """Redshift 클러스터에 연결"""
    # 엔드포인트, 포트, 사용자 정보는 AWS Console에서 확인
    host = "my-redshift-cluster.xxxxxx.us-east-1.redshift.amazonaws.com"
    port = 5439
    dbname = "dev"
    user = "adminuser"
    password = "" # 실무에서는 Secrets Manager 사용 권장

    try:
        conn = psycopg2.connect(
            host=host,
            port=port,
            dbname=dbname,
            user=user,
            password=password
        )
        print("Redshift 연결 성공")
        return conn
    except Exception as e:
        print(f"연결 실패: {e}")
        return None

Step 3: 첫 번째 테이블 생성 및 데이터 로드

Python: 테이블 생성 및 COPY 명령 실행
conn = get_redshift_connection()
cur = conn.cursor()

# 샘플 테이블 생성
create_table_sql = """
CREATE TABLE IF NOT EXISTS sales_transactions (
    transaction_id INTEGER PRIMARY KEY,
    customer_id INTEGER,
    product_id VARCHAR(50),
    sale_amount DECIMAL(10,2),
    sale_date DATE,
    region VARCHAR(20)
)
DISTKEY(customer_id)  -- 고객 ID 기준 분산
SORTKEY(sale_date);   -- 판매일 기준 정렬
"""
cur.execute(create_table_sql)
conn.commit()
print("테이블 생성 완료")

# S3에 있는 CSV 데이터를 COPY (가장 일반적인 로드 방식)
copy_sql = """
COPY sales_transactions
FROM 's3://your-data-bucket/path/to/sales_data.csv'
IAM_ROLE 'arn:aws:iam::123456789012:role/MyRedshiftReadS3Role'
CSV
IGNOREHEADER 1
DELIMITER ','
DATEFORMAT 'auto';
"""
# cur.execute(copy_sql) # S3 버킷과 IAM Role이 준비된 후 실행
# conn.commit()

# 직접 INSERT 예시 (소량 데이터)
insert_sql = """
INSERT INTO sales_transactions VALUES
(1, 1001, 'PROD_A', 299.99, '2026-03-01', 'Seoul'),
(2, 1002, 'PROD_B', 150.50, '2026-03-01', 'Busan');
"""
cur.execute(insert_sql)
conn.commit()
print("샘플 데이터 INSERT 완료")

cur.close()
conn.close()

Step 4: 첫 분석 쿼리 실행

Python: 분석 쿼리 실행 및 결과 확인
conn = get_redshift_connection()
cur = conn.cursor()

analysis_query = """
SELECT
    region,
    DATE_TRUNC('month', sale_date) as sale_month,
    COUNT(*) as transaction_count,
    SUM(sale_amount) as total_sales
FROM sales_transactions
GROUP BY region, DATE_TRUNC('month', sale_date)
ORDER BY sale_month DESC, total_sales DESC;
"""

cur.execute(analysis_query)
results = cur.fetchall()

print("지역별 월간 매출 분석")
for row in results:
    print(f"지역: {row[0]}, 월: {row[1]}, 거래수: {row[2]}, 매출: {row[3]}")

cur.close()
conn.close()

5. 핵심 설정 레퍼런스

성능과 비용을 최적화하기 위해 반드시 이해해야 할 주요 설정들입니다.

설정 항목 설명 권장 값/가이드 영향
DISTKEY 데이터를 컴퓨트 노드에 분산시키는 기준 컬럼 조인 빈도가 높은 컬럼, 카디널리티 높은 컬럼 선택 (e.g., user_id) 노드 간 데이터 이동(Data Skew) 최소화, 조인 성능 향상
SORTKEY 데이터를 물리적으로 정렬하는 기준 컬럼 범위 쿼리/필터에 자주 사용되는 컬럼 (e.g., transaction_date) 쿼리 스캔 범위 축소, 압축 효율 향상
노드 타입 컴퓨트 노드의 하드웨어 사양 dc2: 컴퓨트 집중, ra3: 스토리지-컴퓨트 분리 (유연한 스케일링) 성능, 스토리지 용량, 비용 구조 결정
WLM(Workload Management) 쿼리 대기열 및 동시성 관리 Auto WLM 사용 권장. 우선순위 큐를 통해 ETL/보고서 쿼리 분리 자원 경합 방지, 쿼리 타임아웃 관리
압축 인코딩 컬럼 데이터 압축 방식 CREATE TABLE 시 `ENCODE AUTO` 사용 또는 분석 후 수동 지정 (e.g., DELTA) 스토리지 절감, I/O 성능 향상
VACUUM & ANALYZE 저장 공간 재확보 및 통계 정보 업데이트 대량 DELETE/UPDATE 후 수동 VACUUM 실행. 주기적 ANALYZE 자동화 쿼리 플랜 정확도 유지, 성능 저하 방지

6. 실무 팁 & 트러블슈팅

자주 겪는 문제 + 해결책

문제 1: 쿼리가 매우 느리거나 타임아웃 발생
  • 원인: 잘못된 DISTKEY/SORTKEY, 통계 정보 부실, 데이터 편향(Skew), WLM 설정.
  • 해결:
    1. EXPLAIN 명령어로 실행 계획 확인. "Hash Join" 대신 "Nested Loop"가 보이면 의심.
    2. STL_QUERY, SVL_QUERY_REPORT 시스템 뷰로 실행 단계 분석.
    3. 통계 업데이트: ANALYZE [table_name];
    4. 데이터 편향 확인: SELECT slice, COUNT(*) FROM stv_blocklist WHERE tbl = ... GROUP BY slice;
문제 2: COPY 명령이 S3 데이터를 로드하지 못함
  • 원인: IAM Role 권한 부족, 파일 형식 불일치, 네트워크 문제.
  • 해결:
    1. IAM Role에 S3 버킷에 대한 s3:GetObject, s3:ListBucket 권한 확인.
    2. 파일 헤더, 구분자, 인코딩 옵션(CSV, DELIMITER, IGNOREHEADER) 재확인.
    3. STL_LOAD_ERRORS 시스템 테이블에서 상세 에러 메시지 조회.
문제 3: 디스크 공간 부족
  • 원인: 로그 데이터 누적, VACUUM 미실행, 불필요한 데이터 보관.
  • 해결:
    1. 대량 삭제 후 VACUUM [table_name]; 실행하여 공간 회수.
    2. 오래된 데이터는 아카이빙(S3) 후 테이블에서 삭제.
    3. RA3 노드 타입으로 마이그레이션 고려 (스토리지 자동 확장).

성능 최적화 팁

  • 최소한의 컬럼 조회: SELECT * 금지. 컬럼형 저장의 이점을 살리기 위해 필요한 컬럼만 명시.
  • 조인 순서: 큰 테이블을 DISTKEY로 분산시키고, 작은 테이블은 브로드캐스트 조인 유도.
  • 임시 테이블 활용: 복잡한 쿼리 중간 결과를 임시 테이블에 저장하여 재사용.
  • UNLOAD 활용: 대량 결과 추출은 UNLOAD to S3 후 다운로드가 더 빠름.
