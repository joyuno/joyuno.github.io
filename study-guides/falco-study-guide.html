1. Falco란?

Falco는 클라우드 네이티브 런타임 보안을 위한 오픈소스 위협 탐지 엔진입니다. Linux 시스템 호출(syscall)을 실시간으로 모니터링하여 이상 행위를 탐지하고 알림을 생성합니다. CNCF(Cloud Native Computing Foundation) 졸업 프로젝트로, Kubernetes 환경에서의 보안 표준 도구로 자리잡았습니다.

실시간 Syscall 모니터링

커널 수준에서 시스템 호출을 직접 수집(eBPF/드라이버)하여 모든 프로세스, 파일, 네트워크 활동을 가시화합니다. 기존 호스트 기반 침입 탐지 시스템(HIDS)보다 더 낮은 수준에서 동작합니다.

규칙 기반 위협 탐지

YAML 형식의 선언적 규칙(Falco Rules)을 통해 정상/비정상 행위를 정의. 컨테이너 내 권한 상승, 민감 파일 접근, 이상 네트워크 연결 등을 탐지할 수 있습니다.

클라우드 네이티브 통합

Kubernetes Audit Logs, 클라우드 제공자 로그(CloudTrail, Azure Activity Log 등)를 추가 컨텍스트로 활용하여 다계층 보안 분석이 가능합니다.

핵심 가치: Falco는 "침입 탐지"가 아닌 "위협 탐지"에 초점을 맞춥니다. 알려진 악성 패턴뿐만 아니라 조직의 정책 위반, 구성 오류, 비정상적 활동을 실시간으로 식별합니다.

2. 왜 써야 하는가? (기존 대안과 비교)

Falco는 기존 보안 모니터링 도구와 비교했을 때 다음과 같은 차별점을 가집니다.

도구/접근법 주요 특징 Falco와의 비교
전통적 HIDS (OSSEC, Tripwire) 파일 무결성 검사, 로그 분석, 정기적 스캔 Falco는 실시간 syscall 모니터링으로 지연 시간이 짧고, 컨테이너/쿠버네티스 인식력이 높음.
Network IDS (Suricata, Snort) 네트워크 패킷 분석, 프로토콜 이상 탐지 Falco는 호스트/컨테이너 내부 활동에 초점. 네트워크 IDS와 상호 보완적으로 사용 권장.
CSPM (Cloud Security Posture Management) 구성 오류, 정책 준수 검사 (예: Prisma Cloud, Wiz) CSPM은 정적 구성을, Falco는 런타임 동작을 모니터링. 함께 사용 시 완전한 보안 상태 관리 가능.
eBPF 기반 모니터링 (Tetragon) eBPF를 이용한 실시간 시각화 및 정책 실행 Falco도 eBPF를 지원하며, 더 풍부한 규칙 언어와 커뮤니티 규칙셋을 강점으로 함.
주의: Falco는 방화벽이나 EDR(Endpoint Detection and Response)의 완전한 대체제가 아닙니다. 심층 방어(Defense in Depth) 전략의 한 구성 요소로 통합해야 합니다.

3. 핵심 개념과 아키텍처

아키텍처 개요

Falco는 이벤트 소스, 규칙 엔진, 출력 처리기의 3계층으로 구성됩니다.

+-----------------------------------------------------------------------+ | 데이터 소스 (Event Sources) | +-------------------+-------------------+-------------------------------+ | Syscall (eBPF) | Kubernetes Audit | Cloud Provider Logs | | 드라이버 | Events | (AWS, GCP, Azure) | +-------------------+-------------------+-------------------------------+ | v +-----------------------------------------------------------------------+ | 규칙 엔진 (Falco Engine) | | +---------------------+ +---------------------+ | | | 조건 평가 | -> | 출력 포맷팅 | | | | (Rule Conditions) | | (Output Formatting) | | | +---------------------+ +---------------------+ | +-----------------------------------------------------------------------+ | v +-----------------------------------------------------------------------+ | 출력 처리기 (Outputs) | +-----------+-----------+-----------+-----------+----------------------+ | Stdout | File | Syslog | HTTP | 프로그램 | | | | | Endpoint | (Custom Program) | +-----------+-----------+-----------+-----------+----------------------+

핵심 구성 요소

  • Falco 드라이버: syscall을 캡처하기 위한 커널 모듈 또는 eBPF 프로그램. 최신 커널에서는 eBPF를 권장.
  • 규칙 파일(.yaml): 탐지 로직 정의. 조건(condition), 출력(output), 우선순위(priority)로 구성.
  • 출력: 탐지 이벤트를 전달할 채널. Slack, PagerDuty, 웹훅 등으로 연동 가능.

4. 퀵스타트: 설치부터 첫 탐지까지

단계 1: Linux에 Falco 설치 (Ubuntu/Debian 예시)

터미널
# 공식 저장소 키 추가 및 설정
curl -s https://falco.org/repo/falcosecurity-3672BA8F.asc | apt-key add -
echo "deb https://download.falco.org/packages/deb stable main" | tee -a /etc/apt/sources.list.d/falcosecurity.list

# Falco 설치
apt-get update
apt-get install -y falco

# eBPF 드라이버 사용 (권장, 커널 4.14+)
apt-get install -y linux-headers-$(uname -r)
falco-driver-loader bpf

단계 2: 기본 설정 확인 및 서비스 시작

# 주요 설정 파일 위치 확인
ls -la /etc/falco/  # falco.yaml, rules files

# Falco 서비스 시작
systemctl start falco
systemctl enable falco

# 실시간 로그 확인 (다른 터미널에서)
journalctl -u falco -f

단계 3: 첫 번째 테스트 규칙 트리거

기본 규칙셋에 포함된 "Shell in a container" 규칙을 테스트해봅니다.

터미널 - 테스트 명령어 실행
# 컨테이너 내에서 쉘 실행 (Falco가 탐지해야 함)
docker run --rm -it alpine /bin/sh

위 명령어 실행 후, Falco 로그에서 다음과 같은 경고를 확인할 수 있습니다.

# journalctl 출력 예시
{"output": "17:20:01.123456789: Warning Shell spawned in a container ...",
 "priority": "Warning",
 "rule": "Terminal shell in container",
 "time": "2026-03-15T17:20:01.123456789Z",
 "container": "alpine"}

단계 4: 간단한 커스텀 규칙 작성

/etc/falco/custom_rules.yaml 파일을 생성합니다.

/etc/falco/custom_rules.yaml
- rule: Sensitive File Access in Production
  desc: Production 네임스페이스에서 /etc/shadow 접근 감지
  condition: >
    container.image.repository != "falco" and
    k8s.ns.name = "production" and
    evt.type = open and
    fd.name = "/etc/shadow"
  output: >
    Production에서 민감 파일 접근 (user=%user.name container=%container.name
    image=%container.image.repository file=%fd.name)
  priority: CRITICAL
  tags: [filesystem, production, k8s]
  append: true
# 변경사항 적용
systemctl restart falco

5. 핵심 설정 레퍼런스

설정 항목 (falco.yaml) 기본값/예시 설명
rules_file ["/etc/falco/falco_rules.yaml", "/etc/falco/custom_rules.yaml"] 로드할 규칙 파일 목록. 순서 중요.
outputs.console.enabled true 표준 출력에 이벤트 출력 여부.
outputs.http.url "https://webhook.example.com" HTTP 웹훅 엔드포인트. JSON 형식으로 이벤트 전송.
json_output true 출력을 JSON 형식으로 포맷팅. 머신 파싱에 유용.
log_level info 로그 상세도 (debug, info, warn, error, critical).
webserver.enabled false 내장 웹 서버 활성화 (메트릭, 건강 상태).
webserver.listen_port 8765 웹 서버 포트.
syscall_event_drops action: log, rate: 0.01, max_burst: 1 이벤트 드롭 발생 시 동작 설정.

규칙 조건 필드 예시

필드 카테고리 예시 필드 설명
이벤트 기본 evt.type, evt.dir, evt.arg 이벤트 타입 (open, execve 등), 방향, 인자.
프로세스 proc.name, proc.pid, proc.cmdline 프로세스 이름, PID, 전체 명령어.
파일 디스크립터 fd.name, fd.type, fd.num 파일 경로, 타입 (file, directory, IPv4 등), 번호.
컨테이너 container.id, container.image, container.name 런타임(도커, containerd)에서 제공하는 정보.
Kubernetes k8s.pod.name, k8s.ns.name, k8s.deployment.name K8s Audit Logs 또는 Falco K8s Metadata Fetcher에서 수집.

6. 실무 팁 & 트러블슈팅

자주 겪는 문제

문제: "Unable to load the driver"

원인: 커널 헤더가 없거나, eBPF/드라이버 컴파일 실패.

해결:

# 커널 헤더 설치
apt-get install -y linux-headers-$(uname -r)
# 드라이버 로더 재시도 (자동으로 최적 방법 선택)
falco-driver-loader auto

문제: 너무 많은 경고 (노이즈)

원인: 기본 규칙이 특정 환경에 과도하게 반응.

해결: 규칙 비활성화 또는 조건 수정.

# /etc/falco/falco_rules.yaml 또는 custom_rules.yaml에서
# 특정 규칙에 `enabled: false` 추가 또는
# condition에 예외 추가 (예: and not proc.name in (nginx, app_process))

문제: Kubernetes 메타데이터 누락

원인: Falco K8s Metadata Fetcher가 실행되지 않거나 연결 실패.

해결: K8s 배포 시 서비스 계정 권한 및 네트워크 폴리시 확인.

# Helm으로 설치 시 메타데이터 페처 활성화
helm install falco falcosecurity/falco \
  --set falco.grpc.enabled=true \
  --set falco.grpcOutput.enabled=true \
  --set falco.k8sAuditEndpoint=http://kube-apiserver:8080

성능 최적화 팁

  • 드라이버 선택: 최신 커널(4.14+)에서는 eBPF가 성능 및 안정성 면에서 우수.
  • 규칙 최적화: 복잡한 조건을 가진 규칙은 가능한 간소화. 불필요한 매크로 사용 피하기.
  • 샘플링 고려: 매우 빈번한 이벤트(예: 특정 네트워크 연결)에 대해 sample 키워드로 샘플링 적용.
  • 출력 큐 관리: 대량 이벤트 발생 시 outputs.queue 설정으로 버퍼링.
디버깅 명령어: 
# Falco 디버그 모드 실행 (일회성)
falco --disable-cri-async -u

# 규칙 테스트 (실제 이벤트 없이)
falco --list -r /etc/falco/custom_rules.yaml

# eBPF 프로그램 상태 확인
bpftool prog list | grep falco
