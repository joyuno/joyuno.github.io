1. OWASP ZAP란?

OWASP ZAP(Zed Attack Proxy)는 OWASP 재단이 주도하는 세계적으로 가장 인기 있는 오픈 소스 웹 애플리케이션 보안 테스트 도구입니다. 매뉴얼 테스트부터 자동화된 스캔까지 지원하는 통합 DAST(Dynamic Application Security Testing) 도구로, 프록시, 스파이더, 스캐너, 퍼저 등 다양한 모듈을 제공합니다.

1. 프록시 기반 인터셉션

브라우저와 서버 사이의 모든 HTTP/HTTPS 트래픽을 가로채고 수정할 수 있습니다. 수동 테스트와 자동화 테스트의 핵심 인프라 역할을 합니다.

2. 자동화된 액티브/패시브 스캐너

패시브 스캔: 트래픽을 모니터링하며 자동으로 취약점을 탐지.
액티브 스캔: 의도적으로 악성 페이로드를 전송하여 취약점을 공격적으로 탐지.

3. 확장 가능한 API & 자동화

REST API, CLI, Python/ZAP API를 제공하여 CI/CD 파이프라인에 쉽게 통합할 수 있습니다. AI/ML 모델과의 연동에도 유리한 구조입니다.

2. 왜 써야 하는가? (기존 대안과 비교)

상용 도구 대비 ZAP의 선택은 비용, 유연성, 통합성 측면에서 명확한 장점을 제공합니다. 특히 AI/자동화 워크플로우에 적합합니다.

도구 장점 단점 ZAP 대비 포지션
OWASP ZAP 무료 오픈소스, 강력한 API/자동화, 커뮤니티 활성, 플러그인 확장성 상용 도구 대비 일부 고급 보고 기능 부족, 초기 학습 곡선 본 가이드의 핵심. 자동화/통합 중심 프로젝트에 최적.
Burp Suite Pro 산업 표준, 정교한 수동 테스트 도구, 강력한 인트루더/퍼저 고가의 라이선스, 자동화 API 제한적 심층 수동 테스트 필요 시 보완 도구. ZAP은 자동화/스케일링 우위.
Nessus, Qualys 인프라/네트워크 취약점 스캔에 특화, 폭넓은 CVE 커버리지 웹 앱 특화 취약점 탐지 능력 제한적, 가격 부담 ZAP은 웹 앱 레이어에 특화. 상호 보완적 사용 가능.
상용 SAST 도구 소스 코드 레벨 취약점 조기 발견, 정적 분석 정확도 높은 False Positive, 런타임 환경 반영 불가 ZAP(DAST)는 런타임 동작을 테스트. DevSecOps에서 SAST/DAST 병행 권장.
AI 엔지니어 관점: ZAP의 REST API와 헤드리스 모드는 AI 에이전트가 보안 테스트 워크플로우를 자동으로 운영하고, 결과를 분석하며, 재구성하는 데 이상적인 인터페이스입니다.

3. 핵심 개념과 아키텍처

ZAP는 멀티스레드 프록시 엔진을 중심으로 다양한 컴포넌트가 협력하는 모듈식 아키텍처를 가집니다.

┌─────────────────────────────────────────────────────────────┐ │ 사용자 인터페이스 (UI) │ │ (Desktop GUI / Web UI / API Clients) │ └──────────────────────────┬──────────────────────────────────┘ │ (REST API / Internal Calls) ┌──────────────────────────▼──────────────────────────────────┐ │ ZAP 코어 엔진 (Core Engine) │ │ ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐ │ │ │ 프록시 │ │ 스파이더 │ │ 스캐너 │ │ 퍼저 │ │ │ │ (Proxy) │ │ (Spider) │ │(Scanner) │ │ (Fuzzer) │ │ │ └──────────┘ └──────────┘ └──────────┘ └──────────┘ │ │ │ │ │ │ │ │ └─────────────┼─────────────┼─────────────┘ │ │ │ (공유 세션 & 컨텍스트 데이터) │ └─────────────────────────▼───────────────────────────────────┘ │ ┌─────────────▼─────────────┐ │ 확장 기능 (Add-ons) │ │ (Scripting, Plugins, ...) │ └─────────────┬─────────────┘ │ ┌─────────────▼─────────────┐ │ 타겟 웹 애플리케이션 │ │ (Target Web App) │ └────────────────────────────┘

주요 컴포넌트 설명

  • 프록시: 모든 요청/응답의 중간자. 인증 처리, 트래픽 수정, 재전송의 기초.
  • 스파이더: 웹 앱의 링크와 양식을 크롤링하여 공격 표면을 발견.
  • 액티브/패시브 스캐너: 실제 취약점을 탐지하는 핵심 엔진.
  • 퍼저: 파라미터에 다양한 페이로드를 주입하여 예상치 못한 취약점 발견.
  • 컨텍스트: 테스트 범위, 인증 정보, 기술 스택을 정의하는 논리적 컨테이너.
  • 확장 기능 Python, JavaScript 스크립팅 및 커뮤니티 플러그인으로 기능 무한 확장.

4. 퀵스타트: 설치부터 첫 스캔까지

4.1 설치 (Linux/macOS 기준)

# 공식 다운로드 페이지에서 최신 버전 확인 (예: 2.14.0)
wget https://github.com/zaproxy/zaproxy/releases/download/v2.14.0/ZAP_2.14.0_Linux.tar.gz
tar -xzf ZAP_2.14.0_Linux.tar.gz
cd ZAP_2.14.0/
./zap.sh -daemon -port 8080 -host 0.0.0.0 -config api.disablekey=true &

혹은 Docker를 이용한 설치 (권장 for CI/CD):

docker run -u zap -p 8080:8080 -p 8090:8090 -i \\
  owasp/zap2docker-stable zap.sh -daemon \\
  -host 0.0.0.0 -port 8080 -config api.disablekey=true

4.2 Python API를 이용한 첫 자동화 스캔

#!/usr/bin/env python3
"""
ZAP Python API를 이용한 기본적인 액티브 스캔 예제
"""
from zapv2 import ZAPv2
import time

# ZAP 프록시 설정 (로컬 데몬 실행 가정)
api_key = ''  # -config api.disablekey=true 시 빈 값
target_url = 'http://testphp.vulnweb.com'  # 테스트용 취약한 앱
zap_proxy = 'http://127.0.0.1:8080'

# ZAP 클라이언트 초기화
zap = ZAPv2(apikey=api_key, proxies={'http': zap_proxy, 'https': zap_proxy})

print(f"[1] 타겟 접근: {target_url}")
zap.urlopen(target_url)  # ZAP가 타겟에 접근하도록 유도

print("[2] 스파이더 실행...")
scan_id = zap.spider.scan(target_url)
while int(zap.spider.status(scan_id)) < 100:
    time.sleep(2)
print("스파이더 완료.")

print("[3] 액티브 스캔 실행... (주의: 실제 운영 서버에 실행 금지)")
ascan_id = zap.ascan.scan(target_url)
while int(zap.ascan.status(ascan_id)) < 100:
    time.sleep(5)
print("액티브 스캔 완료.")

print("[4] 리포트 생성")
with open('zap_report.html', 'w') as f:
    f.write(zap.core.htmlreport())

print("[5] 발견된 알람 요약")
alerts = zap.core.alerts(baseurl=target_url)
for alert in alerts:
    print(f"  - [{alert['risk']}] {alert['alert']} on {alert.get('url', 'N/A')}")

4.3 CLI를 이용한 빠른 스캔

# ZAP가 설치된 디렉토리에서 실행
./zap.sh -cmd -quickurl http://testphp.vulnweb.com -quickout report.html
주의: 테스트는 반드시 권한이 있는 서버 또는 testphp.vulnweb.com 같은 합법적인 테스트 사이트에서만 수행하세요. 무단 스캔은 불법입니다.

5. 핵심 설정 레퍼런스

설정 영역 키/파라미터 기본값/예시 설명
API & 네트워크 -config api.disablekey=true false API 키 인증 비활성화 (로컬/신뢰 환경용). 보안상 프로덕션에선 사용 비권장.
-host 0.0.0.0 localhost 어떤 인터페이스에서 수신할지 지정. 컨테이너 환경에선 0.0.0.0 필요.
-port 8080 8080 ZAP 프록시/API 포트.
스캔 정책 scanner.threadPerHost 2 호스트당 스캔 스레드 수. 높을수록 빠르지만 서버 부하 증가.
pscans.maxAlertsPerRule 20 패시브 스캔 규칙당 최대 알람 수. 리포트 폭주 방지.
scanner.delayInMs 0 요청 간 지연 시간(ms). 서버 보호를 위해 설정 가능.
컨텍스트/인증 context.include regex 패턴 스캔 범위에 포함할 URL 정규식.
authentication.method formBased 인증 방식 (formBased, httpAuth, jsonBased, script).
authentication.loginUrl https://.../login 로그인 폼 URL. 인증 테스트에 필수.
보고 report.html.title ZAP Security Report 생성된 HTML 리포트의 제목.
report.generate HTML, JSON, XML 출력할 리포트 형식.

6. 실무 팁 & 트러블슈팅

자주 겪는 문제와 해결책

문제 1: 스파이더가 링크를 찾지 못함

원인: JavaScript 기반 SPA(Single Page Application) 또는 폼에 CSRF 토큰 필요.

해결: AJAX 스파이더 애드온 설치 및 사용. 또는 브라우저를 통해 수동 탐색하여 ZAP가 트래픽을 보게 한 후 '기존 세션에서 탐색' 기능 사용.

문제 2: 인증된 영역 스캔 실패

원인: 로그인 세션이 유지되지 않거나, 인증 설정 오류.

해결: 컨텍스트에서 인증 방법을 정확히 설정하고, '사용자'를 생성하여 해당 사용자로 스캔 실행. 스크립트 기반 인증 사용을 고려.

문제 3: 너무 많은 False Positive

원인: 스캔 정책이 과도하게 공격적이거나, 앱의 정상 동작을 오탐.

해결: 스캔 정책 조정 (예: 'False Positive'로 표시 가능한 규칙 비활성화). 컨텍스트에 기술 스택 추가하여 불필요한 검사 제외.

문제 4: CI/CD 파이프라인에서 스캔 타임아웃

원인: 큰 애플리케이션에 대한 전체 스캔 시간이 파이프라인 제한 초과.

해결: '빠른 스캔' 모드 사용. 또는 스캔을 '증분 스캔'으로 분할. scanner.threadPerHost 증가 (서버 감당 가능 시).

AI/자동화 관점의 고급 팁

  • 스캔 결과 구조화: JSON 리포트(zap.core.jsonreport())를 파싱하여 AI 모델 학습 데이터셋 또는 자동 분류 시스템의 입력으로 활용.
  • 스크립트 확장: Python 스크립팅을 사용하여 커스텀 페이로드 생성, 복잡한 인증 흐름 처리, 결과 필터링 로직 구현 가능.
  • 헤드리스 운영: Docker + ZAP API 조합으로 완전 자동화된 스캔 클러스터 구성. Kubernetes Job으로 주기적 스캔 실행.
  • 통합: Slack, Jira, DefectDojo 등과의 웹훅 통합을 통해 발견된 취약점을 자동으로 티켓 생성 및 할당.
