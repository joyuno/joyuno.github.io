1. DataHub란?

DataHub는 LinkedIn에서 개발한 오픈소스 메타데이터 플랫폼으로, 데이터 탐색, 협업, 거버넌스를 위한 통합 카탈로그입니다. 마이크로서비스 아키텍처로 설계되어 확장성과 유연성이 뛰어납니다.

🔍 데이터 발견

데이터셋, 피플, 파이프라인, 대시보드 등을 검색하고 라인리지를 추적합니다. 강력한 검색과 필터링으로 필요한 데이터를 빠르게 찾을 수 있습니다.

🤝 협업

데이터에 대한 설명(Description), 태그(Tag), 용어집(Glossary Term)을 추가하고, 소유자(Owner)를 지정하여 팀 간 협업을 촉진합니다.

🛡️ 거버넌스

데이터 품질, 개인정보 식별(PII) 태그, 수명주기 관리, 접근 제어 정책을 적용하여 데이터 거버넌스를 체계화합니다.

핵심 가치: DataHub는 단순한 카탈로그가 아닌, 데이터 스택의 모든 메타데이터를 실시간으로 연결하고 동기화하는 "메타데이터 그래프"를 제공합니다.

2. 왜 써야 하는가?

기존 수동 문서화나 지엽적인 도구의 한계를 넘어, 확장 가능한 중앙화된 메타데이터 플랫폼이 필요한 이유입니다.

대안 / 접근법 한계점 DataHub의 해결책
위키/Confluence 수동 문서화 데이터 변경 시 동기화 어려움, 검색 기능 부족, 실시간 정보 아님 자동 메타데이터 수집, 실시간 동기화, 강력한 검색
각 팀별 독립 카탈로그 데이터 소산(Silo)화, 전사적 뷰 부재, 중복 관리 전사적 통합 카탈로그, 일관된 메타데이터 모델
상용 메타데이터 도구 높은 비용, 벤더 종속, 유연성 부족 오픈소스(아파치 2.0), 커뮤니티 기반, 확장 가능
메타데이터 관리 없음 데이터 신뢰도 하락, 분석가/과학자의 시간 낭비, 규제 대응 어려움 데이터 발견성과 신뢰도 향상, 거버넌스 체계 구축

3. 핵심 개념

DataHub의 아키텍처는 프론트엔드, 메타데이터 서비스, 스토리지, 수집 프레임워크로 구성됩니다.

+-----------------------------------------------------------------------+ | DataHub Frontend (React) | | (UI & User Interactions) | +-----------------------------+-----------------------------------------+ | HTTP/GraphQL +-----------------------------v-----------------------------------------+ | DataHub GMS (General Metadata Service) | | (Metadata Store, GraphQL API, Authentication) | +-----------------------------+-----------------------------------------+ | Kafka +-----------------------------v-----------------------------------------+ | Metadata Storage | | (MySQL/PostgreSQL, Elasticsearch, Neo4j*) | +-----------------------------------------------------------------------+ ^ ^ | MAE (Metadata Audit Event) | MCE (Metadata Change Event) +-------|----------------------------------|----------------------------+ | | | | | +-----v------+ +--------v--------+ | | | Ingestion | Ingestion | | | | Sources | | Transformers | | | | (Snowflake,| | (Custom Logic) | | | | BigQuery, | +-----------------+ | | | Kafka...) | | | +------------+ | | DataHub Ingestion Framework | | (Python-based, Extensible) | +-----------------------------------------------------------------------+

주요 컴포넌트

  • GMS (General Metadata Service): 메타데이터 CRUD 및 GraphQL API 제공
  • MAE/MCE: 메타데이터 변경 이벤트를 Kafka 통해 전파
  • Ingestion Framework: 다양한 소스로부터 메타데이터 추출
  • Storage: 관계형 DB(메타데이터), ES(검색), Neo4j(관계)

4. 퀵스타트

Docker Compose를 이용한 로컬 개발 환경 구축부터 첫 메타데이터 수집까지.

Step 1: DataHub 설치 (Docker)

CLI

# 최신 DataHub Docker 이미지 가져오기
docker pull acryldata/datahub:latest

# 공식 docker-compose.yml 파일 다운로드
curl -O https://raw.githubusercontent.com/datahub-project/datahub/master/docker/quickstart/docker-compose.yml

# 서비스 시작 (약 2-3분 소요)
docker-compose -p datahub up -d

# 상태 확인
docker-compose -p datahub ps

Step 2: 기본 설정 및 접속

브라우저에서 http://localhost:9002 접속 (기본 계정: datahub, 비밀번호: datahub).

환경 변수 설정 (선택)

# .env 파일 생성
echo "DATAHUB_VERSION=latest" > .env
echo "DATAHUB_MAPPED_PORT=9002" >> .env

Step 3: 첫 메타데이터 수집 (Python 예제)

Python (datahub CLI 설치 및 예제 수집)

# datahub CLI 설치
pip install 'acryl-datahub[datahub-rest]'

# 기본 PostgreSQL 메타데이터 수집 (예제)
datahub ingest -c ./examples/recipes/postgres_to_datahub.dhub.yaml

# 커스텀 YAML 파일 예제 (my_source.yaml)
cat > my_source.yaml << EOF
source:
  type: "mysql"
  config:
    username: "user"
    password: "pass"
    host_port: "localhost:3306"
    database: "my_db"

sink:
  type: "datahub-rest"
  config:
    server: "http://localhost:8080"
EOF

# 수집 실행
datahub ingest -c my_source.yaml

Step 4: Python SDK로 메타데이터 추가

Python SDK 예제

from datahub.emitter.mce_builder import make_dataset_urn
from datahub.emitter.rest_emitter import DatahubRestEmitter
from datahub.metadata.schema_classes import (
    DatasetPropertiesClass,
    ChangeTypeClass,
    MetadataChangeProposalClass,
)

# DataHub 서버에 연결
emitter = DatahubRestEmitter("http://localhost:8080")

# 데이터셋 URN 생성
dataset_urn = make_dataset_urn(platform="postgres", name="public.users")

# 데이터셋 설명 추가
metadata_change_proposal = MetadataChangeProposalClass(
    entityType="dataset",
    changeType=ChangeTypeClass.UPSERT,
    entityUrn=dataset_urn,
    aspectName="datasetProperties",
    aspect=DatasetPropertiesClass(
        description="사용자 마스터 테이블. 개인정보(PII) 포함."
    )
)

# 메타데이터 전송
emitter.emit(metadata_change_proposal)
print(f"메타데이터 업데이트 완료: {dataset_urn}")

5. 핵심 설정 레퍼런스

구성 요소 설정 키/파일 기본값/예시 설명
Ingestion recipe.yaml source.type, sink.type 수집 소스와 싱크 정의 (file, rest, kafka)
GMS 서버 application.yml server.port: 8080 메타데이터 서버 포트 및 DB 연결 설정
인증 authentication.* type: "SESSION" JWT, OIDC, LDAP 등 인증 방식 설정
스토리지 storage.* neo4j.uri, es.host Neo4j, Elasticsearch, RDBMS 연결 정보
Kafka kafka.bootstrap localhost:9092 MAE/MCE 이벤트 전송용 Kafka 브로커
Frontend env.js/config.js REACT_APP_GMS_HOST UI에서 접속할 GMS 서버 주소

6. 실무 팁 & 트러블슈팅

주의: 프로덕션 환경에서는 반드시 인증/권한 설정을 완료하고, 백업 전략을 수립하세요.

자주 겪는 문제

문제 1: 수집 실패 "Connection refused"

증상: datahub ingest 실행 시 GMS 서버(8080) 연결 실패.

해결:

# GMS 서버 상태 확인
curl http://localhost:8080/health

# Docker 네트워크 확인 (동일 네트워크 사용)
docker network create datahub-network
docker-compose -p datahub down
docker-compose -p datahub --project-directory . up -d

# 또는 sink 설정을 datahub-kafka로 변경
sink:
  type: "datahub-kafka"
  config:
    connection:
      bootstrap: "localhost:9092"

문제 2: 검색이 안 됨

증상: UI에서 데이터셋 검색 시 결과 없음.

해결:

# Elasticsearch 인덱스 상태 확인
curl http://localhost:9200/_cat/indices?v

# 검색 인덱스 재생성 (CLI)
datahub docker nuke    # 주의: 모든 데이터 삭제
datahub docker quickstart

# 또는 수집 후 검색 인덱스 업데이트 지연 확인
# 대량 수집 시 --no-default-report 플래그 사용
datahub ingest -c recipe.yaml --no-default-report

문제 3: 메모리 부족 오류

증상: Docker 컨테이너가 자주 재시작되거나 OOM 발생.

해결:

# docker-compose.yml 메모리 제한 증가
services:
  datahub-gms:
    mem_limit: '2g'
    mem_reservation: '1g'
  elasticsearch:
    mem_limit: '4g'
    environment:
      - "ES_JAVA_OPTS=-Xms2g -Xmx2g"

# 또는 호스트 메모리 확인 후 필요 시 스왑 추가
sudo fallocate -l 4G /swapfile
sudo chmod 600 /swapfile
sudo mkswap /swapfile
sudo swapon /swapfile

실무 팁

  • 점진적 수집: 모든 소스를 한번에 수집하지 말고, 중요 데이터 소스부터 단계적으로 확장.
  • 태그 전략: 팀별, 프로젝트별, 민감도(PII) 태그 체계를 사전에 설계.
  • CI/CD 통합: 데이터 파이프라인 CI에서 datahub CLI로 메타데이터 자동 등록.
  • 모니터링: datahub-actions 프레임워크로 메타데이터 변경 알림(Slack, 이메일) 설정.
