1. Debezium란?

Debezium은 Change Data Capture(CDC)를 위한 오픈소스 분산 플랫폼입니다. 데이터베이스의 트랜잭션 로그(바이너리 로그, WAL, redo log 등)를 모니터링하여 데이터 변경 사항을 실시간으로 캡처하고, 이벤트 스트림(주로 Apache Kafka)으로 전송합니다.

🔍 로그 기반 CDC

트리거나 쿼리 기반이 아닌 데이터베이스 자체의 트랜잭션 로그를 읽어 변경 사항을 캡처합니다. 소스 DB에 부하를 거의 주지 않으며, 모든 변경 사항(INSERT, UPDATE, DELETE)을 정확히 포착합니다.

🚀 실시간 스트리밍

변경 이벤트를 마이크로초 단위로 Kafka 토픽에 전송합니다. 데이터 파이프라인, 캐시 갱신, 검색 인덱싱, 실시간 분석 등 지연 시간이 중요한 시나리오에 적합합니다.

🧩 분산 & 내결함성

Kafka Connect 위에서 동작하여 확장성과 내결함성을 제공합니다. 커넥터 장애 시 오프셋을 기반으로 복구할 수 있으며, 클러스터 환경에서 안정적으로 운영됩니다.

2. 왜 써야 하는가?

기존 데이터 동기화 방식과의 비교를 통해 Debezium의 장점을 확인해 보세요.

방식 설명 장점 단점 Debezium 비교
배치 쿼리 주기적으로 전체/증분 데이터를 SELECT 구현 간단, 모든 DB 지원 실시간성 낮음, 소스 DB 부하, 변경 감지 불확실 ✅ 실시간, 저부하, 정확한 변경 추적
애플리케이션 이중 작성 애플리케이션 코드에서 두 시스템에 동시 저장 직접 제어 가능 코드 복잡도 증가, 데이터 불일치 위험, 유지보수 어려움 ✅ 비침투적, 관심사 분리
트리거 기반 CDC DB 트리거로 변경 히스토리 테이블 생성 실시간성 보장 소스 DB 성능 저하, 트리거 관리 복잡, DDL 변경 대응 어려움 ✅ 로그 기반으로 DB 부하 최소화
ETL 도구 전용 ETL 툴을 사용한 데이터 동기화 GUI, 다양한 변환 기능 비용, 실시간성 부족, 벤더 종속 가능성 ✅ 오픈소스, 커뮤니티 활성, 스트리밍 최적화
핵심 결론: Debezium은 실시간성, 정확성, 확장성, 비침투성을 모두 만족시키는 현대적인 데이터 동기화 패러다임을 제공합니다. 마이크로서비스 아키텍처, 데이터 레이크/웨어하우스 적재, 실시간 분석 등에 필수적인 컴포넌트입니다.

3. 핵심 개념

아키텍처 개요

+----------------+ +---------------------+ +-------------------+ +-------------------+ | | | | | | | | | Source DB |---->| Debezium Connector |---->| Apache Kafka |---->| Sink Applications| | (MySQL, PG, | | (Kafka Connect | | (Event Stream) | | (Elasticsearch, | | MongoDB, etc) | | Worker 내 실행) | | | | Data Lake, etc) | | | | | | | | | +----------------+ +---------------------+ +-------------------+ +-------------------+ ^ ^ ^ ^ | | | | Transaction Log Captures Changes Publishes Events Consume & Process (Binlog, WAL, etc) (Create, Update, as Kafka Messages Change Events Delete)

주요 컴포넌트

  • Debezium Connector: 특정 데이터베이스(MySQL, PostgreSQL, MongoDB 등)용으로 작성된 소스 커넥터. DB 로그를 읽고 변경 이벤트 레코드로 변환.
  • Kafka Connect: Debezium이 실행되는 프레임워크. 커넥터 관리, 확장, 내결함성, 오프셋 추적 제공.
  • Change Event: 데이터베이스 변경 사항을 표현하는 구조화된 메시지. `before`/`after` 상태, 작업 유형, 타임스탬프, 트랜잭션 메타데이터 포함.
  • Snapshot: 커넥터 최초 실행 시 기존 데이터 전체를 캡처하는 과정. 증분 스트리밍의 기반이 됨.

4. 퀵스타트 (Docker 기반)

Docker Compose를 사용해 MySQL → Debezium → Kafka → Python Consumer 파이프라인을 빠르게 구축합니다.

Step 1: Docker Compose 파일 작성

docker-compose.yml
version: '3.8'
services:
  zookeeper:
    image: confluentinc/cp-zookeeper:latest
    environment:
      ZOOKEEPER_CLIENT_PORT: 2181
      ZOOKEEPER_TICK_TIME: 2000

  kafka:
    image: confluentinc/cp-kafka:latest
    depends_on:
      - zookeeper
    environment:
      KAFKA_BROKER_ID: 1
      KAFKA_ZOOKEEPER_CONNECT: zookeeper:2181
      KAFKA_ADVERTISED_LISTENERS: PLAINTEXT://kafka:9092
      KAFKA_OFFSETS_TOPIC_REPLICATION_FACTOR: 1

  kafka-connect:
    image: debezium/connect:latest
    depends_on:
      - kafka
    environment:
      BOOTSTRAP_SERVERS: kafka:9092
      GROUP_ID: 1
      CONFIG_STORAGE_TOPIC: connect_configs
      OFFSET_STORAGE_TOPIC: connect_offsets
      STATUS_STORAGE_TOPIC: connect_statuses
    ports:
      - "8083:8083"

  mysql:
    image: debezium/example-mysql:latest
    environment:
      MYSQL_ROOT_PASSWORD: debezium
      MYSQL_USER: mysqluser
      MYSQL_PASSWORD: mysqlpw
    ports:
      - "3306:3306"

Step 2: 인프라 실행 및 커넥터 등록

터미널 명령어
# 1. Docker Compose로 서비스 실행
docker-compose up -d

# 2. Kafka Connect가 준비될 때까지 대기 (약 30초)
sleep 30

# 3. MySQL Debezium 커넥터 등록 (HTTP API 호출)
curl -i -X POST -H "Accept:application/json" -H "Content-Type:application/json" \
  http://localhost:8083/connectors/ -d '{
    "name": "inventory-connector",
    "config": {
      "connector.class": "io.debezium.connector.mysql.MySqlConnector",
      "tasks.max": "1",
      "database.hostname": "mysql",
      "database.port": "3306",
      "database.user": "debezium",
      "database.password": "dbz",
      "database.server.id": "184054",
      "database.server.name": "dbserver1",
      "database.include.list": "inventory",
      "database.history.kafka.bootstrap.servers": "kafka:9092",
      "database.history.kafka.topic": "schema-changes.inventory",
      "snapshot.mode": "initial"
    }
  }'

Step 3: Python으로 변경 이벤트 소비하기

consumer.py
from kafka import KafkaConsumer
import json

# Kafka Consumer 설정
consumer = KafkaConsumer(
    'dbserver1.inventory.customers',  # 토픽명: {server_name}.{database}.{table}
    bootstrap_servers=['localhost:9092'],
    auto_offset_reset='earliest',
    enable_auto_commit=True,
    value_deserializer=lambda x: json.loads(x.decode('utf-8'))
)

print("📢 MySQL 변경 이벤트 수신 대기중...")
for message in consumer:
    event = message.value
    op = event['op']  # 'c'=create, 'u'=update, 'd'=delete, 'r'=read (snapshot)
    
    if op == 'c':
        print(f"[INSERT] 새 레코드: {event['after']}")
    elif op == 'u':
        print(f"[UPDATE] 변경 전: {event['before']} → 변경 후: {event['after']}")
    elif op == 'd':
        print(f"[DELETE] 삭제된 레코드: {event['before']}")
    elif op == 'r':
        print(f"[SNAPSHOT] 초기 데이터: {event['after']}")
    
    # 추가 메타데이터 접근
    # source_ts = event['source']['ts_ms']
    # table = event['source']['table']
실행 순서: 1) `docker-compose up -d` → 2) 커넥터 등록(CURL) → 3) `python consumer.py` 실행 → 4) 별도 터미널에서 MySQL에 접속(`docker-compose exec mysql mysql -u root -pdebezium`)하여 `INSERT/UPDATE/DELETE` 쿼리 실행 → 5) Python Consumer에서 실시간 이벤트 확인.

5. 핵심 설정 레퍼런스

MySQL 커넥터를 기준으로 실무에서 자주 사용하거나 중요한 설정들을 정리했습니다.

설정 키 기본값 설명 실무 팁
snapshot.mode initial 초기 스냅샷 캡처 방식 설정 initial(기본), when_needed(필요시), never(스냅샷 안함), initial_only(스냅샷만) 중 선택. 대용량 DB는 initial 사용 시 주의.
snapshot.locking.mode minimal 스냅샷 중 테이블 락 방식 none(락 안함, 일관성 떨어짐), minimal(덤프 중 짧은 락), extended(전체 스냅샷 동안 락). 운영 DB는 minimal 권장.
database.server.id - MySQL 복제용 고유 ID 반드시 MySQL 클러스터 내에서 고유한 값으로 설정. 다른 커넥터/복제본과 중복 금지.
database.history io.debezium.relational.history.FileDatabaseHistory DDL 변경 히스토리 저장소 Kafka 기반 저장소(KafkaDatabaseHistory)를 사용하면 커넥터 장애 복구 시 스키마 이력을 유지할 수 있어 프로덕션에 적합.
include.schema.changes true DDL 변경 이벤트 전송 여부 true면 ALTER TABLE 등 스키마 변경도 별도 토픽에 기록. 스키마 추적이 필요 없거나 소음이 될 경우 false로 설정.
tombstones.on.delete true DELETE 이벤트 후 tombstone 레코드 생성 여부 Kafka 로그 압축을 위해 true 유지. false로 설정하면 삭제된 키의 이전 이벤트가 압축에서 제거되지 않을 수 있음.
decimal.handling.mode precise DECIMAL/NUMERIC 타입 처리 방식 precise(문자열), double(부동소수점), string(문자열). 정확한 계산이 필요하면 precise 유지.
max.batch.size 2048 폴링 당 최대 레코드 수 네트워크/처리량에 따라 조정. 너무 크면 메모리 부하, 너무 작으면 오버헤드 증가.
poll.interval.ms 500 로그 폴링 간격(ms) 실시간성을 높이려면 줄이고, DB 부하를 줄이려면 늘림. 일반적으로 100~1000ms 사이 설정.

6. 실무 팁 & 트러블슈팅

🔧 자주 겪는 문제와 해결책

문제: 커넥터 시작 시 스냅샷에서 멈춤

원인: 대용량 테이블의 스냅샷으로 시간이 오래 걸리거나, 락 경합 발생.

해결:

  • snapshot.mode: when_needed로 변경
  • snapshot.max.threads 증가로 병렬 처리
  • 저부하 시간대에 배치 또는 initial_only로 별도 실행

문제: MySQL 바이너리 로그 보관 주기 짧음

원인: expire_logs_days 설정으로 인해 커넥터가 재시작 시 필요한 로그가 삭제됨.

해결:

  • MySQL에서 expire_logs_days 값을 증가 (예: 7일 → 30일)
  • 커넥터 자주 중단하지 않도록 모니터링 및 안정화
  • 스냅샷부터 재시작해야 하는 경우 대비 전략 수립

문제: 이벤트 순서 보장 필요

원인: 파티셔닝, 멀티 스레드 처리 시 순서가 뒤섞일 수 있음.

해결:

  • 테이블/프라이머리 키 기준으로 파티셔닝 설정 (message.key.columns)
  • tasks.max: 1로 유지 (단일 태스크가 순서 보장)
  • 트랜잭션 경계 내 순서는 source.ts_ms로 판단 가능

🚀 실무 운영 팁

모니터링 필수 지표:
  • Connect Worker 상태: JVM 메모리, 스레드 풀, Deadlock
  • 커넥터 상태: `RUNNING`, `FAILED`, `PAUSED`
  • 지연 시간(Lag): `source.ts_ms`와 현재 시간 차이. 증가 시 병목 지점 확인.
  • 변경 이벤트 처리량: 초당 처리 이벤트 수, 배치 크기 추이.
주의: DDL 변경 시

ALTER TABLE로 컬럼 추가/삭제/타입 변경 시 기존 스트리밍 이벤트와의 호환성 문제가 발생할 수 있습니다. 스키마 레지스트리(Schema Registry)를 함께 사용하거나, 백필(backfill) 전략을 미리 수립하세요. 가능하면 Breaking Schema Change는 피하고, 호환되는 변경(컬럼 추가 등)을 수행하세요.

성능 튜닝 가이드라인:

  • 네트워크 대역폭: DB와 Kafka Connect, Kafka 브로커 간 네트워크 지연 시간 최소화.
  • Kafka 토픽 설정: 이벤트 볼륨에 맞는 파티션 수, 복제 계수, 보관 정책 설정.
  • 커넥터 메모리: 대용량 메시지(예: 큰 BLOB 컬럼) 처리 시 Connect Worker의 HEAP 증가 필요.
  • 병렬 처리:`tasks.max` 증가 및 `database.server.id` 범위 설정으로 병렬 처리 (단, 순서 보장 고려).
