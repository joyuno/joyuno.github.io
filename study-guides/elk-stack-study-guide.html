1. ELK Stack란?

ELK Stack은 Elasticsearch, Logstash, Kibana의 세 가지 오픈소스 프로젝트로 구성된 로그 및 데이터 분석 플랫폼입니다. 실시간 검색, 분석, 시각화를 위한 통합 솔루션을 제공하며, 최근에는 경량 데이터 수집기인 Beats의 추가로 Elastic Stack으로도 불립니다.

📈 실시간 검색 & 분석

Elasticsearch의 역인덱싱 엔진을 통해 수십 테라바이트의 데이터에서도 밀리초 단위로 검색하고 집계할 수 있습니다.

🔄 데이터 파이프라인

Logstash와 Beats가 다양한 소스(로그 파일, 메트릭, DB)에서 데이터를 수집, 변환(파싱, 필터링), Elasticsearch로 전송합니다.

📊 시각화 & 대시보드

Kibana를 통해 데이터를 차트, 그래프, 대시보드로 시각화하고, 실시간 모니터링 및 문제 진단을 수행합니다.

2. 왜 써야 하는가?

기존 로그 관리 솔루션(Graylog, Splunk, Loki) 대비 ELK Stack의 장점과 선택 기준을 비교합니다.

솔루션 장점 단점/고려사항 ELK Stack 대비 포지션
ELK Stack 높은 자유도, 강력한 검색/집계, 풍부한 생태계, 오픈소스 운용 복잡도 상대적 높음, 성능 튜닝 필요 기준점
Graylog 통합 UI, 스트림/알람 내장, 설정 간소화 검색 성능/확장성 제한, 커스터마이징 제약 통합 관리 편의성 중시 시
Splunk 엔터프라이즈 기능 완성도 높음, 강력한 지원 매우 높은 라이선스 비용 예산 충분 + 엔터프라이즈 요구사항多
Grafana Loki 경량, 저장 효율 높음, Grafana 통합 용이 로그 내용 검색 기능 제한적(라벨 기반) 메트릭과 로그 통합 + 비용 최소화 시

ELK Stack 선택 포인트: 오픈소스 기반의 높은 자유도와 커스터마이징이 필요할 때, 텍스트 기반의 풀텍스트 검색과 복잡한 집계가 주요 요구사항일 때, 그리고 장기적으로 다양한 데이터 소스(메트릭, 트레이스)까지 통합 플랫폼으로 확장하고 싶을 때 적합합니다.

3. 핵심 개념

ELK Stack의 기본 데이터 흐름과 핵심 컴포넌트의 역할을 이해합니다.

[데이터 소스] │ ▼ (Filebeat, Metricbeat, etc.) [Beats] ────┐ │ │ ▼ ▼ [Logstash] (수집, 파싱, 변환, 풍부화) │ ▼ (JSON 문서) [Elasticsearch] (인덱싱, 샤딩, 검색) │ ▼ (REST API) [Kibana] (검색, 시각화, 대시보드, 관리)

핵심 용어

  • 인덱스(Index): Elasticsearch에서의 논리적 데이터베이스. 하나 이상의 샤드로 구성.
  • 도큐먼트(Document): 인덱스에 저장되는 기본 데이터 단위(JSON 형식).
  • 샤드(Shard): 인덱스의 물리적 조각. 데이터 분산과 병렬 처리를 위해 프라이머리/레플리카 샤드로 구성.
  • Ingest Pipeline: Elasticsearch 내에서 도큐먼트를 변환하는 가벼운 처리 단계.
  • 인덱스 패턴(Index Pattern): Kibana에서 여러 인덱스를 하나의 논리적 뷰로 묶는 방법.

4. 퀵스타트 (Docker Compose 기반)

가장 빠르게 ELK Stack을 경험해보는 방법입니다.

단계 1: Docker Compose 파일 작성

docker-compose.yml
version: '3.8'
services:
  elasticsearch:
    image: docker.elastic.co/elasticsearch/elasticsearch:8.12.0
    environment:
      - discovery.type=single-node
      - xpack.security.enabled=false # 학습용으로 비활성화
      - "ES_JAVA_OPTS=-Xms512m -Xmx512m"
    ports:
      - "9200:9200"
    volumes:
      - es_data:/usr/share/elasticsearch/data

  logstash:
    image: docker.elastic.co/logstash/logstash:8.12.0
    ports:
      - "5000:5000/tcp"
      - "5000:5000/udp"
    volumes:
      - ./logstash-config:/usr/share/logstash/pipeline
    depends_on:
      - elasticsearch

  kibana:
    image: docker.elastic.co/kibana/kibana:8.12.0
    ports:
      - "5601:5601"
    environment:
      - ELASTICSEARCH_HOSTS=http://elasticsearch:9200
    depends_on:
      - elasticsearch

volumes:
  es_data:

단계 2: Logstash 파이프라인 설정

logstash-config/logstash.conf
input {
  tcp {
    port => 5000
    codec => json_lines
  }
}

filter {
  # 간단한 필터 예: timestamp 파싱
  date {
    match => [ "timestamp", "ISO8601" ]
  }
}

output {
  elasticsearch {
    hosts => ["http://elasticsearch:9200"]
    index => "logs-%{+YYYY.MM.dd}"
  }
  stdout { codec => rubydebug } # 콘솔 출력 (디버깅용)
}

단계 3: 실행 및 데이터 전송 테스트

터미널에서 다음 명령어를 실행합니다.

# 스택 실행
docker-compose up -d

# 상태 확인 (Elasticsearch가 준비될 때까지 기다림)
curl -X GET "http://localhost:9200/"

# Python으로 샘플 로그 전송
send_log.py
import socket
import json
import time

def send_log_to_logstash(message):
    sock = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
    sock.connect(('localhost', 5000))
    log_entry = {
        "timestamp": time.strftime("%Y-%m-%dT%H:%M:%SZ", time.gmtime()),
        "level": "ERROR",
        "service": "payment-api",
        "message": message,
        "user_id": 12345
    }
    sock.sendall((json.dumps(log_entry) + "\n").encode())
    sock.close()

if __name__ == "__main__":
    send_log_to_logstash("Failed to process transaction: insufficient funds")
    print("로그 전송 완료. Kibana (http://localhost:5601)에서 확인하세요.")

주의: 프로덕션 환경에서는 보안 설정(xpack.security.enabled=true)을 반드시 활성화하고, 적절한 리소스 할당 및 클러스터 구성을 해야 합니다.

5. 핵심 설정 레퍼런스

실무에서 자주 접하는 주요 구성 요소들의 설정 포인트입니다.

컴포넌트 설정 파일/위치 핵심 파라미터 설명/권장값
Elasticsearch elasticsearch.yml cluster.name, node.name, network.host, discovery.seed_hosts, path.data, path.logs 클러스터 이름을 명시적으로 지정. network.host는 바인딩할 IP. 데이터/로그 경로는 별도 볼륨 권장.
Elasticsearch JVM 옵션 (jvm.options) -Xms, -Xmx 힙 메모리 크기. 시스템 메모리의 50% 이하로 설정, Xms와 Xmx를 동일하게.
Logstash pipeline/*.conf input {}, filter {}, output {} 파이프라인 단위 구성. filter에서 grok, mutate, date 플러그인을 많이 사용.
Logstash logstash.yml pipeline.workers, pipeline.batch.size, queue.type workers는 CPU 코어 수. Persistent Queue(queue.type: persisted) 사용 시 데이터 손실 방지.
Filebeat filebeat.yml inputs (type: log, paths), output.logstash.hosts, processors 여러 로그 파일을 inputs에 배열로 정의. add_fields, drop_event 등 프로세서로 전처리 가능.
Kibana kibana.yml server.host, server.port, elasticsearch.hosts, elasticsearch.username 프록시 뒤에 있을 경우 server.basePath 설정 필요. 보안 설정 시 username/password 필수.

6. 실무 팁 & 트러블슈팅

팁 1: 인덱스 라이프사이클 관리 (ILM)

데이터가 무한정 쌓이지 않도록 핫(활성) → 웜(검색 가능) → 콜드(저장 최적화) → 삭제 단계를 자동화합니다.

# ILM 정책 생성 예시 (Kibana Dev Tools 또는 API)
PUT _ilm/policy/logs-policy
{
  "policy": {
    "phases": {
      "hot": {
        "actions": {
          "rollover": {
            "max_size": "50gb",
            "max_age": "7d"
          }
        }
      },
      "delete": {
        "min_age": "30d",
        "actions": {
          "delete": {}
        }
      }
    }
  }
}

팁 2: 성능 저하 시 확인 포인트

  • Elasticsearch Heap 사용량 높음: jvm.options에서 -Xmx 조정, 불필요한 인덱스 삭제, 필드 수 감소.
  • 검색/색인 느림: 샤드 수 조정(너무 많거나 적지 않게), refresh_interval 증가(기본 1초), replica 일시적 감소.
  • Logstash 병목: pipeline.workers 증가, Persistent Queue 활성화, filter 플러그인 최적화.

트러블슈팅: 자주 겪는 문제

증상 원인 해결책
Kibana에서 인덱스 패턴 생성 불가 인덱스에 데이터 없음, 필드 매핑 문제, 권한 부족 1. Logstash/Beats로 데이터 전송 확인. 2. 인덱스명 와일드카드 정확성 확인. 3. 보안 설정 시 권한 체크.
"fielddata is too large" 에러 text 필드에 집계/정렬 시 fielddata 메모리 초과 1. fielddata 대신 keyword 필드 사용. 2. indices.fielddata.cache.size 증가(임시). 3. 쿼리 최적화.
Logstash에서 파싱 실패(grok) 로그 형식과 grok 패턴 불일치 1. Grok Debugger로 패턴 테스트. 2. dissect 또는 csv 필터로 대체 검토. 3. 실패한 이벤트는 _grokparsefailure 태그로 라우팅.
노드 Yellow/Red 상태 레플리카 샤드 할당 불가(노드 부족, 디스크 공간) 1. disk watermark 확인 및 정리. 2. 노드 추가 또는 레플리카 수 임시 조정(number_of_replicas).

모니터링은 필수: ELK Stack 자체를 모니터링하세요. Elasticsearch의 _cluster/health, _nodes/stats API를 정기적으로 점검하거나, Metricbeat로 Elasticsearch, Logstash, Kibana 메트릭을 수집해 별도 모니터링 시스템에 표시하는 것이 좋습니다.
