1. Splunk Enterprise Security란?

Splunk Enterprise Security(ES)는 Splunk 플랫폼 위에서 작동하는 보안 정보 및 이벤트 관리(SIEM) 솔루션입니다. 기업의 보안 운영 센터(SOC)에서 위협 탐지, 조사, 대응을 통합적으로 지원합니다.

통합 위협 탐지

규칙 기반 Correlation Search와 머신러닝 기반 Anomaly Detection을 결합하여 네트워크, 엔드포인트, 애플리케이션 로그에서 위협을 식별합니다.

조사 및 대응 워크플로우

Notable Event(주목할 만한 이벤트)를 생성하고, 조사 팀이 대응 액션(태깅, 주석, 승격)을 수행할 수 있는 체계적인 프레임워크를 제공합니다.

보안 지능 통합

외부 Threat Intelligence(위협 인텔)를 연동하여 컨텍스트를 추가하고, Asset & Identity 정보를 활용하여 위협의 영향을 평가합니다.

2. 왜 써야 하는가?

다른 SIEM/로그 분석 솔루션과 비교한 Splunk ES의 강점:

솔루션 강점 Splunk ES 비교
Elastic Security 오픈소스, 확장성, 통합 스토어 보다 정교한 워크플로우 관리, 상용 Threat Intel 연동 용이성, 강력한 대시보드 & 리포트
Microsoft Sentinel Azure 네이티브 통합, 클라우드 포커스 하이브리드 환경(온프레미스+클라우드) 지원 우수, 데이터 소스 다양성, 커스텀 Correlation 검색 편의성
기존 로그 분석 도구 단순 검색, 기본 통계 보안 특화 프레임워크(Notable, 리스크 점수), 규제 준수 리포트 자동화, 실시간 대응 액션

AI 엔지니어에게 중요한 점: Splunk ES는 머신러닝 모델을 통합하여 이상 탐지(Anomaly Detection)를 수행할 수 있으며, Python 기반 커스텀 알고리즘 연동이 가능합니다.

3. 핵심 개념

Splunk ES는 Splunk 플랫폼의 확장 앱(App)으로 구현됩니다. 주요 구성 요소와 데이터 흐름:

[데이터 소스] ├── 로그 (Firewall, Endpoint, Cloud) ├── 스트림 데이터 (Netflow, API) └── Threat Intel 피드 | [Splunk 플랫폼] ├── 인덱싱 & 저장 ├── 실시간 검색 엔진 └── 데이터 모델 (CIM) | [Enterprise Security 앱] ├── Correlation Search → Notable Event 생성 ├── Asset & Identity DB → 컨텍스트 추가 ├── Risk Analysis → 리스크 점수 계산 └── 대응 워크플로우 (조사, 태깅, 승격) | [출력] ├── 보안 대시보드 ├── 규제 준수 리포트 └── 외부 시스템 연동 (SOAR, 티켓)

주요 용어

  • CIM (Common Information Model): 로그 데이터의 정규화된 필드 이름을 정의하는 표준 모델.
  • Notable Event: Correlation Search 또는 외부 알림으로 생성된, 조사가 필요한 보안 이벤트.
  • Correlation Search: 주기적으로 실행되어 조건에 맞는 이벤트를 Notable로 전환하는 검색.
  • Asset & Identity Database: 조직의 자산(IP, 호스트명)과 사용자 정보를 저장하여 컨텍스트 제공.

4. 퀵스타트

4.1 설치 (Splunk 플랫폼 기준)

Splunk Enterprise가 설치된 상태에서 ES 앱을 설치합니다.

# Splunk CLI를 사용하여 ES 앱 설치 (예시)
$ splunk install app splunk-enterprise-security -update true

# 또는 관리자 웹 UI에서 "앱 찾기" → "Enterprise Security" 설치

4.2 초기 설정

ES 앱 설치 후 초기 구성 단계:

1. 관리자 → Enterprise Security → "초기 구성" 실행
2. 데이터 소스 연결 설정 (Firewall, Windows Event Log 등)
3. Asset & Identity DB 초기 데이터 입력 (CSV 또는 API)
4. 기본 Correlation Search 활성화 확인

4.3 첫 번째 Correlation Search 예제

Python 스크립트를 사용하여 외부 로그를 Splunk에 입력하고, 간단한 Correlation Search를 만듭니다.

# Python: 로그 전송 예제 (Splunk HEC 사용)
import requests
import json
import time

url = "https://your-splunk-server:8088/services/collector"
headers = {"Authorization": "Splunk YOUR_HEC_TOKEN"}
event = {
    "event": {
        "source_ip": "192.168.1.100",
        "dest_ip": "10.0.0.1",
        "action": "blocked",
        "log_source": "firewall"
    }
}
response = requests.post(url, headers=headers, data=json.dumps(event))
print(f"로그 전송 상태: {response.status_code}")

# Splunk Correlation Search (Splunk 검색 언어)
# 관리자 → Enterprise Security → Correlation Search 생성
search_name: "Firewall Multiple Block Events"
search_query: `
    index=firewall action="blocked"
    | stats count by source_ip
    | where count > 10
    | lookup asset_db source_ip OUTPUT asset_name, asset_tag
`
action: Create Notable Event
notable_title: "Possible Scanning Activity from {source_ip}"

5. 핵심 설정 레퍼런스

설정 항목 위치/명령어 설명
데이터 소스 연결 ES 앱 → 데이터 소스 관리 로그 수집기(Forwarder) 또는 직접 인덱스 연결 설정
Asset DB 업데이트 ES 앱 → Asset & Identity 관리 CSV 파일 업로드 또는 REST API 연동으로 자산 정보 갱신
Correlation Search 스케줄 Correlation Search 편집 → Schedule 탭 실행 간격(분, 시) 및 시간 범위 설정
Threat Intel 연동 ES 앱 → Threat Intelligence 관리 STIX/TAXII 피드 또는 커스텀 CSV 파일 추가
리스크 점수 계산 규칙 Risk Analysis 설정 Notable Event의 심각도, 자산 중요도에 따른 점수 계산 방식 정의

6. 실무 팁 & 트러블슈팅

자주 겪는 문제

문제 1: Notable Event가 생성되지 않음

해결책: Correlation Search의 검색 조건이 실제 데이터와 맞는지 확인. 인덱스 이름, 필드 이름(CIM 준수) 검증. 검색 스케줄이 활성화되었는지 확인.

문제 2: Asset & Identity 정보가 Notable에 표시되지 않음

해결책: Asset DB의 IP/호스트명 필드와 로그 데이터의 필드가 일치해야 함. lookup 명령어 정확성 확인. DB 업데이트 주기 확인.

문제 3: 성능 문제 (검색 느림)

해결책: 인덱스 파티셔닝 적용, 불필요한 필드 제거, 검색 시간 범위 최적화. Summary Index 사용 고려.

실무 팁

  • CIM 준수: 모든 커스텀 로그 소스를 CIM 데이터 모델에 맞게 정규화하면 Correlation Search 작성이 쉬워집니다.
  • Python 연동: Splunk Python SDK를 사용하여 외부 시스템에서 Notable을 생성하거나 조사 데이터를 가져올 수 있습니다.
  • 대응 자동화: ES의 워크플로우 액션과 Splunk Phantom(SOAR)을 연동하여 대응을 자동화할 수 있습니다.
