1. Kafka란? — 왜 써야 하는가

정의

Apache Kafka는 분산 이벤트 스트리밍 플랫폼이다. LinkedIn에서 처음 개발되었고, 현재 Apache Software Foundation이 관리한다. 단순 메시지 큐가 아니라 실시간 데이터 파이프라인의 중추 신경 역할을 한다.

3대 핵심 능력

Publish / Subscribe

이벤트 스트림을 발행(publish)하고 구독(subscribe). 외부 시스템에서 연속적으로 데이터를 import/export할 수 있다.

Store

이벤트 스트림을 내구성 있게(durably) 원하는 기간만큼 저장. 기본 7일이지만 무제한 보관도 가능.

Process

이벤트가 발생하는 즉시 또는 소급하여(retrospectively) 스트림을 처리할 수 있다.

왜 Kafka를 쓰는가? (vs 전통 MQ)

항목전통 MQ (RabbitMQ 등)Apache Kafka
메시지 수명소비 후 삭제보존 기간 동안 영구 보관 (재소비 가능)
처리량수천 msg/s수백만 msg/s (초당 수 GB)
통신 모델Point-to-Point 중심Pub/Sub + Consumer Group (둘 다)
확장성수직 확장 중심파티션 기반 수평 확장
내구성메모리 중심디스크 기반 + 복제 (fault-tolerant)
순서 보장큐 단위파티션 단위 순서 보장
핵심 포인트

Kafka는 "메시지를 보내고 끝"이 아니라, 이벤트 로그를 중심으로 시스템 전체를 연결하는 데이터 허브로 쓴다. 데이터 엔지니어에게 Kafka는 ETL/ELT 파이프라인의 중추이고, AI 엔지니어에게는 실시간 피처 전달 및 모델 서빙의 백본이다.

대표 유즈케이스

  • 실시간 로그 수집(Log Aggregation) — 여러 서버의 로그를 중앙으로 모은다
  • 이벤트 소싱(Event Sourcing) — 상태 변경을 이벤트로 기록
  • CDC(Change Data Capture) — DB 변경분을 실시간 전파 (Debezium)
  • 실시간 스트림 처리 — Flink/Spark Streaming과 연동
  • 마이크로서비스 간 통신 — 비동기 이벤트 기반 아키텍처
  • ML Feature Store 업데이트 — 실시간 피처 파이프라인

2. 핵심 개념 6가지

Event (Record / Message)

"무언가 일어났다"는 사실의 기록. key, value, timestamp, 그리고 선택적 헤더로 구성된다.

Topic

이벤트가 흐르는 논리적 채널. 파일시스템의 폴더, DB의 테이블과 유사한 개념. 토픽 수에 제한 없음.

Partition

토픽을 물리적으로 분할한 단위. 병렬 처리의 기본 단위이자 저장의 기본 단위. 각 이벤트는 파티션 내에서 고유한 offset을 갖는다.

Broker

Kafka 서버 1대 = 1 Broker. 클러스터는 보통 최소 3대로 구성 (1대 유지보수 + 1대 장애 허용).

Producer

이벤트를 토픽에 쓰는 클라이언트. 어떤 파티션에 보낼지 결정. Consumer와 완전히 디커플링되어 있다.

Consumer / Consumer Group

토픽을 구독하여 이벤트를 읽는 클라이언트. Consumer Group 내에서 각 파티션은 1개의 Consumer에만 할당되어 병렬 소비한다.

토픽-파티션-오프셋 관계도

Topic: "user-events" ┌─────────────────────────────────────────────────────────┐ │ │ │ Partition 0: [0] [1] [2] [3] [4] [5] [6] ──▶ write │ │ Partition 1: [0] [1] [2] [3] [4] ──▶ write │ │ Partition 2: [0] [1] [2] [3] [4] [5] ──▶ write │ │ │ │ ※ 각 파티션 내에서 offset은 절대 재사용되지 않는다 │ │ ※ 파티션 간 순서 보장은 없다 │ └─────────────────────────────────────────────────────────┘

Consumer Group 동작 방식

Topic "orders" (3 partitions) P0 ──▶ Consumer A ┐ P1 ──▶ Consumer B ├── Consumer Group "order-processor" P2 ──▶ Consumer C ┘ P0 ──▶ Consumer X ┐ P1 ──▶ Consumer X ├── Consumer Group "analytics" P2 ──▶ Consumer X ┘ ※ 같은 그룹 내 1개 파티션 = 1개 컨슈머 (병렬 처리) ※ 다른 그룹은 독립적으로 같은 메시지를 읽을 수 있다

Replication (복제)

각 파티션은 여러 브로커에 복제된다. 하나가 Leader(읽기/쓰기 담당), 나머지는 Follower(복제만). Leader 장애 시 Follower 중 하나가 새 Leader로 승격된다.

  • replication.factor=3 → 데이터 3벌 보관 (운영 권장값)
  • ISR (In-Sync Replicas) — Leader와 동기화된 Replica 집합. ISR에 포함된 Follower만 Leader 후보가 될 수 있다.

4. KRaft 모드 — ZooKeeper 없는 Kafka

중요 타임라인

Kafka 4.0부터 ZooKeeper 지원이 완전히 제거되었다. 3.9가 ZooKeeper를 지원하는 마지막 버전이며, 2025년 11월부터 ZooKeeper 기반 클러스터는 공식 지원이 종료된다. 지금 새로 시작한다면 무조건 KRaft 모드를 사용해야 한다.

KRaft란?

KRaft(Kafka Raft)는 ZooKeeper를 대체하는 Kafka 자체 내장 메타데이터 관리 프로토콜이다. Raft 합의 알고리즘의 이벤트 기반 변형을 사용하여, 별도 클러스터(ZooKeeper) 없이 Kafka 브로커만으로 메타데이터를 관리한다.

KRaft의 장점

  • 운영 단순화 — ZooKeeper 클러스터를 별도로 관리할 필요 없음
  • 더 빠른 컨트롤러 페일오버 — 메타데이터가 Kafka 내부에 있으므로 전환 속도 향상
  • 확장성 향상 — 수백만 파티션까지 더 효율적으로 관리
  • 배포 간소화 — 하나의 시스템만 관리하면 됨

KRaft 모드 설치 (로컬 개발용)

# 1. 클러스터 UUID 생성
KAFKA_CLUSTER_ID="$(bin/kafka-storage.sh random-uuid)"

# 2. 로그 디렉토리 포맷
bin/kafka-storage.sh format \
  -t $KAFKA_CLUSTER_ID \
  -c config/kraft/server.properties

# 3. Kafka 서버 시작 (KRaft 모드)
bin/kafka-server-start.sh config/kraft/server.properties

KRaft 핵심 설정

# config/kraft/server.properties 주요 항목

# 노드 역할: controller, broker, 또는 둘 다 (개발용: controller,broker)
process.roles=broker,controller

# 노드 ID (클러스터 내 고유)
node.id=1

# 컨트롤러 쿼럼 투표자 목록
controller.quorum.voters=1@localhost:9093

# 리스너 설정
listeners=PLAINTEXT://:9092,CONTROLLER://:9093
controller.listener.names=CONTROLLER

# 로그 디렉토리
log.dirs=/tmp/kraft-combined-logs

Producer 예제 (confluent-kafka)

from confluent_kafka import Producer
import json

# 설정
conf = {
    'bootstrap.servers': 'localhost:9092',
    'client.id': 'my-python-producer',
    # 성능 튜닝
    'linger.ms': 10,          # 배치 대기 시간
    'batch.size': 16384,      # 배치 크기 (bytes)
    'compression.type': 'lz4' # 압축
}

producer = Producer(conf)

# 전송 콜백
def delivery_callback(err, msg):
    if err:
        print(f'전송 실패: {err}')
    else:
        print(f'전송 성공: {msg.topic()} [{msg.partition()}] @ {msg.offset()}')

# 메시지 전송
for i in range(10):
    event = {'user_id': i, 'action': 'page_view', 'page': f'/products/{i}'}
    producer.produce(
        topic='user-events',
        key=str(i),               # 같은 key → 같은 파티션
        value=json.dumps(event),
        callback=delivery_callback
    )

# 버퍼에 남은 메시지 전송 완료 대기
producer.flush()

Consumer 예제 (confluent-kafka)

from confluent_kafka import Consumer, KafkaError
import json

conf = {
    'bootstrap.servers': 'localhost:9092',
    'group.id': 'my-consumer-group',
    'auto.offset.reset': 'earliest',   # 처음부터 읽기
    'enable.auto.commit': True,         # 자동 오프셋 커밋
    'auto.commit.interval.ms': 5000
}

consumer = Consumer(conf)
consumer.subscribe(['user-events'])

try:
    while True:
        msg = consumer.poll(timeout=1.0)  # 1초 대기

        if msg is None:
            continue
        if msg.error():
            if msg.error().code() == KafkaError._PARTITION_EOF:
                print(f'파티션 끝 도달: {msg.topic()} [{msg.partition()}]')
            else:
                print(f'에러: {msg.error()}')
            continue

        # 메시지 처리
        event = json.loads(msg.value().decode('utf-8'))
        print(f'수신: partition={msg.partition()}, '
              f'offset={msg.offset()}, event={event}')

except KeyboardInterrupt:
    pass
finally:
    consumer.close()  # Consumer Group 탈퇴 & 오프셋 커밋
주의사항

KafkaProducer는 스레드 안전(thread-safe)하지만, KafkaConsumer는 그렇지 않다. Consumer를 멀티스레드로 사용하려면 thread-local로 만들거나, multiprocessing을 권장한다.

8. 메시지 전달 보장 (Delivery Semantics)

수준설정특징
At-Most-Once acks=0 메시지 손실 가능. 처리량 최대. 로그/메트릭 등 손실 허용 가능한 경우.
At-Least-Once acks=all + 재시도 메시지 중복 가능. 대부분의 운영 환경 기본값. Consumer 측 멱등성 필요.
Exactly-Once enable.idempotence=true + 트랜잭션 API 중복·손실 없음. 결제 등 정확성 필수 시나리오. 약간의 성능 오버헤드.

운영 권장 조합

# Producer: 안전한 기본값
acks=all
enable.idempotence=true
max.in.flight.requests.per.connection=5
retries=2147483647

# Broker: 데이터 안전성
min.insync.replicas=2
unclean.leader.election.enable=false

# Consumer: 수동 커밋으로 정확한 처리
enable.auto.commit=false
# → 메시지 처리 완료 후 consumer.commit() 호출
실무 팁

대부분의 데이터 파이프라인은 At-Least-Once + Consumer 측 멱등성(Idempotency)이 현실적 최선이다. Exactly-Once는 Kafka Streams 내부 처리에서 가장 쉽게 달성할 수 있다.

9. Kafka + AI/ML 파이프라인

왜 AI 파이프라인에 Kafka가 필요한가?

대부분의 AI 실패는 모델 정확도 문제가 아니라 데이터 파이프라인 병목에서 온다. 배치 파이프라인으로 실시간 예측을 서빙하려는 것이 근본 원인이다. Kafka는 이 문제를 해결하는 실시간 데이터 백본이다.

핵심 아키텍처 패턴

Kafka-ML 프레임워크

Kafka-ML은 Kafka의 고처리량·저지연 특성을 활용하여 ML/AI 파이프라인을 관리하는 오픈소스 프레임워크다. TensorFlow와 PyTorch를 지원하며, 연속적인 데이터 스트림에서 동작하는 ML 시스템 구축에 특화되어 있다.

모델 드리프트 대응

Kafka + Flink 조합으로 온라인 ML 학습과 모델 드리프트 감지를 구현할 수 있다. 연속적인 모델 훈련과 실시간 추론을 통해 항상 최신 상태의 고정확도 예측을 유지할 수 있다.

간단한 예제: Kafka → ML 추론

from confluent_kafka import Consumer, Producer
import json, pickle

# 학습된 모델 로드
model = pickle.load(open('fraud_model.pkl', 'rb'))

consumer = Consumer({
    'bootstrap.servers': 'localhost:9092',
    'group.id': 'ml-inference',
    'auto.offset.reset': 'latest'
})
consumer.subscribe(['transactions'])

producer = Producer({'bootstrap.servers': 'localhost:9092'})

while True:
    msg = consumer.poll(1.0)
    if msg is None or msg.error():
        continue

    # 1. 이벤트 파싱
    event = json.loads(msg.value().decode('utf-8'))

    # 2. 피처 추출
    features = extract_features(event)  # 사용자 정의 함수

    # 3. 모델 추론
    prediction = model.predict([features])[0]
    score = model.predict_proba([features])[0][1]

    # 4. 결과를 다른 토픽에 발행
    result = {
        'transaction_id': event['id'],
        'is_fraud': bool(prediction),
        'fraud_score': float(score)
    }
    producer.produce(
        'fraud-results',
        key=str(event['id']),
        value=json.dumps(result)
    )
    producer.flush()

자주 겪는 문제 & 해결

Consumer Lag 증가

  • 원인: Consumer 처리 속도 < Producer 속도
  • 해결: 파티션 수 늘리기 + Consumer 인스턴스 추가, max.poll.records 조정, 처리 로직 최적화

Rebalancing이 너무 자주 발생

  • 원인: session.timeout.ms가 너무 짧거나, max.poll.interval.ms 초과
  • 해결: session.timeout.ms 늘리기 (45s → 60s), 무거운 처리는 별도 스레드로 분리

메시지 순서가 뒤바뀜

  • 원인: 같은 key의 메시지가 다른 파티션에 들어감, 또는 재시도 시 순서 역전
  • 해결: key 기반 파티셔닝 확인, enable.idempotence=true + max.in.flight.requests.per.connection=5

디스크 부족

  • 원인: retention 기간이 너무 길거나, 토픽이 과도하게 많음
  • 해결: log.retention.hours 줄이기, log.retention.bytes 설정, 불필요한 토픽 정리
