1. ClickHouse란?

ClickHouse는 Yandex가 개발한 컬럼형 오픈소스 OLAP 데이터베이스 관리 시스템(DBMS)입니다. 수 페타바이트 규모의 데이터에 대해 아주 빠른 애널리틱 쿼리를 실행하도록 설계되었습니다.

⚡ 실시간 분석

대규모 데이터셋에 대한 집계 쿼리를 밀리초에서 초 단위로 응답합니다. 데이터 수신과 동시에 분석 쿼리가 가능합니다.

📊 컬럼형 스토리지

로우 단위가 아닌 컬럼 단위로 데이터를 저장하고 압축합니다. 분석 쿼리에서 필요한 컬럼만 읽어 성능을 극대화합니다.

🔧 대량 병렬 처리(MPP)

쿼리를 여러 서버와 코어에 분산하여 병렬 실행합니다. 선형 확장성을 제공하며, 클러스터 구성이 용이합니다.

2. 왜 써야 하는가?

기존 대안과의 비교를 통해 ClickHouse의 적합한 사용 사례를 확인합니다.

시스템 유형 ClickHouse 대비 장점 ClickHouse 대비 단점 적합한 작업
ClickHouse 컬럼형 OLAP 압도적인 실시간 분석 속도, 뛰어난 압축률, SQL 지원 트랜잭션 지원 부족, 고빈도 단일 행 업데이트 비적합 실시간 대시보드, 로그 분석, 사용자 행동 분석
PostgreSQL 로우형 OLTP ACID 트랜잭션, 풍부한 기능, 확장성 대규모 분석 쿼리 속도 느림, 컬럼형 압축 없음 트랜잭션 처리, 운영 데이터 저장
Apache Druid 컬럼형 OLAP 이벤트 스트림 수집 최적화, 고가용성 설정 복잡성, SQL 기능 제한적 이벤트 기반 시계열 데이터, 실시간 수집 분석
BigQuery / Snowflake 클라우드 DWH 완전 관리형, 자동 확장, 서버리스 비용(스캔 기반), Vendor Lock-in, 온프레미스 불가 클라우드 기반 데이터 웨어하우스, 간편한 관리
결론: 초고속 실시간 분석, 대용량 배치 집계, 비용 효율적인 대규모 데이터 저장이 필요할 때 ClickHouse는 최고의 선택지입니다.

3. 핵심 개념

ClickHouse의 아키 처는 데이터 수신, 저장, 처리, 제공의 단순하고 효율적인 흐름을 따릅 니다.

┌─────────────────────────────────────────────────────────────────────┐ │ Client Applications │ │ (Python, Go, BI Tools, Grafana, SQL Clients) │ └──────────────────────────────┬──────────────────────────────────────┘ │ (HTTP/TCP, Native Protocol) ▼ ┌─────────────────────────────────────────────────────────────────────┐ │ ClickHouse Server (Single Node) │ ├─────────────┬─────────────┬──────────────┬────────────┬────────────┤ │ Parser │ Analyzer │ Optimizer │ Interpreter│ Storage │ │ (SQL → │ (Query │ (Query │ (Query │ Engine │ │ AST) │ Semantics) │ Plan) │ Execution) │ (MergeTree)│ ├─────────────┴─────────────┴──────────────┴────────────┴────────────┤ │ Columnar Storage on Local Disk / S3 │ └─────────────────────────────────────────────────────────────────────┘ │ ┌─────────────────────────────────────────────────────────────────────┐ │ ClickHouse Cluster (Sharding) │ ├───────────────────┬───────────────────┬─────────────────────────────┤ │ Shard 1 │ Shard 2 │ ... (Distributed Table) │ │ (Replica Set) │ (Replica Set) │ │ └───────────────────┴───────────────────┴─────────────────────────────┘

주요 엔진

  • MergeTree: 모든 프로덕션 용도의 핵심 엔진. 데이터 파티셔닝, 인덱싱, 병합을 관리.
  • ReplicatedMergeTree: 고가용성을 위한 MergeTree의 복제 버전.
  • Distributed: 물리적 테이블이 아닌, 클러스터 전체에 쿼리를 분산하는 "가상" 엔진.
  • Kafka: Apache Kafka 토픽에서 직접 데이터를 수신하는 엔진.

4. 퀵스타트

4.1 도커로 설치하기

CLI
# 최신 안정 버전 도커 컨테이너 실행
docker run -d \
  --name some-clickhouse-server \
  -p 8123:8123 -p 9000:9000 -p 9009:9009 \
  --ulimit nofile=262144:262144 \
  clickhouse/clickhouse-server:latest

4.2 Python 클라이언트 연결 및 첫 테이블 생성

Python (clickhouse-driver 설치 필요)
# pip install clickhouse-driver pandas
from clickhouse_driver import Client

# 클라이언트 연결 (도커 기본 설정)
client = Client(host='localhost', port=9000, user='default', password='')

# 테이블 생성 (MergeTree 엔진)
create_table_sql = """
CREATE TABLE IF NOT EXISTS tutorial.user_events
(
    user_id UInt64,
    event_time DateTime,
    event_type String,
    country_code LowCardinality(String),
    revenue Decimal(10, 2)
)
ENGINE = MergeTree()
PARTITION BY toYYYYMM(event_time)
ORDER BY (user_id, event_time)
"""
client.execute(create_table_sql)
print("테이블 생성 완료.")

4.3 데이터 삽입 및 조회

# 배치 데이터 삽입
insert_sql = "INSERT INTO tutorial.user_events VALUES"
data = [
    (101, '2026-03-15 10:00:00', 'purchase', 'US', 49.99),
    (101, '2026-03-15 10:05:00', 'view', 'US', 0),
    (102, '2026-03-15 11:00:00', 'purchase', 'KR', 29.99),
    (103, '2026-03-15 12:00:00', 'login', 'JP', 0),
]
client.execute(insert_sql, data)

# 분석 쿼리 실행
result = client.execute("""
    SELECT
        country_code,
        count() as total_events,
        sum(revenue) as total_revenue,
        avg(revenue) as avg_revenue
    FROM tutorial.user_events
    WHERE event_time >= '2026-03-15'
    GROUP BY country_code
    ORDER BY total_revenue DESC
""")
for row in result:
    print(row)
참고: 프로덕션 환경에서는 반드시 적절한 사용자, 비밀번호, 네트워크 보안 설정을 구성하세요.

5. 핵심 설정 레퍼런스

주로 수정하는 설정 파일 (/etc/clickhouse-server/config.xml 또는 users.xml)의 주요 항목입니다.

설정 경로 기본값 권장/설명 영향
<listen_host> :: 프로덕션: 특정 IP 또는 0.0.0.0 네트워크 접근 제어
<max_concurrent_queries> 100 시스템 리소스에 따라 조정 (예: 200) 동시 쿼리 수 제한
<max_memory_usage> 10GB 물리적 RAM의 70-80%로 설정 단일 쿼리 메모리 제한
<background_pool_size> 16 CPU 코어 수에 맞춰 증가 (예: 32) 병렬 머지/뮤테이션 성능
<compression> lz4 압축률 높임: zstd, 속도 우선: lz4 저장 공간 vs 쿼리 속도
유저 설정 <max_execution_time> 0 (무제한) 예: 300 (초) - 장시간 쿼리 방지 쿼리 타임아웃

6. 실무 팁 & 트러블슈팅

🚀 성능 팁

  • 파티셔닝: 너무 많은 파티션(수천 개)을 만들지 마세요. 월/일 단위가 일반적.
  • ORDER BY 키: 가장 자주 필터링하는 컬럼을 앞에 배치하세요.
  • 적절한 데이터 타입: LowCardinality(String)으로 카디널리티가 낮은 문자열을 압축하세요.

🔧 운영 팁

  • 모니터링: 시스템 메트릭스(system.metrics), 쿼리 로그(system.query_log)를 적극 활용.
  • TTL: 데이터 보관 정책을 위해 테이블/컬럼 수준 TTL을 설정하세요.
  • 머지 최적화: optimize table ... final은 프로덕션에서 신중하게 사용.

자주 겪는 문제

문제: "Memory limit (for query) exceeded" 오류.
원인: 집계 시 중간 상태가 너무 커지거나, DISTINCT, GROUP BY 카디널리티가 매우 높음.
해결: 1) max_memory_usage 증가, 2) 샘플링 쿼리 사용, 3) GROUP BY 컬럼 최적화.
문제: Too many parts (파티션 수 백/천 개)로 인한 느린 쿼리.
원인: 소량의 데이터를 매우 자주 삽입하면 각 삽입이 별도의 파트를 생성.
해결: 1) 더 큰 배치로 삽입, 2) optimize table 실행, 3) 백그라운드 머지 설정 조정.
문제: 복제 지연 (ReplicatedMergeTree).
원인: 네트워크 지연, 디스크 I/O 병목, ZooKeeper 성능 이슈.
해결: 1) ZooKeeper 모니터링, 2) system.replication_queue 확인, 3) 네트워크 대역폭 점검.
