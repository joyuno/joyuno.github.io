1. Ray Serve란?

Ray Serve는 Ray 생태계의 고성능 모델 서빙 프레임워크로, Python으로 작성된 머신러닝 모델 및 임의의 비즈니스 로직을 위한 확장 가능한 모델 서빙 레이어를 제공합니다. 단일 머신에서 대규모 클러스터까지 일관된 API로 서비스를 배포하고 관리할 수 있습니다.

🚀 분산 서빙

단일 애플리케이션으로 여러 모델(Ensemble, Pipeline)을 하나의 통합 서비스로 배포. 자동 로드 밸런싱과 내결함성을 제공합니다.

⚙️ 동적 스케일링

트래픽 패턴에 따라 replica 수를 자동으로 조정(Autoscaler). 리소스 사용률을 최적화하면서도 짧은 지연 시간을 보장합니다.

🔗 다중 프레임워크

PyTorch, TensorFlow, Scikit-learn, XGBoost, 임의의 Python 코드를 동일한 서빙 인프라 위에서 운영. Vendor Lock-in을 피할 수 있습니다.

2. 왜 써야 하는가?

기존 모델 서빙 솔루션 대비 Ray Serve가 제공하는 실무적 이점을 비교합니다.

솔루션 주요 특징 Ray Serve 대비 차이점
Flask/FastAPI + Gunicorn 간단한 REST API, 친숙한 생태계 수동 스케일링, 분산 처리 부재, 모델 관리 기능 미비
TensorFlow Serving TF 모델 전용, 고성능 프레임워크 종속적, 멀티 프레임워크 지원 안 됨
KServe (KFServing) Kubernetes 네이티브, 표준 인터페이스 설정 복잡도 높음, Ray Serve는 KServe 위에서도 실행 가능
Seldon Core 엔터프라이즈 기능, Advanced Metrics 학습 곡선 가파름, Ray Serve는 더 경량화된 개발자 경험

💡 선택 가이드

Ray Serve를 선택해야 할 때: (1) Python 중심의 빠른 프로토타이핑과 프로덕션 전환을 원할 때, (2) 단일 인프라로 다양한 프레임워크의 모델을 서빙할 때, (3) Ray의 분산 컴퓨팅 생태계(예: Ray Train, Ray Tune)와 통합이 필요할 때.

3. 핵심 개념

Ray Serve의 아키텍처는 사용자 정의 가능한 Deployment를 기본 빌딩 블록으로 하여, Ray Cluster 위에서 분산 실행됩니다.

Client (HTTP/gRPC) | v +-------------------------------+ | Ray Serve API Gateway | | (HTTP Proxy, gRPC Proxy) | +-------------------------------+ | | (내부 RPC) v +-------------------------------+ | Deployment Graph | | +---------+ +---------+ | | | DAG | | Model | | | | Node | | A | | | +---------+ +---------+ | | | | | | v v | | +---------+ +---------+ | | | Model | | Model | | | | B | | C | | | +---------+ +---------+ | +-------------------------------+ | | (Ray GCS - Global Control Store) v +------------------------------------------------+ | Ray Cluster | | +-------------+ +-------------+ | | | Head Node | | Worker | | | | - Serve | | Node 1 | | | | Controller | | - Replica | | | | - API | | - Model A | | | | Gateway | | + Replica | | | +-------------+ | - Model B | | | | +-------------+ | | |-------------------------> +-------------+ | | Worker | | | Node N | | | - Replica | | | - Model C | | +-------------+ +------------------------------------------------+

주요 컴포넌트

  • Deployment: 서빙할 모델이나 로직의 단위. Python Class로 정의.
  • Replica: Deployment의 인스턴스. 수평 확장의 기본 단위.
  • API Gateway: 외부 요청을 받아 적절한 Replica로 라우팅.
  • Serve Controller: Deployment의 라이프사이클(생성, 스케일링, 업데이트) 관리.
  • Ray Cluster: 분산 실행을 위한 기반 인프라.

4. 퀵스타트

5분 안에 첫 Ray Serve 애플리케이션을 배포해봅니다.

Step 1: 설치

# Ray Core + Serve 설치
pip install "ray[serve]"

⚠️ 호환성 확인

Python 3.8 이상 필요. Linux/macOS 환경을 권장하며, Windows는 WSL2에서 실행해야 완전한 기능을 사용할 수 있습니다.

Step 2: 기본 애플리케이션 작성 (app.py)

from ray import serve
from ray.serve.handle import DeploymentHandle
import requests

# 1. Deployment 정의
@serve.deployment
class TextProcessor:
    def __init__(self):
        self.prefix = "Processed: "

    def process(self, text: str) -> str:
        return self.prefix + text.upper()

# 2. Deployment 그래프 구성 (Ensemble 예제)
@serve.deployment
class EnsembleDeployment:
    def __init__(self, processor: DeploymentHandle):
        self.processor = processor

    async def __call__(self, request) -> dict:
        data = await request.json()
        text = data.get("text", "")
        # 다른 Deployment 호출
        processed = await self.processor.process.remote(text)
        return {"original": text, "result": processed}

# 3. 애플리케이션 빌드 및 바인딩
processor_app = TextProcessor.bind()
ensemble_app = EnsembleDeployment.bind(processor_app)

# 또는 최신 ServeApplication API 사용
app = ensemble_app

Step 3: 로컬에서 서비스 실행

# 터미널 1: Ray 클러스터 시작
ray start --head

# 터미널 2: Serve 애플리케이션 배포
serve run app:ensemble_app
# 또는 특정 포트와 호스트 지정
# serve run --host 0.0.0.0 --port 8000 app:ensemble_app

Step 4: 요청 테스트

# Python 클라이언트
import requests
response = requests.post("http://localhost:8000/",
                         json={"text": "hello ray serve"})
print(response.json())
# 출력: {"original": "hello ray serve", "result": "Processed: HELLO RAY SERVE"}

# curl 사용
curl -X POST http://localhost:8000/ \
     -H "Content-Type: application/json" \
     -d '{"text": "test input"}'

5. 핵심 설정 레퍼런스

프로덕션 배포를 위해 알아야 할 주요 Deployment 설정 옵션입니다.

옵션 기본값 설명 실무 예시
num_replicas 1 Deployment의 복제본 수. 수평 확장 기본 매개변수. @serve.deployment(num_replicas=4)
autoscaling_config None 자동 스케일링 정책. target_num_ongoing_requests_per_replica로 트래픽 기반 조정. autoscaling_config={"target_num_ongoing_requests_per_replica": 10, "min_replicas": 2, "max_replicas": 10}
ray_actor_options {} Replica Actor의 리소스 설정 (num_cpus, num_gpus, memory). ray_actor_options={"num_cpus": 2, "num_gpus": 1}
max_concurrent_queries 100 Replica당 동시 처리 가능한 요청 수. 배치 처리 시 중요. max_concurrent_queries=5 (무거운 모델용)
health_check_period_s 10 헬스 체크 주기(초). 비정상 Replica 자동 재시작. health_check_period_s=30
graceful_shutdown_timeout_s 20 우아한 종료 대기 시간. 진행 중인 요청 완료 보장. graceful_shutdown_timeout_s=60

📝 설정 팁

@serve.deployment 데코레이터의 설정은 런타임에 serve.status() 또는 Dashboard에서 확인 가능합니다. Kubernetes 환경에서는 serve config를 YAML로 익스포트하여 GitOps에 활용하세요.

6. 실무 팁 & 트러블슈팅

🔧 자주 겪는 문제와 해결책

문제: GPU 메모리 누수

PyTorch/TF 모델 서빙 시 GPU 메모리가 서서히 증가.

해결: Deployment 클래스의 __call__ 메서드 내에서 torch.cuda.empty_cache() 호출. 또는 ray_actor_options={"num_gpus": 1}로 GPU 격리.

문제: Cold Start 지연

첫 요청 시 모델 로딩으로 인한 높은 지연 시간.

해결: @serve.deployment(ray_actor_options={"lifetime": "detached"})로 Actor 재사용. 또는 min_replicas를 1 이상으로 설정하여 상시 대기.

문제: 의존성 패키지 충돌

서로 다른 모델의 라이브러리 버전 요구사항 충돌.

해결: 각 Deployment를 별도의 Runtime Environment로 격리. ray_actor_options={"runtime_env": {"pip": ["torch==1.13.1"]}}

문제: 배포 업데이트 실패

새 버전 배포 시 일부 Replica가 오래된 코드 실행.

해결: Rolling Update 정책 사용. serve deploy config.yaml --yes 전에 serve status로 현재 상태 확인. 단계적 업데이트를 위해 --num-old-versions-to-keep 옵션 활용.

🚀 성능 최적화 팁

  • 배치 처리: @serve.batch 데코레이터로 여러 요청을 묶어 처리하여 처리량(throughput) 극대화.
  • 메트릭 수집: 내장 Counter, Histogram 또는 Prometheus 통합으로 성능 모니터링.
  • Ingress 컨트롤러: 프로덕션에서는 Nginx/Istio 앞에 두고, Ray Serve는 ClusterIP로 내부 서비스화.
  • 설정 관리: 모든 배포 설정을 serve build로 YAML 파일 생성 후 버전 관리.
