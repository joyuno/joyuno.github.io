1. Wireshark란?

Wireshark는 네트워크 패킷을 캡처하고 분석하는 데 사실상 표준(de facto standard)으로 사용되는 오픈 소스 프로토콜 분석기입니다. 네트워크 트래픽을 깊이 들여다볼 수 있는 "X-Ray 비전"을 제공한다고 볼 수 있습니다.

1. 캡처 (Capture)

네트워크 인터페이스에서 실시간으로 패킷을 수집합니다. Promiscuous 모드 지원으로 호스트에 도착하는 모든 트래픽을 캡처 가능합니다.

2. 디코딩 (Decode)

원시 이더넷 프레임을 3,000개 이상의 프로토콜로 해석하여 가독성 있는 형태로 계층별로 표시합니다. (Ethernet → IP → TCP → HTTP)

3. 분석 (Analyze)

통계, 대화(Conversation) 추적, 흐름 그래프 생성, 필터링을 통해 패턴, 이상 징후, 성능 문제를 발견합니다.

2. 왜 써야 하는가? (기존 대안과 비교)

AI/데이터 파이프라인의 네트워크 문제를 진단할 때 명령줄 도구만으로는 한계가 있습니다. Wireshark의 시각적, 상호작용적 분석이 주는 이점을 비교합니다.

도구 장점 단점 (AI/데이터 엔지니어 관점) Wireshark 차별점
tcpdump 가볍고, 서버 헤드리스 환경에 최적, 스크립팅 용이 실시간 시각화 부재, 복잡한 필터 학습 곡선, 대용량 덤프 분석 비효율 GUI를 통한 직관적 탐색, 프로토콜 디코딩 깊이, 통계 도구 풍부
tshark (CLI) Wireshark 엔진의 CLI 버전, 자동화에 적합 GUI의 시각적 연결성 부재, 일회성 분석에 비효율적 동일 엔진 기반, GUI와 CLI 혼용 가능 (캡처는 tshark, 분석은 GUI)
Cloud Logs (AWS VPC Flow 등) 관리형 서비스, 메타데이터 수집, 대규모 스케일 패킷 페이로드 내용 불가, 실시간 디버깅 어려움, 세부 프로토콜 분석 제한 Full Packet Capture, 암호화 전/후 내용 분석(가능 시), 실제 애플리케이션 데이터 확인
Python (scapy, pyshark) 완전한 프로그래밍 가능성, AI 파이프라인 통합 용이 실시간 대화형 분석 부재, 프로토콜 지원 범위 제한, 성능 이슈 (대용량) 분석 결과를 바탕으로 Python 스크립트 작성에 활용 가능 (상호보완)
핵심 결론: Wireshark는 심층 조사(Deep Dive)와 근본 원인 분석(Root Cause Analysis)에 필수적입니다. tshark로 캡처하고 Wireshark GUI로 분석하는 하이브리드 워크플로우를 추천합니다.

3. 핵심 개념: Wireshark 아키텍처

Wireshark는 모듈식 아키텍처로 구성되어 있으며, 캡처부터 표시까지의 데이터 흐름을 이해해야 효율적으로 사용할 수 있습니다.

+-----------------------+ | [네트워크 인터페이스] | | (eth0, en0, adapter) | +-----------+-----------+ | (Raw Packets) v +-----------------------+ | 캡처 엔진 (libpcap) | | (WinPcap/Npcap) | +-----------+-----------+ | (Packet Buffer) v +-----------------------+ | 디스섹터/디코더 계층 | | (Dissectors) - Protocol| | TCP, HTTP, TLS, etc. | +-----------+-----------+ | (Decoded Data) v +-----------------------+ | 패킷 분석 코어 | | (Filters, Statistics, | | Flow Tracking) | +-----------+-----------+ | (UI/CLI Output) v +-----------------------+ | 사용자 인터페이스 | | (GUI: Qt / CLI: TShark)| +-----------------------+

주요 컴포넌트 설명

  • libpcap / WinPcap / Npcap: OS별 패킷 캡처 라이브러리. Wireshark의 캡처 기반.
  • 디스섹터(Dissector): 특정 프로토콜(예: HTTP/2, Kafka, MQTT)을 해석하는 플러그인 모듈. 커스텀 디스섹터 작성 가능.
  • 필터 엔진: BPF(Berkeley Packet Filter) 구문과 Wireshark Display Filter 구문을 처리.
  • Qt GUI: 기본 사용자 인터페이스. TShark는 동일 엔진의 명령줄 버전.

4. 퀵스타트: 설치 → 설정 → 첫 예제

Step 1: 설치 (macOS 예시)

# Homebrew를 통한 설치 (가장 간단)
brew install --cask wireshark

# 설치 후 권한 설정 (패킷 캡처를 위해 필요)
sudo chmod 644 /dev/bpf*
# 또는 Wireshark를 그룹에 추가 (영구적)
sudo dseditgroup -o edit -a $(whoami) -t user access_bpf

Step 2: 첫 캡처 및 Python 연동 예제

Python으로 HTTP 요청을 생성하고, Wireshark(tshark)로 캡처하는 예제입니다.

# generate_traffic.py
import requests
import time
import threading

def make_request(url):
    try:
        resp = requests.get(url, timeout=5)
        print(f"Request to {url}: Status {resp.status_code}")
    except Exception as e:
        print(f"Error: {e}")

# 테스트 URL (공개 서비스)
urls = [
    "https://httpbin.org/get",
    "https://api.github.com",
    "https://www.google.com"
]

# 병렬 요청 생성
threads = []
for url in urls:
    t = threading.Thread(target=make_request, args=(url,))
    t.start()
    threads.append(t)
    time.sleep(0.5) # 패킷 간격 생성

for t in threads:
    t.join()
print("Traffic generation complete.")

터미널에서 tshark로 캡처 (별도 창에서 실행):

# 인터페이스 확인
tshark -D

# en0에서 HTTP/HTTPS 트래픽 캡처 (파일 저장)
tshark -i en0 -f "port 80 or port 443" -w first_capture.pcapng -c 100

# 실시간으로 HTTP 패킷만 보기
tshark -i en0 -Y "http or tls.handshake" -V

Step 3: Wireshark GUI에서 파일 열기 및 필터 적용

캡처한 first_capture.pcapng 파일을 Wireshark GUI에서 열고, 다음과 같은 Display Filter를 적용해보세요.

# 특정 IP (예: GitHub)
ip.addr == 140.82.121.3

# HTTPS 핸드셰이크 확인
tls.handshake

# HTTP 요청 중 GET만 보기
http.request.method == GET

# 특정 도메인 (SNI 필드 활용)
tls.handshake.extensions_server_name contains "github.com"

5. 핵심 설정 레퍼런스

설정 항목 경로 (GUI) 기본값 / 권장값 설명
캡처 인터페이스 Capture → Options 모든 인터페이스 Promiscuous 모드 활성화 시 호스트의 모든 트래픽 캡처. 가상 환경(도커, VM)의 인터페이스도 확인.
이름 해석 View → Name Resolution MAC, Network, Transport 해제 권장 성능 저하와 프라이버시 노출 우려. 분석 후 필요 시 개별 활성화.
프로토콜 디스섹터 Analyze → Enabled Protocols 모두 활성화 불필요한 프로토콜은 비활성화하여 성능 향상. (예: IPX, DECnet)
컬러링 규칙 View → Coloring Rules 기본 규칙 세트 에러(빨강), HTTP(녹청), DNS(파랑) 등 시각적 구분. 커스텀 규칙 생성 가능.
프로파일 Edit → Configuration Profiles Default 작업별 프로파일 생성 권장 (예: "HTTP_Analysis", "TLS_Debug"). 필터, 컬러링, 컬럼을 그룹 관리.
시간 표시 형식 View → Time Display Format Seconds Since Beginning 상대 시간이 일반적. 지연 분석 시 "Seconds Since Previous Packet" 유용.

6. 실무 팁 & 트러블슈팅

주의: 회사 네트워크에서 패킷 캡처 시 보안 정책과 법적 규정을 반드시 준수하세요. 본인 소유가 아니거나 권한 없는 트래픽 캡처는 엄격히 금지됩니다.

자주 겪는 문제 & 해결책

문제 1: 인터페이스가 보이지 않음

증상: Capture → Interfaces에 아무것도 나열되지 않음.

원인: Npcap/WinPcap 설치 오류 또는 권한 부족.

해결:

  • Windows: Npcap 재설치 (WinPcap 호환 모드 옵션 체크).
  • Linux/macOS: sudo wireshark 실행 (권장하지 않음) 또는 usermod -a -G wireshark $USER로 그룹 추가 후 재로그인.

문제 2: HTTPS 트래픽이 복호화되지 않음

증상: TLS 패킷 내용이 암호화된 채로 표시됨.

해결 (클라이언트 키 로그):

  1. 환경변수 설정: SSLKEYLOGFILE=/path/to/keylog.txt
  2. 브라우저/클라이언트 재시작.
  3. Wireshark: Edit → Preferences → Protocols → TLS → (Pre)-Master-Secret log filename에 동일 파일 지정.
* 서버 프라이빗 키를 사용하는 방법도 있으나 제한적.

문제 3: 너무 많은 패킷, 분석 집중 불가

해결: 체계적 필터링 전략

# 1. 대화(Conversation) 추적: Statistics → Conversations → TCP/UDP 탭에서 트래픽량 큰 IP 쌍 확인.
# 2. 디스플레이 필터 점진적 적용:
   ip.addr == 10.0.0.1 and tcp.port == 5432  # 특정 IP와 DB 포트
   http.request  # 모든 HTTP 요청만
   tcp.analysis.flags  # TCP 문제(재전송, 중복 ACK) 패킷만
# 3. "Follow TCP Stream" 활용: 특정 연결의 전체 대화 내용 확인.

문제 4: AI 파이프라인 내 통신 분석

시나리오: TensorFlow Serving, MLflow 서버, Kafka, Redis 간 통신 지연.

접근법:

  • gRPC: 프로토콜 디스섹터 활성화. HTTP/2 스트림으로 분석.
  • Kafka: Wireshark 기본 디스섹터로 토픽, 오프셋 확인 가능.
  • 지연 분석: I/O Graph (Statistics → I/O Graph) 또는 "tcp.time_delta" 필터로 패킷 간 간격 확인.

성능 팁

  • 캡처 필터(Capture Filter) 사용: 캡처 시점에서 노이즈 제거 (예: host 192.168.1.5 and not port 22).
  • 파일 롤오버 설정: 장기 캡처 시 "Capture → Output"에서 파일 크기/시간 기준 자동 새 파일 생성 설정.
  • 대용량 파일: 수십 GB 파일은 editcap 도구로 분할하거나, 필터 적용 후 "File → Export Specified Packets"로 추출 후 분석.
