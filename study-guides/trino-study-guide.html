1. Trino란?

Trino(구 PrestoSQL)는 오픈소스 분산 SQL 쿼리 엔진입니다. 대규모 데이터를 여러 데이터 소스(HDFS, S3, MySQL, PostgreSQL, Cassandra 등)에서 직접 쿼리할 수 있도록 설계되었습니다. 메모리 내 병렬 처리로 높은 성능을 제공하며, ETL/ELT 워크플로우나 실시간 분석에 널리 사용됩니다.

분산 쿼리 실행

클러스터 환경에서 쿼리를 여러 노드에 분산하여 병렬 처리. Coordinator와 Worker 아키텍처로 수백 노드 규모 확장 가능.

데이터 소스 연동(Federation)

하나의 SQL로 여러 이종 데이터 소스(예: S3의 Parquet + MySQL의 메타데이터)를 조인. 데이터 이동 없이 분석.

ANSI SQL 지원

표준 SQL 문법을 준수. 다양한 데이터베이스 커넥터를 통해 원본 데이터의 위치를 알 필요 없이 쿼리 가능.

2. 왜 써야 하는가?

기존 대안과 비교하여 Trino의 강점을 확인합니다.

솔루션 주요 특징 Trino와 비교
Apache Hive MapReduce 기반, 높은 레이턴시 Trino는 메모리 내 처리로 훨씬 빠름. 실시간 쿼리 가능.
Spark SQL 범용 처리 엔진, 배치 중심 Trino는 쿼리 전문 엔진으로 인터랙티브 쿼리에 더 빠름. 설정이 간단.
DB별 Native 쿼리 각 데이터소스의 전용 쿼리 언어/엔진 Trino는 Federation으로 하나의 SQL로 여러 소스 조인 가능. 데이터 이동 최소화.
상용 OLAP 엔진 고성능, 높은 비용 Trino는 오픈소스로 비용 효율적이며 커뮤니티 활발.
핵심 선택 이유: 여러 데이터 저장소에 분산된 데이터를 하나의 쿼리로 실시간 분석할 필요가 있을 때, Trino는 최적의 선택입니다.

3. 핵심 개념

아키텍처 다이어그램

+----------------+ +----------------+ | Client | | Data Sources | | (CLI, JDBC, |-------| (Hive, S3, | | Python, ...)| | MySQL, ...) | +----------------+ +----------------+ | v +----------------+ | Coordinator | | (쿼리 파싱, | | 플랜 생성, | | 스케줄링) | +----------------+ | v +----------------+ | Workers | | (N개 노드, | | 태스크 실행, | | 결과 조합) | +----------------+

주요 컴포넌트 설명

  • Coordinator: 클라이언트 요청을 받아 쿼리를 파싱, 실행 플랜 생성, Worker 노드에 태스크 스케줄링. 단일 노드로 운영.
  • Worker: 실제 데이터 처리 수행. 여러 노드로 확장 가능. 데이터 소스와 직접 통신.
  • Catalog: 데이터 소스 연결 정의 파일 (예: hive.properties). Connector, 스키마, 테이블 정보 포함.
  • Connector: 특정 데이터 소스(예: Hive, PostgreSQL)와 통신하는 플러그인.

4. 퀵스타트

1. Docker로 빠른 설치

docker run -d --name trino \
  -p 8080:8080 \
  trinodb/trino

2. 기본 Catalog 설정 (Hive 연결 예시)

컨테이너 내부 또는 일반 설치에서 /etc/trino/catalog/hive.properties 생성:

connector.name=hive-hadoop2
hive.metastore.uri=thrift://localhost:9080
hive.config.resources=/etc/hadoop/conf/core-site.xml,/etc/hadoop/conf/hdfs-site.xml

3. 첫 번째 쿼리 실행 (CLI)

# Trino CLI 다운로드 및 연결
./trino --server localhost:8080

# SQL 실행
SELECT * FROM hive.web.logs LIMIT 10;

4. Python (pytrino)으로 연결

from pytrino import connect

conn = connect(
    host="localhost",
    port=8080,
    user="user"
)
cur = conn.cursor()
cur.execute("SELECT COUNT(*) FROM hive.web.logs")
print(cur.fetchone())
주의: 실제 운영 환경에서는 Worker 노드 수, 메모리 설정(query.max-memory-per-node 등), 보안 설정을 필수로 조정해야 합니다.

5. 핵심 설정 레퍼런스

설정 키 설명 권장값 (예시)
query.max-memory 클러스터 전체 쿼리 최대 메모리 100GB
query.max-memory-per-node 노드별 쿼리 최대 메모리 10GB
query.max-total-memory-per-node 노드별 총 메모리 15GB
task.concurrency 태스크 내 스레드 수 4 (하이퍼스레딩 환경)
http-server.http.port Coordinator/Worker HTTP 포트 8080
discovery.uri Coordinator 위치 (Worker 설정) http://coordinator:8080
hive.metastore.uri Hive Metastore 주소 thrift://metastore-host:9080

설정 파일: /etc/trino/config.properties (Coordinator/Woker), /etc/trino/catalog/*.properties (Catalog별)

6. 실무 팁 & 트러블슈팅

자주 겪는 문제 + 해결책

쿼리 메모리 초과

증상: Query exceeded max memory 에러.
해결: config.properties에서 query.max-memory, query.max-memory-per-node 증가. 또는 쿼리를 더 작은 배치로 분할.

Worker 노드 죽음

증상: 쿼리 실패, Coordinator 로그에 Worker 연결 실패.
해결: Worker 노드의 메모리/디스크 상태 확인. discovery.uri 설정이 Coordinator를 올바르게 지정하는지 확인.

Hive Metastore 연결 실패

증상: Hive 테이블 조회 시 연결 에러.
해결: hive.properties의 hive.metastore.uri 확인. 네트워크 방화벽/포트 체크.

성능 저하 (특정 쿼리)

증상: 간단한 쿼리가 느림.
해결: 데이터 소스 자체의 성능 체크(예: PostgreSQL 인덱스). Trino 쿼리 실행 플랜 확인(EXPLAIN 명령어).

모니터링 필수: Trino 웹 UI(http://coordinator:8080)에서 활성 쿼리, 클러스터 상태, 큐 상태를 실시간 확인하세요.
