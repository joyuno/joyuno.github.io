1. Microsoft Sentinel란?

Microsoft Sentinel은 클라우드 네이티브 SIEM(보안 정보 및 이벤트 관리) 및 SOAR(보안 오케스트레이션, 자동화 및 대응) 솔루션입니다. Azure 환경에 구축되어 확장성 있는 데이터 수집, 고급 분석, 위협 인텔리전스, 자동화된 워크플로를 제공합니다.

📊 데이터 수집 & 정규화

Azure Monitor Agent, 커넥터, REST API를 통해 모든 소스(온프레미스, 멀티클라우드, SaaS)에서 로그를 수집하고 ASIM(Advanced Security Information Model)을 통해 정규화하여 통합 분석을 가능하게 합니다.

🤖 AI 기반 분석 & 탐지

내장 머신러닝, 사용자 행동 분석(UEBA), Fusion 커릴레이션 엔진을 활용해 알려진 및 알려지지 않은 위협을 탐지합니다. KQL(Kusto Query Language)을 통한 커스텀 탐지 규칙 작성이 핵심입니다.

⚡ 자동화된 대응 (SOAR)

플레이북(Azure Logic Apps 기반)을 통해 경고 조사, 위협 헌팅, 인시던트 대응 작업을 자동화합니다. 타사 도구와의 통합을 통해 보안 운영 효율성을 극대화합니다.

2. 왜 써야 하는가?

기존 온프레미스 SIEM 및 클라우드 경쟁사 대비 Microsoft Sentinel의 차별화된 가치를 비교합니다.

솔루션주요 특징Sentinel 대비 주요 차이점
Microsoft Sentinel클라우드 네이티브, 종량제, Azure 통합 심화, ASIM, KQL, Logic Apps 기반 플레이북-
온프레미스 SIEM (QRadar, ArcSight)하드웨어 의존, 높은 초기 비용, 유지보수 복잡확장성과 유연성이 낮음, 클라우드 데이터 수집 및 분석에 비효율적
Splunk Enterprise Security강력한 검색 엔진, 대시보드, 앱 생태계라이선스 비용이 높음(데이터 볼륨 기반), 네이티브 SOAR 기능이 상대적으로 약함
클라우드 SIEM 대안 (예: Panther, Sumo Logic)클라우드 네이티브, 규모 확장 용이Microsoft 생태계(M365, Azure AD)와의 통합 깊이가 낮음, 엔터프라이즈 지원 체계
💡 선택 기준: 이미 Azure/M365 생태계를 사용 중이라면 통합 보안 관점에서 최적의 선택입니다. KQL 숙련도가 팀에 있다면 학습 곡선이 낮고, Logic Apps를 통한 자동화 구축이 상대적으로 쉽습니다.

3. 핵심 개념

아키텍처 개요

+----------------------------------------------------------------------------------------+ | 데이터 소스 (Data Sources) | | [Azure Resources] [On-Prem (AMA/Syslog)] [M365] [AWS/GCP] [Security Appliances] | +------------|-------------------|----------------|-------------|----------------|--------+ | | | | | v v v v v +----------------------------------------------------------------------------------------+ | Microsoft Sentinel (Log Analytics Workspace) | | | | +---------------------+ +---------------------+ +-----------------------------+ | | | 데이터 수집 | | 분석 & 탐지 | | 대응 & 오케스트레이션 | | | | - 커넥터 | | - KQL 쿼리 | | - 인시던트 | | | | - ASIM 파서 | | - 분석 규칙 | | - 플레이북(Logic Apps) | | | | - 사용자 지정 로그 | | - ML/AI 탐지 | | - 워치리스트 | | | +---------------------+ +---------------------+ +-----------------------------+ | | | +------------|----------------------------------------------------------------------------+ | (저장/보관) v +----------------------------------------------------------------------------------------+ | Azure Data Explorer / Archive | | (장기 보관, 콜드 티어) | +----------------------------------------------------------------------------------------+

주요 용어

  • Log Analytics Workspace (LAW): 모든 로그 데이터가 저장되는 핵심 리포지토리. Sentinel은 LAW 위에 배포됩니다.
  • Analytic Rule: KQL로 작성된 탐지 로직. Scheduled, NRT(Near Real-Time), Microsoft Security 규칙이 있습니다.
  • Incident: 하나 이상의 경고(Alert)가 그룹화되어 생성된 보안 사건. 조사의 기본 단위.
  • Playbook: 인시던트 또는 경고에 대해 자동으로 실행되는 Logic Apps 워크플로.
  • Watchlist: IP, 해시, 사용자명 등 커스텀 지표 목록. 탐지 규칙에서 참조 가능.
  • Hunting: 사전 탐지되지 않은 위협을 사전에 찾기 위한 KQL 기반의 프로액티브 쿼리 활동.

4. 퀵스타트

4.1. 전제 조건 및 배포

Azure 구독, Contributor 이상 권한, Log Analytics Workspace가 필요합니다.

# Azure CLI를 통한 Sentinel 활성화
az account set --subscription "your-subscription-id"
az provider register --namespace 'Microsoft.SecurityInsights'
az sentinel onboarding create --resource-group "your-rg" \
                              --workspace-name "your-law-name" \
                              --name "default"

4.2. 첫 데이터 커넥터 구성 (Azure Activity Log)

# Azure Portal에서 구성하거나, ARM Template/Bicep 사용
# 데이터 커넥터는 Sentinel 리소스 내 'Data connectors'에서 'Azure Activity'를 찾아 'Open connector page' 후 Connect
# 또는 PowerShell
Connect-AzAccount
$Workspace = Get-AzOperationalInsightsWorkspace -ResourceGroupName "your-rg" -Name "your-law-name"
Set-AzSentinelDataConnector -ResourceGroupName $Workspace.ResourceGroupName `
                            -WorkspaceName $Workspace.Name `
                            -AzureActivity -Enabled

4.3. 첫 분석 규칙 만들기 (KQL 예제)

관리자 그룹에 새 사용자가 추가되는 활동 탐지

// Sentinel -> Analytics -> Create -> Scheduled query rule
SecurityEvent
| where EventID == 4728 or EventID == 4732 or EventID == 4756
| where TargetAccount == "Domain Admins" or TargetAccount == "Administrators"
| project TimeGenerated, Computer, Activity, TargetAccount, SubjectUserName, SubjectDomainName
| extend AccountCustomEntity = TargetAccount, HostCustomEntity = Computer

4.4. 간단한 플레이북 생성 (Logic Apps)

인시던트가 생성되면 Teams 채널에 알림 보내기

# 플레이북은 Azure Logic Apps 리소스로 생성됩니다.
# 트리거: When a Microsoft Sentinel incident creation rule is triggered
# 액션: Post a message in a channel (Teams 커넥터)

6. 실무 팁 & 트러블슈팅

⚠️ 문제: 로그 수집 지연 또는 누락
원인: LAW 용량 제한, 네트워크 방화벽, 에이전트 구성 오류, 커넥터 제한.
해결: 'Log Analytics workspace'의 'Usage and estimated costs'에서 수집량 확인. AMA 에이전트 상태 검사(Get-AzConnectedMachine). 네트워크 서비스 태그('AzureMonitor') 허용.
💡 팁: KQL 쿼리 성능 최적화
- 시간 범위를 가능한 좁힙니다. | where TimeGenerated > ago(1h)
- 왼쪽 테이블(큰 테이블)을 먼저 필터링합니다.
- has 연산자는 contains보다 빠릅니다.
- 조인 시 inner와 leftouter를 신중히 선택.
⚠️ 문제: 플레이북이 실행되지 않음
원인: 트리거 연결 실패, 관리 ID 권한 부족, Logic Apps 리소스 상태.
해결: Sentinel Automation 블레이드에서 플레이북 연결 확인. Logic Apps의 'ID' 설정에서 Azure 역할 할당 확인. 실행 기록에서 오류 로그 확인.
💡 팁: 비용 관리 전략
- 수집 필터링 적용: 중요 로그만 수집 (AMA DCR에서 변환 필터).
- 보존 정책 최적화: 자주 쿼리하는 데이터는 핫 티어, 그 외는 보관.
- 커스텀 로그는 JSON 형식으로 수집하여 파싱 효율화.
- Commitment Tier(예: 100GB/일)를 고려하여 단가 절감.
💡 팁: Python으로 Sentinel 데이터 조회
from azure.identity import DefaultAzureCredential
from azure.monitor.query import LogsQueryClient

credential = DefaultAzureCredential()
client = LogsQueryClient(credential)
workspace_id = "your-law-workspace-id"

query = """SecurityIncident
| where TimeGenerated > ago(7d)
| summarize IncidentCount=count() by Severity
| order by IncidentCount desc"""

response = client.query_workspace(workspace_id, query, timespan=None)
for table in response.tables:
    for row in table.rows:
        print(row)
