1. Redpanda란?

Redpanda는 Kafka API와 완벽 호환되는 C++ 기반의 고성능 스트리밍 데이터 플랫폼입니다. JVM을 사용하지 않고, 단일 바이너리로 구성되어 운영 복잡성을 크게 줄였습니다.

⚡ 성능

기존 Kafka 대비 최대 10배 낮은 지연 시간과 높은 처리량을 제공합니다. 커널 바이패스, 바벨 큐 등 최적화 기법을 활용합니다.

🔧 운영 단순화

ZooKeeper 의존성 제거, 단일 바이너리 배포, 내장 웹 UI를 통해 운영 부담을 획기적으로 줄였습니다.

🔄 완전 호환성

Kafka 클라이언트 라이브러리(Producer/Consumer), CLI 도구(kcat), Connect, Schema Registry와의 호환성을 유지합니다.

2. 왜 써야 하는가?

기존 Apache Kafka와의 주요 차이점을 비교합니다.

항목 Apache Kafka Redpanda 실무적 의미
아키텍처 JVM 기반, ZooKeeper 의존 C++ 네이티브, ZooKeeper 없음 GC 지연 없음, 구성 요소 감소
성능 높은 처리량, 비교적 높은 지연 초저지연(~ms), 높은 처리량 실시간 분석/ML에 적합
운영 튜닝 포인트 많음, 전문성 요구 자동 튜닝, 내장 모니터링 소규모 팀도 운영 가능
호환성 표준 Kafka API 완전 호환 기존 코드/도구 재사용 가능
라이선스 Apache 2.0 BSL 1.1 → Apache 2.0 상용 사용 가능 (조건부)
💡 선택 가이드: 실시간 AI/ML 피처 파이프라인, 저지연 트랜잭션 처리, 운영 인력이 부족한 팀이라면 Redpanda를 고려하세요. 매우 큰 클러스터(100+ 노드)나 특정 커뮤니티 도구가 필수라면 Kafka를 검토하세요.

3. 핵심 개념

Redpanda의 핵심 아키텍처는 Raft 컨센서스 프로토콜을 기반으로 한 단순화된 설계입니다.

┌─────────────────────────────────────────────────────────┐ │ Redpanda Cluster │ │ ┌──────────┐ ┌──────────┐ ┌──────────┐ │ │ │ Node 1 │ │ Node 2 │ │ Node 3 │ │ │ │ (Leader) │ │(Follower)│ │(Follower)│ │ │ └────┬─────┘ └────┬─────┘ └────┬─────┘ │ │ │ Raft │ Raft │ Raft │ │ ▼ ▼ ▼ │ │ ┌─────────────────────────────────────────┐ │ │ │ 내장 Raft 컨센서스 레이어 │ │ │ └─────────────────────────────────────────┘ │ │ │ │ │ │ │ ▼ ▼ ▼ │ │ ┌──────────┐ ┌──────────┐ ┌──────────┐ │ │ │ 로그 │ │ 로그 │ │ 로그 │ │ │ │ 세그먼트 │ │ 세그먼트 │ │ 세그먼트 │ │ │ └──────────┘ └──────────┘ └──────────┘ │ └─────────────────────────────────────────────────────────┘ │ │ │ └───────────────┼───────────────┘ ▼ ┌─────────────────────┐ │ Kafka API Endpoint │ │ (포트 9092, 9644) │ └─────────────────────┘ │ ┌───────────────┼───────────────┐ ▼ ▼ ▼ ┌────────────┐ ┌────────────┐ ┌────────────┐ │ Producer │ │ Consumer │ │ Admin │ │ (기존 라이브러리)│ (기존 라이브러리) │ (CLI/UI) │ └────────────┘ └────────────┘ └────────────┘

주요 용어

  • Broker: Redpanda의 단일 노드. 데이터 저장 및 서빙.
  • Topic: 데이터 스트림의 논리적 분류 (파티션으로 분할).
  • Partition: 토픽의 수평 분할 단위. 리더-팔로워 복제.
  • Raft: 내장 컨센서스 프로토콜. ZooKeeper 불필요.
  • Controller: 클러스터 메타데이터 관리 (자동 선출).

4. 퀵스타트

Docker를 이용한 로컬 개발 환경 구축부터 첫 메시지 생산/소비까지.

4.1 Docker로 실행

CLI
# Redpanda 단일 노드 클러스터 실행
docker run -d \
  --name redpanda-1 \
  -p 9092:9092 \
  -p 9644:9644 \
  docker.redpanda.com/redpandadata/redpanda:v24.2.1 \
  redpanda start \
  --overprovisioned \
  --smp 1 \
  --memory 1G \
  --reserve-memory 0M \
  --node-id 0 \
  --check=false \
  --kafka-addr PLAINTEXT://0.0.0.0:9092 \
  --advertise-kafka-addr PLAINTEXT://127.0.0.1:9092

4.2 Python 클라이언트 예제

Python: producer.py
from kafka import KafkaProducer
import json
import time

producer = KafkaProducer(
    bootstrap_servers='localhost:9092',
    value_serializer=lambda v: json.dumps(v).encode('utf-8'),
    acks='all'  # 강력한 일관성 보장
)

topic = 'user-events'
for i in range(10):
    message = {
        'user_id': i,
        'event': 'page_view',
        'timestamp': int(time.time() * 1000),
        'properties': {'page': '/home', 'source': 'web'}
    }
    # 메시지 전송 (비동기)
    future = producer.send(topic, value=message)
    # 결과 확인 (동기화)
    metadata = future.get(timeout=10)
    print(f"Sent: {message} → Partition: {metadata.partition}, Offset: {metadata.offset}")
    time.sleep(0.5)

producer.flush()
producer.close()
Python: consumer.py
from kafka import KafkaConsumer
import json

consumer = KafkaConsumer(
    'user-events',
    bootstrap_servers='localhost:9092',
    auto_offset_reset='earliest',  # 'latest' 가능
    enable_auto_commit=True,
    group_id='data-eng-group',
    value_deserializer=lambda x: json.loads(x.decode('utf-8'))
)

print("📥 메시지 수신 대기중...")
try:
    for message in consumer:
        print(f"""
Partition: {message.partition}
Offset:    {message.offset}
Key:       {message.key}
Value:     {message.value}
Timestamp: {message.timestamp}
        """)
except KeyboardInterrupt:
    print("Consumer 종료")
finally:
    consumer.close()

4.3 CLI로 토픽 관리

CLI (rpk 명령어)
# Docker 컨테이너 내에서 실행
docker exec -it redpanda-1 rpk topic create log-events --partitions 3 --replicas 1

# 토픽 리스트 확인
docker exec -it redpanda-1 rpk topic list

# 토픽 상세 정보
docker exec -it redpanda-1 rpk topic describe log-events

# 메시지 직접 생산
echo '{"log": "error occurred"}' | docker exec -i redpanda-1 rpk topic produce log-events

# 메시지 소비 (테일)
docker exec -it redpanda-1 rpk topic consume log-events -n 5

5. 핵심 설정 레퍼런스

프로덕션 환경에서 주로 튜닝하는 설정들입니다. redpanda.yaml 파일에서 수정합니다.

설정 항목 기본값 권장 값 (프로덕션) 설명
log_segment_size 1GB 2GB ~ 5GB 로그 세그먼트 파일 크기. 너무 작으면 파일 수 증가.
default_topic_replications 1 3 토픽 복제 계수 (가용성 보장).
log_retention_ms 7 days 604800000 (7일) 데이터 보존 기간 (밀리초).
enable_idempotence true true 멱등성 생산자 활성화 (중복 메시지 방지).
fetch_reads_debounce_timeout_ms 10ms 5ms (저지연) 소비자 페치 요청 대기 시간. 지연 감소 but CPU 증가.
auto_create_topics_enabled true false (프로덕션) 자동 토픽 생성 비활성화 (관리 정책).
cloud_storage_cache_size 20% of disk 50GB 클라우드 스토리지 캐시 크기 (Tiered Storage 사용 시).
⚠️ 메모리 설정 주의: reserve-memory는 Redpanda가 시스템에서 예약할 메모리 양입니다. 컨테이너 환경에서는 호스트 메모리의 일부를 명시적으로 할당하는 것이 좋습니다.

6. 실무 팁 & 트러블슈팅

자주 겪는 문제

❌ "Leader not available" 오류

원인: 토픽 생성 직후 클라이언트가 즉시 생산/소비 시도. 리더 선출이 아직 완료되지 않음.

해결: 클라이언트에 재시도 로직 추가 (exponential backoff). 또는 min.isr 설정 확인.

❌ 디스크 공간 부족

원인: 보존 정책(log_retention)이 제대로 적용되지 않거나, 세그먼트 삭제가 느림.

해결: rpk cluster config set log_cleanup_interval_ms 30000으로 청정 주기 단축. Tiered Storage(클라우드)로 오래된 데이터 오프로드.

❌ 높은 지연 시간

원인: 네트워크, 디스크 I/O 병목, 또는 너무 많은 파티션.

해결: 내장 메트릭(localhost:9644/metrics)에서 p99 대기 시간 확인. SSD 사용 권장. 불필요한 파티션 수 줄이기.

성능 튜닝 팁

  • 파티션 수 결정: CPU 코어 수의 2~3배를 시작점으로. 너무 많으면 메타데이터 오버헤드 증가.
  • 배치 크기: Producer의 batch.size와 linger.ms를 증가시켜 처리량 향상 (지연 시간 약간 증가).
  • 컴프레션: compression.type=lz4 또는 zstd로 네트워크/디스크 사용량 감소.
  • 모니터링: Prometheus + Grafana로 vectorized_io_queue_total_latency, under_replicated_replicas 지표 추적.
🚨 주의사항: Redpanda는 JVM이 없어 GC 튜닝은 불필요하지만, 여전히 메모리와 디스크 I/O 모니터링은 필수입니다. 특히 컨테이너 환경에서는 메모리 한계를 명시적으로 설정하세요.
