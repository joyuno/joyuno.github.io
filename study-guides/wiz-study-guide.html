1. Wiz란?

Wiz는 클라우드 네이티브 애플리케이션 보호 플랫폼(CNAPP)으로, 멀티클라우드 환경에서의 보안 상태를 통합적으로 관리하고 위험을 식별하는 솔루션입니다. 에이전트리스(Agentless) 방식으로 빠른 온보딩과 광범위한 가시성을 제공하는 것이 특징입니다.

🔍 통합 가시성

AWS, Azure, GCP, Kubernetes, IaC, 컨테이너 이미지, 워크로드, 네트워크, ID까지 모든 레이어의 자산과 구성을 실시간으로 탐지 및 인벤토리화합니다.

🎯 컨텍스트 기반 위험 분석

단순 규칙 위반이 아닌, 자산의 중요도, 노출 경로, 공격 가능성을 연결하여 실제 위험도를 산정하고 우선순위를 부여합니다.

🤖 AI 기반 인사이트

보안 팀의 조사 시간을 줄이기 위해 AI가 위협 인텔리전스, 이상 징후, 공격 경로를 분석하여 실행 가능한 대응 방안을 제시합니다.

2. 왜 써야 하는가?

기존 점검식 보안 도구와의 비교를 통해 Wiz의 차별점을 이해합니다.

💡 핵심 차이점

Wiz는 "어디에 중요한 문제가 있는가?"가 아닌, "어떤 문제가 실제로 공격으로 이어질 수 있는가?"에 답하기 위해 설계되었습니다. 클라우드 리소스 간의 연결 관계와 컨텍스트를 분석하여 노이즈를 줄이고, 조치가 필요한 진짜 위험에 집중할 수 있게 합니다.

3. 핵심 개념

Wiz의 데이터 수집 및 분석 아키텍처는 다음과 같습니다.

+-------------------+ +----------------------+ +-------------------+ | Cloud Accounts | | Wiz Backend | | Security Team | | (AWS, Azure, GCP)|----->| (Risk Engine, AI) |----->| (Console, API) | +-------------------+ +----------------------+ +-------------------+ | | | | Read-Only API | Correlation & | Alerts, | Permissions | Path Analysis | Dashboards, v v v Tickets +-------------------+ +----------------------+ +-------------------+ | Data Sources: | | Graph Database | | Integrations: | | - Cloud Config | | (Resource Relations)| | Slack, Jira, | | - K8s Manifests | | | | ServiceNow, etc. | | - Container Img | +----------------------+ +-------------------+ | - IaC (TF, CF) | | - Identity & Ent.| +-------------------+

주요 용어

  • Resource: Wiz가 관리하는 모든 클라우드 객체 (VM, DB, Storage, User 등).
  • Issue: 리소스에서 발견된 보안 문제 또는 위험.
  • Project: 리소스와 이슈를 논리적으로 그룹화하는 단위 (예: 프로덕션, 개발).
  • Security Graph: 리소스 간 연결 관계(네트워크, IAM)를 저장하는 그래프 DB.
  • Attack Path: 공격자가 외부에서 중요한 자산에 도달할 수 있는 가능한 경로.

4. 퀵스타트

Wiz CLI와 Python SDK를 이용해 첫 연동을 시작합니다.

단계 1: Wiz CLI 설치 및 인증

Shell
# macOS/Linux
curl -s https://install.wiz.io/wizcli.sh | bash

# Windows (PowerShell)
iwr -useb https://install.wiz.io/wizcli.ps1 | iex

# 인증 (API 토큰 필요)
wiz auth login --token YOUR_WIZ_API_TOKEN

단계 2: 첫 번째 클라우드 계정 연결 (AWS 예시)

Shell
# CloudFormation 스택 생성 명령어 출력
wiz cloud-accounts generate-cloudformation --cloud-provider AWS

# 생성된 템플릿 URL로 AWS 콘솔에서 스택 생성
# 또는 CLI로 직접 생성 (예시)
aws cloudformation create-stack \
  --stack-name WizIntegrationStack \
  --template-url https://static.wiz.io/cf/aws/wiz.json \
  --capabilities CAPABILITY_NAMED_IAM

단계 3: Python SDK로 이슈 조회하기

Python
import requests
import json

# Wiz API 엔드포인트 및 토큰
WIZ_API_URL = "https://api.us1.app.wiz.io/graphql"
API_TOKEN = "your-api-token-here"

headers = {
    "Authorization": f"Bearer {API_TOKEN}",
    "Content-Type": "application/json"
}

# GraphQL 쿼리: 상위 10개 높은 위험 이슈 가져오기
query = """
{
  issues(first: 10, filter: { severity: [HIGH, CRITICAL] }) {
    nodes {
      id
      name
      severity
      description
      resource {
        name
        cloudPlatform
      }
    }
  }
}
"""

response = requests.post(WIZ_API_URL, json={"query": query}, headers=headers)
if response.status_code == 200:
    issues = response.json()["data"]["issues"]["nodes"]
    for issue in issues:
        print(f"[{issue['severity']}] {issue['name']}")
        print(f"Resource: {issue['resource']['name']} ({issue['resource']['cloudPlatform']})")
        print("---")
else:
    print(f"Error: {response.status_code}")
    print(response.text)

⚠️ 주의사항

API 토큰은 안전하게 관리하세요. 코드에 하드코딩하지 말고 환경 변수나 시크릿 매니저를 사용하십시오. Wiz 연결용 IAM 역할은 최소 권한 원칙(Read-Only)으로 구성되어야 합니다.

6. 실무 팁 & 트러블슈팅

💡 팁 1: 프로젝트(Project)로 환경 분리

개발/스테이징/프로덕션 환경을 별도의 Wiz 프로젝트로 생성하면, 각 환경별 위험 점수와 이슈를 독립적으로 관리하고 대시보드를 구성할 수 있습니다. 프로젝트는 리소스 태그 기반으로 자동 할당 규칙을 설정할 수 있습니다.

💡 팁 2: 커스텀 폴리시 작성

Wiz의 기본 검사 규칙 외에 조직 정책을 반영한 커스텀 폴리시를 Rego(Open Policy Agent)로 작성할 수 있습니다. 예: "모든 프로덕션 S3 버킷은 반드시 'CostCenter' 태그를 가져야 함".

Rego 예시
package wiz.policies.s3

default allow = false

allow {
    input.resource.type == "aws.s3.Bucket"
    input.resource.tags["CostCenter"]
    input.resource.tags["Environment"] == "prod"
}

🚨 문제: 데이터 수집 지연

증상: 새로 생성된 리소스가 Wiz 콘솔에 나타나지 않거나, 이슈 업데이트가 느림.
해결책:

  1. 클라우드 계정의 IAM 역할/정책이 유효한지 확인 (ReadOnlyAccess).
  2. Wiz 백엔드 상태 페이지(status.wiz.io)에서 서비스 중단 확인.
  3. 대량 리소스 변경 후에는 최대 2시간의 처리 시간이 소요될 수 있음.

🚨 문제: False Positive 이슈

증상: 실제 위험이 아닌 구성이 높은 심각도 이슈로 보고됨.
해결책:

  1. 이슈 상세 페이지에서 "Risk Factors"를 확인해 컨텍스트 이해.
  2. 해당 리소스에 "Exemption" (면제)를 적용하여 알림에서 제외.
  3. 해당 검사 규칙의 심각도를 조정하거나, 커스텀 폴리시로 정교화.
