1. Snowflake란?

Snowflake는 클라우드 네이티브 데이터 플랫폼으로, 데이터 웨어하우징, 데이터 레이크, 데이터 공유, 머신러닝/애널리틱스 워크로드를 하나의 통합 서비스로 제공합니다. 전통적인 아키텍처와 달리 스토리지와 컴퓨팅이 분리되어 독립적으로 확장 가능합니다.

📊 통합 데이터 플랫폼

ETL/ELT, 저장, 처리, 분석, 공유까지의 전체 데이터 파이프라인을 단일 플랫폼에서 관리. 구조화/반구조화 데이터 모두 지원.

⚡ 스토리지-컴퓨팅 분리

데이터 스토리지와 컴퓨팅 클러스터가 독립적. 워크로드별로 컴퓨팅 리소스를 유연하게 할당하고 초 단위로 과금.

🌐 멀티 클라우드 & 리전

AWS, Azure, GCP를 동시에 지원. 크로스 클라우드 데이터 복제 및 쿼리 가능. 글로벌 데이터 거버넌스 정책 적용.

2. 왜 써야 하는가?

기존 온프레미스 및 클라우드 대안과 비교한 Snowflake의 차별점.

비교 항목 Snowflake 전통적 DW (e.g., Redshift) 빅데이터 플랫폼 (e.g., Hadoop)
관리 부담 완전 관리형 (NoOps) 부분 관리형 (클러스터 튜닝 필요) 자가 관리 (고비용)
확장성 자동 스케일 아웃/인, 초 단위 수동 클러스터 조정, 시간 단위 수평 확장 복잡
데이터 형식 JSON, Avro, Parquet, XML 등 네이티브 지원 ETL을 통한 변환 필요 모든 형식 가능 but 코드 복잡
동시성 자동 워크로드 분리, 무제한 동시 쿼리 동시성 제한, 큐잉 발생 리소스 경합 심함
가격 모델 스토리지 + 컴퓨팅 크레딧 분리 과금 인스턴스 예약 또는 온디맨드 인프라 유지비 + 인력 비용
💡 핵심 가치: 데이터 인프라 운영 복잡성을 제거하고, 데이터 팀이 비즈니스 가치 창출에 집중할 수 있게 합니다.

3. 핵심 개념

Snowflake 아키텍처는 세 가지 계층으로 구성됩니다.

┌─────────────────────────────────────────────────────────────┐ │ CLOUD SERVICES LAYER │ │ (인증, 메타데이터 관리, 쿼리 최적화, 트랜잭션 관리) │ └───────────────────────────┬─────────────────────────────────┘ │ ┌───────────────────────────┼─────────────────────────────────┐ │ QUERY PROCESSING LAYER │ │ (가상 웨어하우스: 컴퓨팅 클러스터) │ │ ┌──────┐ ┌──────┐ ┌──────┐ │ │ │ WH1 │ │ WH2 │ │ WH3 │ (워크로드별 독립적 스케일) │ │ └──────┘ └──────┘ └──────┘ │ └───────────────────────────┼─────────────────────────────────┘ │ ┌───────────────────────────┼─────────────────────────────────┐ │ DATABASE STORAGE LAYER │ │ (고도로 최적화된 칼럼형 스토리지) │ │ ┌────────────────────────────────────────────────────┐ │ │ │ AWS S3 | Azure Blob | Google Cloud Storage │ │ │ └────────────────────────────────────────────────────┘ │ └─────────────────────────────────────────────────────────────┘

주요 용어

  • 가상 웨어하우스 (Virtual Warehouse): 컴퓨팅 클러스터. 크기(T-Shirt)와 자동 일시정지 설정 가능.
  • 데이터베이스 & 스키마: 전통적인 RDBMS와 유사한 논리적 데이터 구조.
  • 스테이지 (Stage): 데이터 파일을 임시로 저장하는 위치 (내부/외부).
  • 파이프 (Pipe): 스테이지의 파일을 테이블로 지속적으로 로드하는 오브젝트.
  • 태스크 (Task): 예약된 SQL 또는 스트림 작업을 실행.
  • 스트림 (Stream): 테이블 변경 사항(CDC)을 추적하는 오브젝트.

4. 퀵스타트

4.1 설치 및 설정

Python 커넥터 설치 및 계정 연결.

# Snowflake Python 커넥터 설치
pip install snowflake-connector-python pandas

# 환경 변수 설정 (보안 권장)
export SNOWFLAKE_ACCOUNT='your_account'
export SNOWFLAKE_USER='your_user'
export SNOWFLAKE_PASSWORD='your_password'
export SNOWFLAKE_WAREHOUSE='COMPUTE_WH'
export SNOWFLAKE_DATABASE='TEST_DB'
export SNOWFLAKE_SCHEMA='PUBLIC'

4.2 첫 연결 및 쿼리 실행

import snowflake.connector
import pandas as pd

# 연결 설정
ctx = snowflake.connector.connect(
    user='your_user',
    password='your_password',
    account='your_account',
    warehouse='COMPUTE_WH',
    database='TEST_DB',
    schema='PUBLIC'
)

# 커서 생성 및 쿼리 실행
cur = ctx.cursor()
try:
    cur.execute("CREATE OR REPLACE TABLE sample (id INT, name STRING)")
    cur.execute("INSERT INTO sample VALUES (1, 'Alice'), (2, 'Bob')")
    cur.execute("SELECT * FROM sample")
    df = cur.fetch_pandas_all()
    print(df)
finally:
    cur.close()
ctx.close()

4.3 CLI (SnowSQL) 사용법

# SnowSQL 설치 후 로그인
snowsql -a your_account -u your_user

# 웨어하우스, 데이터베이스 선택
USE WAREHOUSE COMPUTE_WH;
USE DATABASE TEST_DB;

-- 파일에서 SQL 실행
!source /path/to/query.sql;

-- 쿼리 결과를 CSV로 내보내기
COPY INTO @~/output/data.csv FROM (SELECT * FROM sample) FILE_FORMAT=(TYPE=CSV);

5. 핵심 설정 레퍼런스

설정 항목 권장값 / 옵션 설명
웨어하우스 크기 X-SMALL ~ 4X-LARGE 작은 배치: X-SMALL, 대용량 ETL: LARGE 이상. AUTO_SUSPEND 설정 필수.
AUTO_SUSPEND 60 ~ 300 (초) 유휴 시간 후 웨어하우스 자동 정지. 비용 절감 핵심.
AUTO_RESUME TRUE 쿼리 요청 시 웨어하우스 자동 재개.
SCALING_POLICY STANDARD / ECONOMY STANDARD: 빠른 스케일 아웃, ECONOMY: 비용 최적화.
DATA_RETENTION_TIME 1 (일) / 0 (Fail-safe) 테이블 삭제 후 Time Travel 기간. 0은 Fail-safe만.
FILE_FORMAT (CSV) TYPE=CSV, SKIP_HEADER=1 COPY INTO 명령어 사용 시 기본 파일 형식 정의.

6. 실무 팁 & 트러블슈팅

🚀 성능 최적화

  • 큰 테이블 조인 시 웨어하우스 크기 업스케일.
  • 자주 사용되는 조인 키는 CLUSTER BY 설정.
  • 반복 쿼리는 MATERIALIZED VIEW 고려.

💸 비용 관리

  • 웨어하우스 AUTO_SUSPEND 필수 설정.
  • 리소스 모니터로 크레딧 사용량 경고 설정.
  • 큰 웨어하우스 단기 사용 후 다운스케일.

자주 겪는 문제

문제: "Out of memory" 오류
원인: 웨어하우스 크기가 쿼리 복잡도에 비해 작음.
해결: 웨어하우스 크기를 증가 (예: X-SMALL → MEDIUM). 쿼리를 작은 배치로 분할.
문제: COPY INTO 실패 - 파일 형식 불일치
원인: CSV 파일 내 구분자, 인코딩, 헤더 불일치.
해결: FILE_FORMAT 명시적 정의. 오류된 행은 VALIDATION_MODE으로 확인. 
COPY INTO my_table FROM @my_stage
FILE_FORMAT = (TYPE=CSV FIELD_DELIMITER='|' SKIP_HEADER=1 ENCODING='UTF8');
팁: Python에서 대용량 데이터 로드
PUT/COPY 명령어를 사용하여 로컬 파일을 스테이지에 업로드 후 로드하는 것이 효율적. 
# 비효율적: INSERT 한 줄씩
# 효율적: 파일 업로드 후 COPY
cur.execute("PUT file:///data/*.csv @my_stage")
cur.execute("COPY INTO my_table FROM @my_stage")
