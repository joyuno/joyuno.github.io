1. OpenTelemetry란?

OpenTelemetry(OTel)는 오픈소스 관찰 가능성(Observability) 프레임워크입니다. 애플리케이션의 트레이스(Traces), 메트릭스(Metrics), 로그(Logs)를 생성, 수집, 관리하는 도구와 API를 제공하는 표준입니다.

표준화

다양한 언어와 백엔드에 대한 통일된 API 및 데이터 모델을 제공합니다. Vendor-neutral 표준으로, 데이터 수집과 전송을 분리합니다.

자동 수집

인스트루먼테이션(Instrumentation) 라이브러리를 통해 애플리케이션 코드 변경을 최소화하면서 데이터를 자동으로 수집할 수 있습니다.

확장성

커스텀 인스트루먼테이션, 프로세서, 익스포터를 작성하여 특정 요구사항에 맞게 확장 가능합니다.

핵심: OTel은 데이터를 생성하고 내보내는(export) 도구입니다. 데이터 저장, 분석, 시각화는 백엔드(예: Prometheus, Jaeger, Datadog)의 역할입니다.

2. 왜 써야 하는가?

기존 모니터링 방식과의 비교를 통해 OTel의 장점을 확인합니다.

방식 장점 단점 OTel 차이점
각 백엔드별 Agent
(예: Datadog Agent, New Relic Agent)		 백엔드와 긴밀 통합 Agent 부하, Vendor Lock-in, 멀티 백엔드 지원 어려움 단일 Collector로 모든 백엔드 지원. Vendor Lock-in 해소.
직접 코드 로깅
(print/logging 라이브러리)		 간단한 구현 표준화 없음, 구조화 어려움, 컨텍스트 전파 복잡 구조화된 트레이스/메트릭 표준 제공. 컨텍스트 자동 전파.
많은 독립 도구
(Prometheus + Jaeger + Fluentd)		 각 도구의 전문성 설정 복잡, 데이터 모델 불일치, 통합 관리 어려움 통합 데이터 모델 (트레이스, 메트릭, 로그). 설정 통합.

3. 핵심 개념

아키텍처 다이어그램

Application (Instrumented) ──[Spans/Metrics]──> OpenTelemetry SDK ──> Collector (OTelCol) │ ├──> Exporter 1 (Jaeger) ├──> Exporter 2 (Prometheus) └──> Exporter 3 (Custom Backend) Components: 1. API: 트레이스/메트릭 생성 인터페이스 2. SDK: 인스트루먼테이션 구현, 프로세서, 익스포터 관리 3. Collector: 수집, 처리, 필터링, 내보내기 (Gateway/Agent 모드) 4. Exporter: 백엔드별 프로토콜 변환 (OTLP, Jaeger, Prometheus 등)

주요 용어

  • Span: 작업의 단위 (예: HTTP 요청, DB 쿼리). 트레이스의 구성 요소.
  • Trace: 관련된 Span들의 집합. 전체 트랜잭션 흐름.
  • Metric: 숫자 측정값 (예: 요청 수, 에러율, 지연 시간).
  • Context Propagation: 트레이스 ID 등을 요청 간 전파하여 분산 시스템 연결.
  • Instrumentation: 데이터 수집을 코드에 추가하는 방법 (자동/수동).
  • Exporter: 수집된 데이터를 백엔드로 내보내는 모듈.

4. 퀵스타트 (Python 예제)

Step 1: 설치

pip install opentelemetry-api opentelemetry-sdk
pip install opentelemetry-exporter-otlp  # OTLP exporter (백엔드 전송)
# 자동 인스트루먼테이션 필요 시 추가 (예: opentelemetry-instrumentation-flask)

Step 2: 기본 설정 (트레이스 생성)

from opentelemetry import trace
from opentelemetry.sdk.trace import TracerProvider
from opentelemetry.sdk.trace.export import BatchSpanProcessor
from opentelemetry.exporter.otlp.proto.grpc.trace_exporter import OTLPSpanExporter

# 트레이서 프로바이더 설정
trace.set_tracer_provider(TracerProvider())

# 익스포터 생성 (예: OTLP로 Jaeger에 전송)
otlp_exporter = OTLPSpanExporter(endpoint="http://jaeger-collector:4317")
span_processor = BatchSpanProcessor(otlp_exporter)
trace.get_tracer_provider().add_span_processor(span_processor)

# 트레이서 획득
tracer = trace.get_tracer(__name__)

Step 3: 첫 트레이스 생성

with tracer.start_as_current_span("main_operation") as span:
    span.set_attribute("http.method", "GET")
    span.set_attribute("http.url", "/api/data")
    # 작업 수행...
    with tracer.start_as_current_span("sub_operation") as child_span:
        child_span.set_attribute("db.query", "SELECT * FROM table")
        # 자식 작업 수행...
    span.add_event("operation_completed")
print("트레이스 생성 완료. 백엔드로 전송됨.")

Step 4: Collector 실행 (CLI)

# OpenTelemetry Collector 다운로드 및 실행 (예: Docker)
docker run -p 4317:4317 otel/opentelemetry-collector-contrib

# 기본 설정 파일 (config.yaml) 예시
receivers:
  otlp:
    protocols:
      grpc:
        endpoint: 0.0.0.0:4317
processors:
  batch:
exporters:
  jaeger:
    endpoint: jaeger-all-in-one:14250
    tls:
      insecure: true
  prometheus:
    endpoint: 0.0.0.0:8889
service:
  pipelines:
    traces:
      receivers: [otlp]
      processors: [batch]
      exporters: [jaeger]
    metrics:
      receivers: [otlp]
      processors: [batch]
      exporters: [prometheus]

5. 핵심 설정 레퍼런스

설정 항목 설명 예시 값 / 환경 변수
엔드포인트 Collector 또는 백엔드 주소 OTLP_GRPC_ENDPOINT=http://collector:4317
OTLP_HTTP_ENDPOINT=http://collector:4318
인스트루먼테이션 자동 수집 라이브러리 활성화 OTEL_PYTHON_INSTRUMENTATION_ENABLED=true
OTEL_INSTRUMENTATION_HTTP_ENABLED=true
샘플링 트레이스 샘플링 전략 OTEL_TRACES_SAMPLER=parentbased_always_on
OTEL_TRACES_SAMPLER_ARG=0.1 (확률)
배치 처리 Span/Metric 배치 크기 및 대기 시간 OTEL_BSP_SCHEDULE_DELAY=5000 (ms)
OTEL_BSP_MAX_EXPORT_BATCH_SIZE=512
서비스 이름 애플리케이션 식별자 OTEL_SERVICE_NAME=my-api-service
OTEL_RESOURCE_ATTRIBUTES=service.name=my-service
로그 레벨 OTel SDK 로그 출력 레벨 OTEL_LOG_LEVEL=DEBUG
OTEL_PYTHON_LOG_LEVEL=info

6. 실무 팁 & 트러블슈팅

팁 1: 자동 vs 수동 인스트루먼테이션
빠른 시작은 자동 인스트루먼테이션(예: opentelemetry-instrument 커맨드)으로, 정밀한 컨트롤 필요 시 수동 인스트루먼테이션을 사용하세요.
팁 2: Collector 배포 모드
Gateway 모드: 모든 서비스의 데이터를 중앙 수집.
Agent 모드: 각 노드/서비스에 배포하여 로컬 수집. 혼합 사용 가능.
문제 1: 데이터가 백엔드에 나타나지 않음
해결책: 1) Collector 로그 확인 (docker logs otel-collector). 2) Exporter 엔드포인트/프로토콜 확인. 3) SDK 로그 레벨을 DEBUG로 설정하여 내보내기 에러 확인.
문제 2: 성능 오버헤드 우려 해결책: 1) 배치 처리 사용 및 적절한 배치 크기 설정. 2) 샘플링 적용 (항상 모든 트레이스를 수집하지 않음). 3) 비동기 익스포터 사용.
문제 3: 분산 트레이스 연결 안됨
해결책: 컨텍스트 전파가 올바르게 설정되었는지 확인 (HTTP 헤더 traceparent 등). 동일한 트레이서 프로바이더 및 처리 파이프라인 사용 확인.
팁 3: 메트릭과 트레이스 연동
같은 SDK로 메트릭과 트레이스를 생성하면, 메트릭에 트레이스 컨텍스트를 첨부하여 상관관계 분석이 가능합니다.
