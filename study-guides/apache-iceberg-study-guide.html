1. Apache Iceberg란?

Apache Iceberg는 대규모 분석 데이터 세트를 관리하기 위한 오픈 테이블 포맷(Open Table Format)입니다. 데이터 레이크 상에서 트랜잭션, 스키마 진화, 버전 관리, 파티셔닝 추상화 등 데이터 웨어하우스 수준의 기능을 제공하며, Spark, Trino, Flink, Hive 등 다양한 컴퓨팅 엔진에서 일관되게 작동합니다.

🔍 정교한 메타데이터 계층

Manifest 파일을 통해 데이터 파일 목록을 효율적으로 관리하고, 스냅샷 기반 버전 관리를 지원합니다. 이로 인해 파티션 변화에 강하고, 시간 여행(Time Travel) 및 롤백이 가능합니다.

⚡ 성능 최적화

파일 수준의 통계(Min/Max, Null Count 등)를 메타데이터에 저장하여 쿼리 시 불필요한 파일 스캔을 제거합니다. 동시성 제어와 효율적인 Compaction을 지원합니다.

🔄 스키마 & 파티션 진화

기존 데이터를 재작성하지 않고도 컬럼 추가/삭제, 타입 변경, 파티션 전략 변경이 가능합니다. 하위 호환성을 보장하며 안전한 스키마 변경이 특징입니다.

2. 왜 써야 하는가? (기존 대안과 비교)

Hive 형식의 한계(파일 레벨 동시성 제어 부재, 비효율적인 파티션 발견)와 Delta Lake, Hudi 등 다른 테이블 포맷과의 차이점을 이해해야 합니다.

특징 Apache Iceberg Apache Hive (기존) Delta Lake
ACID 트랜잭션 ✅ 스냅샷 격리 ❌ 테이블 락만 가능 ✅ 쓰기 직렬화
시간 여행 ✅ 모든 스냅샷 쿼리 가능 ❌ 불가능 ✅ 제한적 (로그 보존 기간)
스키마 진화 ✅ 안전한 컬럼 추가/삭제/이름 변경 ✅ 제한적 (컬럼 추가만) ✅ 제한적 (DROP 제한)
파티션 진화 ✅ Hidden Partitioning 지원 ❌ 파티션 변경 시 데이터 재작성 필요 ❌ 변경 불가
컴퓨팅 엔진 지원 Spark, Flink, Trino, Hive, Presto 등 Hive, Spark Spark, Databricks 런타임
성능 최적화 파일 수준 통계, 동적 파티션 프루닝 디렉토리 스캔, 통계 부재 Z-Ordering, 데이터 스킵

💡 선택 가이드

Iceberg를 선택할 때: 멀티 엔진 환경(Spark, Trino, Flink 혼용), 복잡한 스키마/파티션 진화 요구사항, 데이터 레이크의 성숙도가 높은 경우.
다른 대안을 고려할 때: Databricks 단일 플랫폼(Delta Lake), 간단한 ETL 파이프라인(Hive), 실시간 스트리밍 우선(Hudi)인 경우.

3. 핵심 개념

Iceberg의 아키텍처는 메타데이터 계층을 통해 물리적 데이터 파일을 추상화합니다.

┌─────────────────────────────────────────────────────────────────────────┐ │ 쿼리 엔진 (Spark, Trino, Flink) │ └───────────────────────────────┬─────────────────────────────────────────┘ │ (Catalog API 호출) ▼ ┌─────────────────────────────────────────────────────────────────────────┐ │ Iceberg Catalog │ │ (Hive, Glue, JDBC, Nessie, REST) - 현재 스냅샷 ID, 메타데이터 위치 관리 │ └───────────────────────────────┬─────────────────────────────────────────┘ │ (메타데이터 파일 조회) ▼ ┌─────────────────────────────────────────────────────────────────────────┐ │ 메타데이터 계층 (JSON) │ │ ┌─────────────┐ ┌──────────────┐ ┌──────────────┐ │ │ │ 메타데이터 │◄───│ Manifest │◄───│ Manifest │ │ │ │ 파일 │ │ 리스트 │ │ 파일 │ │ │ │ (v1, v2) │ │ │ │ (AVRO) │ │ │ └─────────────┘ └──────────────┘ └──────┬───────┘ │ │ │ │ │ │ │ │ 현재 스냅샷 │ Manifest 파일 │ 데이터 파일 포인터 │ │ │ 포인터 │ 경로 목록 │ (경로, 통계) │ │ ▼ ▼ ▼ │ └─────────────────────────────────────────────────────────────────────────┘ │ ▼ ┌─────────────────────────────────────────────────────────────────────────┐ │ 데이터 파일 계층 (Parquet, ORC, Avro) │ │ ┌────────────┐ ┌────────────┐ ┌────────────┐ │ │ │ 데이터 │ │ 데이터 │ │ 데이터 │ │ │ │ 파일 1 │ │ 파일 2 │ │ 파일 N │ │ │ └────────────┘ └────────────┘ └────────────┘ │ └─────────────────────────────────────────────────────────────────────────┘

주요 용어

  • Snapshot: 테이블의 특정 시점 상태. 각 커밋은 새로운 스냅샷을 생성합니다.
  • Manifest File: 데이터 파일 목록과 각 파일의 통계(Min/Max, Null Count 등)를 포함하는 Avro 파일.
  • Manifest List: Manifest 파일들의 목록과 각각의 스냅샷 정보를 가진 파일.
  • Catalog: 현재 메타데이터 파일 위치를 추적하는 구성 요소 (Hive, Glue, JDBC 등).
  • Hidden Partitioning: 파티션 키를 데이터 값에서 추출(예: event_time → date_trunc('day', event_time)). 사용자는 원본 컬럼으로 쿼리.

4. 퀵스타트 (Python + Spark)

로컬 환경에서 PySpark와 Iceberg를 사용해 첫 테이블을 생성하고 조작해봅니다.

Step 1: 환경 설정 (PySpark with Iceberg)

터미널
# pip를 이용한 패키지 설치
pip install pyspark==3.5.0 pyiceberg

# 또는 Spark 세션 내에서 패키지 사용 시
export PYSPARK_SUBMIT_ARGS="--packages org.apache.iceberg:iceberg-spark-runtime-3.5_2.12:1.5.0 pyspark-shell"

Step 2: SparkSession 생성 및 카탈로그 설정

Python 코드 (iceberg_quickstart.py)
from pyspark.sql import SparkSession

spark = SparkSession.builder \
    .appName("IcebergQuickstart") \
    .config("spark.sql.extensions", "org.apache.iceberg.spark.extensions.IcebergSparkSessionExtensions") \
    .config("spark.sql.catalog.local", "org.apache.iceberg.spark.SparkCatalog") \
    .config("spark.sql.catalog.local.type", "hadoop") \
    .config("spark.sql.catalog.local.warehouse", "/tmp/iceberg-warehouse") \
    .config("spark.sql.defaultCatalog", "local") \
    .getOrCreate()

# Spark Catalog 확인
spark.sql("SHOW CURRENT NAMESPACE").show()

Step 3: 테이블 생성 및 데이터 삽입

# 테이블 생성 (Hidden Partitioning 예시)
spark.sql("""
CREATE TABLE local.db.user_events (
    user_id bigint,
    event_time timestamp,
    event_name string,
    country string
) USING iceberg
PARTITIONED BY (days(event_time), country)
""")

# 샘플 데이터 삽입
sample_data = [
    (1, "2026-03-15 10:30:00", "login", "KR"),
    (2, "2026-03-15 11:15:00", "purchase", "US"),
    (3, "2026-03-16 09:45:00", "view_item", "KR")
]
df = spark.createDataFrame(sample_data, ["user_id", "event_time", "event_name", "country"])
df.writeTo("local.db.user_events").append()

# 데이터 조회
spark.sql("SELECT * FROM local.db.user_events").show()

Step 4: 스키마 진화 및 시간 여행

# 컬럼 추가 (스키마 진화)
spark.sql("ALTER TABLE local.db.user_events ADD COLUMN device_type string")

# 새로운 데이터로 업데이트
new_data = [
    (4, "2026-03-16 14:20:00", "logout", "JP", "mobile")
]
df_new = spark.createDataFrame(new_data, ["user_id", "event_time", "event_name", "country", "device_type"])
df_new.writeTo("local.db.user_events").append()

# 시간 여행: 첫 번째 스냅샷 조회
spark.sql("SELECT * FROM local.db.user_events VERSION AS OF 1").show()

# 스냅샷 히스토리 확인
spark.sql("SELECT * FROM local.db.user_events.snapshots").show(truncate=False)

5. 핵심 설정 레퍼런스

프로덕션 환경에서 중요한 Iceberg 관련 설정들입니다.

카테고리 설정 키 기본값/예시 설명
카탈로그 spark.sql.catalog.<catalog_name>.type hadoop, hive, rest, glue 사용할 카탈로그 구현체 지정
spark.sql.catalog.<catalog_name>.warehouse s3://bucket/warehouse/ 데이터와 메타데이터가 저장될 루트 경로
spark.sql.catalog.<catalog_name>.uri thrift://hive-metastore:9083 Hive Metastore URI (type=hive일 때)
작성 최적화 write.target-file-size-bytes 536870912 (512MB) 파일 당 목표 크기. Compaction에 영향
write.format.default parquet 기본 파일 포맷 (parquet, orc, avro)
write.metadata.delete-after-commit.enabled true 커밋 후 이전 메타데이터 파일 삭제 여부
읽기 최적화 read.split.open-cost 4194304 (4MB) 파일을 열기 위한 예상 비용. 스플릿 계획에 사용
read.split.target-size 134217728 (128MB) 목표 스플릿 크기
engine.hive.enabled false Hive 엔진 호환 모드 (레거시 지원)
유지 관리 commit.retry.num-retries 4 커밋 실패 시 재시도 횟수
commit.retry.min-wait-ms 100 재시도 간 최소 대기 시간(ms)

6. 실무 팁 & 트러블슈팅

⚠️ 문제: 작은 파일 문제 (Small Files)

증상: 수많은 작은 Parquet 파일로 인해 쿼리 성능 저하, 메타데이터 부풀림.
해결: 정기적인 Compaction(rewrite_data_files) 수행. 스트리밍 작업 시 write.target-file-size-bytes 설정 확인.

-- Spark SQL로 Compaction 실행
CALL local.system.rewrite_data_files(
  table => 'db.user_events',
  strategy => 'binpack'
)

💡 팁: Hidden Partitioning 활용

파티션을 물리적 경로가 아닌 컬럼 값에서 추출하도록 정의하여, 파티션 전략 변경 시 데이터 재작성 없이 가능합니다.

-- 좋은 예: Hidden Partitioning
CREATE TABLE events (
    ts timestamp,
    user_id bigint,
    data string
) PARTITIONED BY (hours(ts), bucket(16, user_id))

-- 나쁜 예: 직접 컬럼 파티셔닝 (파티션 변경 시 데이터 이동 필요)
CREATE TABLE events (
    ts timestamp,
    user_id bigint,
    data string
) PARTITIONED BY (date(ts), user_id % 16)

⚠️ 문제: 동시 쓰기 충돌

증상: 여러 작업이 동시에 같은 테이블에 쓰기 시도 시 충돌 발생.
해결: Optimistic Concurrency Control(OCC)을 이해. 충돌 시 재시도 로직 구현 또는 작업 스케줄링 조정.

# Python에서 재시도 로직 예시
from tenacity import retry, stop_after_attempt, wait_exponential
@retry(stop=stop_after_attempt(3), wait=wait_exponential(multiplier=1, min=4, max=10))
def write_with_retry(df, table_path):
    df.writeTo(table_path).append()

💡 팁: 효율적인 Expire Snapshots & Remove Orphans

오래된 스냅샷과 더 이상 참조되지 않는 데이터 파일을 정리하여 스토리지 비용 절감.

-- 30일 이전 스냅샷 정리
CALL local.system.expire_snapshots(
  table => 'db.user_events',
  older_than => TIMESTAMP '2026-02-01 00:00:00',
  retain_last => 10 -- 최소 10개 스냅샷은 유지
)

-- 고아 파일 정리 (카탈로그에 없는 데이터 파일 삭제)
CALL local.system.remove_orphan_files(
  table => 'db.user_events',
  older_than => TIMESTAMP '2026-02-01 00:00:00'
)
