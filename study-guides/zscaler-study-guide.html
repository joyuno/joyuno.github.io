1. Zscaler란?

Zscaler는 클라우드 기반의 보안 및 네트워크 플랫폼으로, 기존의 하드웨어 중심 퍼실리티(방화벽, 프록시)를 클라우드 서비스로 대체합니다. 사용자/디바이스가 어디에 있든 가장 가까운 Zscaler 클라우드로 트래픽을 라우팅하여 보안 정책을 적용하는 Security Service Edge(SSE) 및 Zero Trust Network Access(ZTNA)의 선도적 제공자입니다.

Zero Trust Exchange

전 세계 150개 이상의 데이터센터에 구축된 플랫폼. 모든 트래픽은 이 플랫폼을 통해 검사 및 라우팅되며, 사용자/애플리케이션/디바이스에 대한 신뢰를 지속적으로 검증합니다.

AI 기반 위협 방어

대규모 트래픽 데이터를 기반으로 한 머신러닝 모델을 활용해 제로데이 공격, 피싱, 악성코드를 실시간으로 탐지 및 차단합니다. 샌드박스, DNS 보안, CASB 기능을 통합합니다.

애플리케이션 중심 접근

네트워크 세그먼트가 아닌 개별 애플리케이션에 대한 접근을 제어(ZTNA). 사용자에게는 필요한 앱만 보이며, 네트워크 전체에 대한 가시성은 차단됩니다.

2. 왜 써야 하는가? (기존 대안과 비교)

하이브리드/멀티클라우드 환경과 원격 근무 확대로 인한 전통적 네트워크 보안(MPLS, 온프레미스 방화벽)의 한계를 해결합니다.

비교 요소 전통적 모델 (온프레미스 방화벽/VPN) Zscaler 클라우드 플랫폼
아키텍처 허브-앤-스포크, 트래픽 허브로 역류 분산형 클라우드 게이트웨이, 직접 인터넷 브레이크아웃
확장성 하드웨어 용량 한계, 업그레이드 주기 필요 클라우드 네이티브, 탄력적 자동 확장
성능 지연 증가, 병목 현상 발생 가능 최단 경로 라우팅, 글로벌 가속
보안 모델 네트워크 중심 (신뢰 구간 내부/외부) 제로 트러스트 (사용자/디바이스/앱 신뢰 지속 검증)
관리 복잡도 다중 장비 정책 관리, 고가의 전문 인력 필요 중앙 집중식 콘솔, 정책 일관성, API 기반 자동화
AI/데이터 활용 제한적, 로컬 데이터만 분석 글로벌 위협 인텔리전스, 집단적 보안 학습 가능
핵심 가치: 복잡성 감소, 지연 시간 단축, 제로 트러스트 구현, AI 기반 위협 방어의 글로벌 집단 학습 효과.

3. 핵심 개념 & 아키텍처

Zscaler의 동작은 사용자/디바이스에 설치된 Zscaler Client Connector (ZCC) 또는 네트워크 장비의 PAC 파일/라우팅을 통해 시작됩니다.

+-------------------+ Secure Tunnel +---------------------------+ | User/Device | -----------------------> | Zscaler Cloud (POP) | | (ZCC or PAC) | | | +-------------------+ | +---------------------+ | | | | Identity Engine | | | | | (User/Device Auth) | | Internet Traffic | +---------------------+ | | | | | v | v | +-------------------+ | +---------------------+ | | Local Breakout | | | Policy Engine | | | (Direct to Cloud)| | | (App/URL/Threat) | | +-------------------+ | +---------------------+ | | | | | +--------------------------------------+ v | +---------------------+ | Security Stack | | - FWaaS | | - SWG | | - CASB | | - Sandbox | | - DNS Security | +---------------------+ | v +---------------------+ | Destination | | (Internet / SaaS / | | Private App) | +---------------------+

주요 컴포넌트

  • ZIA (Zscaler Internet Access): 인터넷 및 SaaS 앱으로의 모든 아웃바운드 트래픽 보안.
  • ZPA (Zscaler Private Access): 프라이빗 애플리케이션에 대한 제로 트러스트 접근(ZTNA).
  • ZCC (Zscaler Client Connector): 엔드포인트에 설치되어 트래픽을 클라우드로 안전하게 터널링.
  • Zscaler Central Authority: 전 세계 POP를 관리하고 정책/구성/로그를 중앙에서 처리하는 두뇌.

4. 퀵스타트: API 연동 및 정책 테스트

Zscaler 관리 포털과 API를 사용한 기본적인 자동화 및 정책 테스트 흐름입니다.

  1. 필수 준비: API 키 생성 및 환경 설정

    Zscaler 관리 포털(admin.zscaler.com)에서 API 키를 생성합니다.

    # Python 가상환경 설정 및 라이브러리 설치
python -m venv zscaler-env
source zscaler-env/bin/activate  # Windows: zscaler-env\Scripts\activate
pip install requests
    # 환경 변수 설정 (실무에서는 비밀 관리 도구 사용)
export ZSCALER_BASE_URL="https://zsapi.zscaler.net/api/v1"
export ZSCALER_API_KEY="your_api_key_here"
export ZSCALER_USERNAME="admin_username"
export ZSCALER_PASSWORD="admin_password"

  2. 인증 및 세션 획득 (Python 예제)

    import requests
import os
import json

base_url = os.getenv('ZSCALER_BASE_URL')
api_key = os.getenv('ZSCALER_API_KEY')
username = os.getenv('ZSCALER_USERNAME')
password = os.getenv('ZSCALER_PASSWORD')

# 1. API 키를 사용한 인증 및 JSESSIONID 획득
auth_url = f"{base_url}/authenticatedSession"
headers = {
    'Content-Type': 'application/json',
    'Cache-Control': 'no-cache'
}
payload = {
    'apiKey': api_key,
    'username': username,
    'password': password
}

response = requests.post(auth_url, headers=headers, json=payload)
if response.status_code == 200:
    jsessionid = response.cookies.get('JSESSIONID')
    print(f"[+] 인증 성공. JSESSIONID: {jsessionid[:15]}...")
    # 이후 요청에 쿠키로 사용
    session_cookies = {'JSESSIONID': jsessionid}
else:
    print(f"[-] 인증 실패: {response.status_code}")
    print(response.text)
    exit(1)

  3. URL 카테고리 조회 및 정책 생성 예제

    # 2. URL 카테고리 목록 조회
categories_url = f"{base_url}/urlCategories"
cat_response = requests.get(categories_url, cookies=session_cookies)
if cat_response.status_code == 200:
    categories = cat_response.json()
    print(f"[+] 총 {len(categories)}개의 URL 카테고리 로드됨")
    # 'Social Networking' 카테고리 ID 찾기
    social_cat = next((cat for cat in categories if cat['configuredName'] == 'SOCIAL_NETWORKING'), None)
    if social_cat:
        print(f"    - SOCIAL_NETWORKING ID: {social_cat['id']}")
else:
    print(f"[-] 카테고리 조회 실패: {cat_response.status_code}")

# 3. 샘플 URL 필터링 정책 생성 (CLI 대안)
print("\n# Zscaler CLI 명령어 예시 (공식 Zscaler CLI 도구 사용)")
print("zscaledr url-policies create \\")
print('  --name "Block Social Media during Work" \\')
print('  --action "BLOCK" \\')
print('  --url-categories "SOCIAL_NETWORKING" \\')
print('  --user-groups "All_Employees" \\')
print('  --time-window "Work_Hours"')

  4. 세션 종료

    # 4. 인증 세션 종료 (중요)
logout_url = f"{base_url}/authenticatedSession"
logout_response = requests.delete(logout_url, cookies=session_cookies)
if logout_response.status_code == 200:
    print("[+] API 세션 종료 완료")
else:
    print(f"[-] 세션 종료 실패: {logout_response.status_code}")

5. 핵심 설정 레퍼런스 (표)

설정 영역 키 파라미터 설명 및 권장값 관련 API 엔드포인트
인증 (Admin) apiKey, username, password, JSESSIONID API 키는 포털에서 생성, 세션 타임아웃 기본 30분. /authenticatedSession
URL 필터링 urlCategories, action (BLOCK/ALLOW), order 카테고리 기반 정책이 성능 최적. 사용자 정의 카테고리 생성 가능. /urlPolicies, /urlCategories
파일 샌드박스 fileTypes, action (BLOCK/ISOLATE), malwareDb 실행 파일(.exe, .msi), 문서 형식 필수 검사 권장. 클라우드 샌드박스 활성화. /sandboxReport
ZTNA (ZPA) 앱 세그먼트 appSegment, segmentGroup, connectorGroup 애플리케이션을 논리적 세그먼트로 그룹화. TCP/UDP 포트 및 도메인 정의. /appSegments (ZPA API)
AI/ML 위협 방어 threatDb, aiScan, anomalyDetection AI 기반 스캔 기본 활성화. 위협 DB는 자동 업데이트. /security (고급 설정)
로그 & 모니터링 logType, duration, format (JSON/CSV) 트래픽, 위협, DNS 로그 수집. SIEM 연동(Splunk, QRadar) 권장. /logs (다양한 로그 타입)

6. 실무 팁 & 트러블슈팅

팁 1: 정책 테스트는 "감사 모드"로 시작하라.

새로운 URL/파일 필터링 정책을 적용할 때는 먼저 action을 BLOCK이 아닌 MONITOR 또는 ALLOW (로그만)로 설정하여 예상치 못한 비즈니스 영향도를 확인하세요.

문제 1: ZCC 연결 실패 또는 지연

증상: 클라이언트 커넥터가 "Connecting..." 상태에서 멈추거나 자주 끊김.
해결책:
1. 클라이언트 로그 확인: %ProgramData%\Zscaler\logs (Windows) 또는 /var/log/zscaler (macOS/Linux).
2. 네트워크 방화벽에서 Zscaler POP IP 범위 및 포트(80, 443, 4433) 허용 확인.
3. PAC 파일/자동 구성 스크립트가 최신인지 확인.

팁 2: API 호출 제한 및 배치 처리

Zscaler API는 분당/일별 호출 제한이 있습니다. 대량의 정책이나 객체를 관리할 때는 배치(Batch) API를 사용하거나, 적절한 지연 시간을 두고 호출하세요. 429(Too Many Requests) 응답을 처리하는 재시도 로직을 구현하는 것이 좋습니다.

문제 2: 특정 SaaS 앱 성능 저하

증상: Salesforce, Office 365 등 특정 앱만 느리게 작동.
해결책:
1. Zscaler 관리 포털에서 해당 앱의 트래픽이 어떤 정책과 일치하는지 로그로 확인.
2. 해당 SaaS 앱의 IP/URL을 예외 목록에 추가하지 말고, “SaaS 애플리케이션” 정책 템플릿을 사용해 최적화된 경로를 적용.
3. 클라우드에서 직접 브레이크아웃(Cloud DIA)이 제대로 구성되었는지 확인.

팁 3: 인프라 as Code (IaC) 적용

Zscaler 구성을 Terraform Provider나 Ansible Collection을 통해 코드로 관리하세요. 이를 통해 정책의 버전 관리, 변경 추적, 일관된 다중 환경 배포가 가능해집니다.

# Terraform 예시 (공식 Provider 사용)
resource "zscaler_url_category" "custom_ai_tools" {
  configured_name = "CUSTOM_AI_TOOLS"
  urls            = ["api.openai.com", "api.anthropic.com", "*.github.dev"]
}

resource "zscaler_url_policy" "allow_ai_research" {
  name          = "Allow AI Research Tools"
  order         = 100
  action        = "ALLOW"
  url_categories = [zscaler_url_category.custom_ai_tools.id]
  user_groups   = [data.zscaler_user_groups.researchers.id]
}
