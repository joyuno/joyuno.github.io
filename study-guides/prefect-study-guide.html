1. Prefect란?

Prefect는 현대적인 데이터 워크플로우 오케스트레이션 플랫폼입니다. Python 네이티브로 설계되어 개발자 친화적이며, 복잡한 데이터 파이프라인의 스케줄링, 실행, 모니터링을 단순화합니다.

핵심 철학: "Workflows as Code"와 "Positive Engineering"으로, 실패를 예상하고 처리하는 대신 성공적인 실행을 위한 시스템을 구축합니다.

🚀 동적 워크플로우

런타임에 조건부 분기, 반복, 매개변수화를 지원하는 동적 DAG 생성. Airflow의 정적 DAG 한계를 극복합니다.

🔧 개발자 경험 최적화

순수 Python 데코레이터(@flow, @task) 사용. 로컬 테스트가 간편하며, 디버깅이 직관적입니다.

☁️ 하이브리드 아키텍처

완전 관리형 클라우드(Prefect Cloud)와 자체 호스팅(Prefect Server/UI) 모두 지원. 에이전트 기반의 유연한 실행 모델.

2. 왜 써야 하는가? (기존 대안과 비교)

Airflow, Luigi 등 기존 오케스트레이션 도구의 한계를 해결하기 위해 등장했습니다.

기능 Prefect Apache Airflow Luigi
DAG 정의 동적 생성 가능 (런타임) 정적 (컴파일 타임) 정적
개발 편의성 데코레이터 기반, 로컬 테스트 쉬움 DAG 파일 작성, 운영자 학습 필요 클래스 상속, 보일러플레이트多
파라미터화 네이티브 지원 (Flow 실행 시) 제한적 (템플릿 변수) 제한적
에러 처리 자동 재시도, 지연, 알림 기본 제공 수동 구성 필요 기본 기능 부족
실행 환경 하이브리드 (에이전트 기반) 주로 중앙 스케줄러 중앙 스케줄러
상태 관리 명시적 상태 전이 (성공, 실패, 지연 등) 성공/실패 이진 상태 성공/실패

💡 선택 가이드

Prefect를 선택할 때: 동적 워크플로우가 필요하거나, 빠른 프로토타이핑과 우수한 개발자 경험을 중시할 때, 하이브리드/멀티 클라우드 환경을 고려할 때.

Airflow를 고려할 때: 대규모 팀과 성숙한 CI/CD가 이미 구축되어 있고, 정적 DAG으로 충분하며, 커뮤니티와 레퍼런스가 많은 환경이 필요할 때.

3. 핵심 개념 & 아키텍처

기본 구성 요소

  • Flow: 워크플로우의 컨테이너. @flow 데코레이터로 정의.
  • Task: Flow 내의 개별 작업 단위. @task 데코레이터로 정의.
  • Deployment: Flow를 스케줄, 트리거, 모니터링할 수 있도록 패키징.
  • Work Pool: 작업을 실행할 인프라 그룹 (K8s, Docker, EC2 등).
  • Agent: Work Pool에서 작업을 가져와 실행하는 프로세스.
  • Orion Server: 오케스트레이션 엔진 (API 서버).
  • Prefect UI: 워크플로우 모니터링 대시보드.

아키텍처 다이어그램 (하이브리드 모드)

+---------------------------------------------------+ | 개발자 환경 (Local/CI) | | +----------------+ +-------------------+ | | | Python 코드 | -> | prefect deployment| | | | @flow, @task | | create | | | +----------------+ +-------------------+ | +---------------------------------------------------+ | (메타데이터 등록) +---------------------------------------------------+ | 오케스트레이션 레이어 | | (Prefect Cloud / Self-hosted Server) | | +----------------+ +-------------------+ | | | Orion API | | UI Dashboard | | | | (스케줄링, | | (모니터링, 실행) | | | | 상태 관리) | | | | | +----------------+ +-------------------+ | +---------------------------------------------------+ | (작업 풀링) +---------------------------------------------------+ | 실행 인프라 (에이전트) | | +----------------+ +-------------------+ | | | K8s Agent | | Docker Agent | | | | | | | | | +----------------+ +-------------------+ | | +----------------+ +-------------------+ | | | EC2 Agent | | Local Subprocess | | | | | | | | | +----------------+ +-------------------+ | +---------------------------------------------------+ | (실제 실행) +---------------------------------------------------+ | 실행 환경 (Worker) | | +-------------------------------------------+ | | | Docker 컨테이너 / K8s Pod / EC2 인스턴스 | | | | 내에서 Flow Task 실행 | | | +-------------------------------------------+ | +---------------------------------------------------+

📌 개념 정리

에이전트(Agent)는 오케스트레이션 레이어(서버)와 실행 인프라 사이의 '연결고리'입니다. 특정 Work Pool을 바라보며, 해당 풀에 할당된 작업을 가져와 지정된 인프라(예: Docker 컨테이너)에서 실행합니다. 하나의 인프라 환경(예: 프로덕션 K8s 클러스터)에 하나의 에이전트를 띄워놓고, 여러 Work Pool을 관리할 수 있습니다.

4. 퀵스타트: 5분 안에 첫 Flow 실행하기

단계 1: 설치

터미널

pip install -U prefect

# 추가 패키지 (선택)

단계 2: 첫 번째 Flow 작성

first_flow.py
from prefect import flow, task
from datetime import datetime
import httpx

@task(retries=2, retry_delay_seconds=5)
def fetch_weather(lat: float, lon: float):
    """Open-Meteo API에서 날씨 데이터 가져오기"""
    base_url = "https://api.open-meteo.com/v1/forecast"
    resp = httpx.get(
        base_url,
        params=dict(latitude=lat, longitude=lon, hourly="temperature_2m"),
    )
    resp.raise_for_status()
    return resp.json()

@task
def save_result(data: dict, path: str = "./weather.json"):
    """결과를 JSON 파일로 저장"""
    import json
    with open(path, "w") as f:
        json.dump(data, f, indent=2)
    return path

@flow(name="weather-pipeline", log_prints=True)
def weather_pipeline(lat: float = 37.56, lon: float = 126.97):
    """서울 날씨 데이터 수집 파이프라인"""
    print(f"날씨 데이터 수집 시작: {datetime.now()}")
    
    # Task 실행
    weather_data = fetch_weather(lat, lon)
    
    # 데이터 추출 예시
    current_temp = weather_data["hourly"]["temperature_2m"][0]
    print(f"현재 기온: {current_temp}°C")
    
    # 결과 저장
    saved_path = save_result(weather_data)
    print(f"데이터 저장 완료: {saved_path}")
    return current_temp

if __name__ == "__main__":
    # 로컬 실행
    weather_pipeline()

단계 4: Deployment 생성 및 스케줄 실행

터미널
# Deployment 생성 (스케줄, 트리거 설정)
prefect deployment create first_flow.py

# 또는 Python 코드 내에서 생성
# from prefect.deployments import Deployment

5. 핵심 설정 레퍼런스

설정 항목 기본값 / 예시 설명 위치
PREFECT_API_URL http://localhost:4200/api Orion 서버 API 엔드포인트 환경 변수 / prefect config set
PREFECT_API_KEY your_api_key_here (Cloud 사용 시) Prefect Cloud 인증 키 환경 변수
Task 재시도 @task(retries=3, retry_delay_seconds=10) 실패 시 자동 재시도 횟수 및 지연 시간 Task 데코레이터
타임아웃 @task(timeout_seconds=300) Task 실행 최대 시간 (초) Task 데코레이터
캐싱 @task(cache_key_fn=task_input_hash, cache_expiration=3600) 입력값 기반 캐싱 (중복 실행 방지) Task 데코레이터
Concurrency Limit prefect work-pool set-concurrency-limit "pool-name" 5 동시 실행 작업 수 제한 CLI / UI
로그 레벨 PREFECT_LOGGING_LEVEL=INFO 로그 상세도 (DEBUG, INFO, WARNING, ERROR) 환경 변수
결과 저장 PREFECT_LOCAL_STORAGE_PATH=~/.prefect/results Task 결과 직렬화 저장 경로 환경 변수

6. 실무 팁 & 트러블슈팅

⚠️ 자주 겪는 문제 1: 에이전트가 작업을 가져오지 않음

증상: UI에서 Flow를 실행했지만 "Scheduled" 상태에서 멈춤. 에이전트 로그에 아무 메시지 없음.

해결:

  1. 에이전트가 올바른 Work Pool을 바라보는지 확인: prefect agent start <work-pool-name>
  2. 에이전트와 서버(API)의 연결 확인: prefect config view | grep PREFECT_API_URL
  3. Work Pool의 타입과 에이전트 타입이 일치하는지 확인 (예: process pool에는 process agent).

💡 팁 1: 로컬 개발 시에는 서버 모드보다 Ephemeral API 활용

# 별도 서버 실행 없이 임시 API로 빠르게 테스트
prefect dev start

# 또는 코드에서 직접 ephemeral API 사용
if __name__ == "__main__":
    weather_pipeline.serve(name="dev-deployment", interval=600)

⚠️ 자주 겪는 문제 2: Task 결과가 너무 커서 직렬화 실패

증상: "Result serialization failed" 또는 메모리 부족 오류.

해결:

  1. 대용량 데이터는 Task 간 전달하지 말고 외부 저장소(S3, GCS) 경로만 전달.
  2. @task(persist_result=False)로 결과 직렬화 비활성화.
  3. 커스텀 결과 직렬화기 구현 또는 prefect.filesystems 사용.

💡 팁 2: Docker 실행 시 의존성 관리

# prefect.yaml 예시 (Docker 빌드 명세)
build:
- prefect_docker.deployments.steps.build_docker_image:
    id: build-image
    image_name: my-repo/my-flow
    dockerfile: auto
    requires: prefect-docker

# 또는 커스텀 Dockerfile 지정
deployments:
- name: docker-deploy
  entrypoint: "flows/my_flow.py:weather_pipeline"
  work_pool:
    name: "docker-pool"
    work_queue_name: "prod"
    job_variables:
      image: "{{ build-image.image }}"

💡 팁 3: 슬랙 알림 통합

# blocks 사용 (UI에서도 설정 가능)
from prefect.blocks.notifications import SlackWebhook
from prefect import get_run_logger

slack_block = SlackWebhook.load("prod-alerts")

@flow
def my_flow():
    try:
        # 작업 실행
        pass
    except Exception as e:
        logger = get_run_logger()
        logger.error("Flow 실패: %s", e)
        slack_block.notify(f"🚨 Flow 실행 실패: {e}")
        raise
