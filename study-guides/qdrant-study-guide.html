1. Qdrant란?

Qdrant는 Rust로 작성된 오픈소스 벡터 데이터베이스 & 벡터 검색 엔진입니다. 고차원 벡터 데이터를 저장하고 유사성 검색을 효율적으로 수행하도록 설계되었으며, 단순한 벡터 검색을 넘어 필터링, 페이로드 관리 등 프로덕션급 기능을 제공합니다.

⚡ 고성능 검색

Rust 기반의 메모리 안전성과 C++ 수준의 성능을 결합. HNSW, IVF 등 최신 ANN 알고리즘을 지원하며, SIMD 최적화로 초고속 검색 제공.

🔧 프로덕션 기능

단순 검색 엔진이 아닌 완전한 DBMS. 지속성, 샤딩, 복제, 권한 관리, REST/gRPC API, 다양한 클라이언트 SDK를 기본 지원.

📦 유연한 배포

단일 바이너리, Docker, Kubernetes, 클라우드 서비스 등 다양한 환경에 배포 가능. 메모리 내, 디스크 기반, 하이브리드 스토리지 모드 선택.

2. 왜 써야 하는가?

기존 벡터 검색 솔루션 대비 Qdrant의 차별점은 프로덕션 환경을 고려한 설계에 있습니다.

핵심 선택 기준

Qdrant 선택이 유리한 경우: 오픈소스 기반의 자체 호스팅이 필요할 때, Rust의 성능/안정성을 요구할 때, 복잡한 필터링이 포함된 벡터 검색이 필요할 때, gRPC를 통한 저지연 통신이 중요할 때.

3. 핵심 개념

아키텍처 개요

+-----------------------------------------------------------------------+ | Client Applications | | (Python, Go, JS, Java, Rust, HTTP, gRPC) | +-----------------------------+-----------------------------------------+ | +-----------v-----------+ +------------------------+ | REST API | | gRPC API | | (포트: 6333) | | (포트: 6334) | +-----------+-----------+ +------------+-----------+ | | +-----------v----------------------------v-----------+ | Qdrant 서버 (단일/클러스터) | | +---------------------+ +---------------------+ | | | 스토리지 엔진 | | 검색 엔진 | | | | • 세그먼트 관리 | | • HNSW/IVF | | | | • WAL (Write-Ahead| | • 거리 계산 | | | | Log) | | • 필터링 | | | +---------------------+ +---------------------+ | +-----------------------------------------------------+ | +---------------------------------------+ | 스토리지 백엔드 | | • 메모리 (Memmap) | | • 디스크 (RocksDB) | | • S3 호환 객체 스토리지 (원격) | +---------------------------------------+

주요 용어

  • 컬렉션(Collection): 벡터와 페이로드를 저장하는 논리적 컨테이너 (RDBMS의 테이블과 유사)
  • 포인트(Point): 벡터, 페이로드, 고유 ID로 구성된 단일 데이터 항목
  • 페이로드(Payload): 벡터와 함께 저장되는 메타데이터 (JSON 형식)
  • 세그먼트(Segment): 컬렉션 내의 물리적 데이터 분할 단위
  • HNSW: Hierarchical Navigable Small World, 근사 최근접 이웃 검색 알고리즘

4. 퀵스타트

Step 1: 설치 (Docker)

# 최신 버전 실행
docker run -p 6333:6333 -p 6334:6334 \
    -v $(pwd)/qdrant_storage:/qdrant/storage:z \
    qdrant/qdrant

# 특정 버전 지정 (프로덕션 권장)
docker run -p 6333:6333 -p 6334:6334 \
    -v $(pwd)/qdrant_storage:/qdrant/storage:z \
    qdrant/qdrant:v1.10.4

Step 2: Python 클라이언트 설치 및 연결

pip install qdrant-client
from qdrant_client import QdrantClient
from qdrant_client.models import VectorParams, Distance

# 로컬 Qdrant 서버에 연결
client = QdrantClient(host="localhost", port=6333)

# 또는 클라우드 Qdrant Cloud 사용 시
# client = QdrantClient(
#     url="https://xxxxxx-xxxxx-xxxxx.cloud.qdrant.io",
#     api_key="your-api-key",
# )

Step 3: 첫 번째 컬렉션 생성 및 데이터 삽입

# 컬렉션 생성 (128차원 벡터, 코사인 유사도)
client.create_collection(
    collection_name="my_collection",
    vectors_config=VectorParams(size=128, distance=Distance.COSINE),
)

# 벡터 데이터 준비 (예: 5개의 128차원 벡터)
import numpy as np
vectors = np.random.rand(5, 128).tolist()
ids = [1, 2, 3, 4, 5]
payloads = [
    {"category": "document", "source": "file1.pdf"},
    {"category": "image", "source": "photo.jpg"},
    {"category": "document", "source": "report.pdf"},
    {"category": "audio", "source": "podcast.mp3"},
    {"category": "document", "source": "contract.pdf"},
]

# 데이터 업서트 (업데이트 또는 삽입)
client.upsert(
    collection_name="my_collection",
    points=[
        {"id": idx, "vector": vec, "payload": payload}
        for idx, vec, payload in zip(ids, vectors, payloads)
    ],
)

Step 4: 검색 및 필터링

# 쿼리 벡터 생성 (동일 차원)
query_vector = np.random.rand(128).tolist()

# 기본 검색 (상위 3개 결과)
search_result = client.search(
    collection_name="my_collection",
    query_vector=query_vector,
    limit=3,
)

# 필터링이 포함된 검색 (카테고리가 'document'인 항목만)
from qdrant_client.models import Filter, FieldCondition, MatchValue

filtered_result = client.search(
    collection_name="my_collection",
    query_vector=query_vector,
    query_filter=Filter(
        must=[
            FieldCondition(
                key="category",
                match=MatchValue(value="document"),
            )
        ]
    ),
    limit=3,
)

print("필터링 결과:", [(hit.id, hit.score) for hit in filtered_result])

5. 핵심 설정 레퍼런스

설정 항목 기본값 권장값 (프로덕션) 설명
storage.performance.max_optimization_threads 1 CPU 코어 수의 50%~75% 인덱스 최적화에 사용할 스레드 수
storage.optimizer.optimization_threads 1 2~4 세그먼트 병합 최적화 스레드
storage.optimizer.max_segment_size 50,000 100,000 (메모리 여유시) 세그먼트당 최대 포인트 수
hnsw_config.m 16 24~32 (정확도 우선)
12~16 (속도 우선)		 HNSW 그래프의 간선 수 (정확도↑ 메모리↑)
hnsw_config.ef_construct 100 200~400 (고정확도 필요시) 인덱스 구축 시 탐색 범위
wal_config.wal_capacity_mb 32 1024 (대용량 쓰기) WAL(Write-Ahead Log) 크기(MB)
collection_config.params.vectors.size - 임베딩 모델 차원에 맞춤 벡터 차원 수 (필수 설정)
service.grpc_port 6334 6334 (변경 시 클라이언트도 변경) gRPC 서비스 포트

설정 파일 예시 (config.yaml)

log_level: INFO
storage:
  performance:
    max_optimization_threads: 4
  optimizer:
    optimization_threads: 2
    max_segment_size: 100000
service:
  http_port: 6333
  grpc_port: 6334
  enable_cors: true

실행: docker run -p 6333:6333 -p 6334:6334 -v $(pwd)/config.yaml:/qdrant/config/production.yaml qdrant/qdrant

6. 실무 팁 & 트러블슈팅

  • 메모리 부족 문제

    증상: OOM Killer에 의해 Qdrant 프로세스 종료. 해결: storage.performance.max_optimization_threads 값을 낮추고, HNSW의 m 파라미터를 줄이며, storage.optimizer.max_segment_size를 조정합니다. 디스크 모드(storage_type: on_disk) 사용을 고려하세요.

  • 느린 검색 속도

    증상: 검색 latency가 높음. 해결: hnsw_config.ef 값을 낮추고(기본 10), 적절한 hnsw_config.m 값을 찾기 위해 벤치마킹합니다. 필터링 조건이 많다면 페이로드 인덱스를 생성하세요.

  • 벡터 차원 불일치 오류

    증상: "Wrong vector dimension" 에러. 해결: 컬렉션 생성 시 지정한 size와 실제 삽입하는 벡터의 차원이 일치하는지 확인하세요. 임베딩 모델 변경 시 새 컬렉션이 필요할 수 있습니다.

  • 대량 삽입 시 타임아웃

    증상: 대량 벡터 삽입 시 timeout 발생. 해결: 배치 크기를 줄이고(예: 100-200개), client.upsert()의 wait=true 옵션을 사용하며, 쓰기 스레드를 늘리세요(storage.performance.max_optimization_threads).

  • 디스크 공간 급증

    증상: 스토리지 디렉토리가 빠르게 증가. 해결: 세그먼트 병합이 제대로 이루어지지 않을 수 있습니다. storage.optimizer.* 설정을 조정하고, 불필요한 컬렉션을 삭제하세요.

주의사항

• 프로덕션 배포 전 반드시 스냅샷 백업 전략을 수립하세요.
• HNSW 파라미터(m, ef_construct, ef)는 메모리 사용량과 정확도/속도에 직접적인 영향을 미칩니다. 실제 데이터로 테스트하세요.
• 클러스터 모드에서 네트워크 지연 시간을 모니터링하세요. 5ms 이상의 지연은 성능에 큰 영향을 줍니다.
