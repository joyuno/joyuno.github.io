1. Apache Flink란?

Apache Flink는 상태 기반의 정확히 한 번(exactly-once) 처리 의미론을 보장하는 분산 스트림 처리 프레임워크입니다. 배치 처리를 유한 스트림의 특별한 경우로 간주하여 스트림과 배치를 통합한 단일 엔진으로 동작합니다.

🔁 상태 기반 처리

연산 중간 결과를 내부 상태(State)로 유지하며, 체크포인트를 통해 장애 발생 시 정확한 상태로 복구할 수 있습니다. 이는 실시간 애널리틱스, 이벤트 기반 애플리케이션에 필수적입니다.

⚡ 저지연 & 고처리량

이벤트 단위 처리와 마이크 배치 처리 방식을 혼용하여 밀리초 단위의 지연 시간과 초당 수백만 이벤트의 처리량을 동시에 달성할 수 있습니다.

🌐 통합 배치 & 스트림

DataSet API(배치)와 DataStream API(스트림)를 동일한 런타임 위에서 제공합니다. 최근에는 더욱 간결한 Table API & SQL로 통합되고 있습니다.

2. 왜 써야 하는가? (기존 대안과 비교)

실시간 데이터 처리 영역에서 Flink는 상태 관리와 정확성 보장 측면에서 두드러집니다.

프레임워크 처리 모델 상태 관리 정확성 보장 주요 사용 사례
Apache Flink True Streaming (이벤트 단위) 내장된 분산 상태, 체크포인트 Exactly-once 실시간 대시보드, 복잡 이벤트 처리(CEP), 실시간 ETL
Apache Spark Streaming Micro-batching (작은 배치) 외부 저장소 의존 (예: HDFS) Exactly-once (구현 복잡) 배치와 스트림 혼합 워크로드, 주기적 리포트
Apache Kafka Streams True Streaming 로컬 상태 (RocksDB) + Kafka Exactly-once (Kafka 한정) Kafka 기반 간단한 스트림 앱, 토픽 간 변환
Apache Storm True Streaming 제한적 (Trident API 필요) At-least-once 극저지연 처리, 트랜잭션 필요 없는 모니터링

💡 선택 가이드

Flink를 선택할 때: 상태가 복잡하고 정확한 한 번 처리가 필수이며, 지연 시간과 처리량을 모두 고려해야 하는 경우 (예: 실시간 추천, 사기 탐지, 주식 거래 모니터링).
다른 대안을 고려할 때: 이미 Spark 생태계가 구축되었거나, 마이크로 배치로 충분하며, 상태 관리가 간단한 경우.

3. 핵심 개념

Flink 애플리케이션은 소스(Source) → 변환(Transformation) → 싱크(Sink)의 흐름으로 구성됩니다. JobManager와 TaskManager가 클러스터를 구성합니다.

+------------------+ +---------------------+ +------------------+ | Data Sources | ---> | Flink Cluster | ---> | Data Sinks | | (Kafka, File, | | | | (Kafka, DB, | | Socket, etc.) | | +----------------+ | | File, etc.) | +------------------+ | | JobManager | | +------------------+ | | (Master) | | | | - Schedules | | | | - Coordinates | | | +----------------+ | | ^ | | | | | +------v--------+ | | | TaskManagers | | | | (Workers) | | | | - Execute | | | | - State | | | +---------------+ | +---------------------+ | +-------v-------+ | State Backend | | (Memory, | | RocksDB, | | FS) | +---------------+

주요 컴포넌트

  • JobManager (마스터): 작업 스케줄링, 체크포인트 조정, 장애 복구 담당.
  • TaskManager (워커): 실제 데이터 연산 태스크를 실행하고 상태를 관리.
  • State Backend: 상태 저장 방식을 정의 (메모리, 파일시스템, RocksDB).
  • Checkpoint & Savepoint: 장애 복구(Checkpoint)와 버전 업그레이드/스케일 아웃(Savepoint)을 위한 상태 스냅샷.

단계 3: 첫 번째 PyFlink 스트리밍 작업

소켓에서 들어오는 단어를 실시간으로 카운트하는 예제입니다.

Python
from pyflink.datastream import StreamExecutionEnvironment
from pyflink.datastream.connectors import SocketTextStreamSource
from pyflink.common.typeinfo import Types
from pyflink.datastream.functions import MapFunction, FlatMapFunction

# 실행 환경 생성
env = StreamExecutionEnvironment.get_execution_environment()
# 체크포인트 간격 설정 (장애 복구용)
env.enable_checkpointing(10000)  # 10초마다

# 소켓 소스 정의 (로컬 9999 포트)
source = SocketTextStreamSource("localhost", 9999, "\n", 0)
text_stream = env.add_source(source, "Socket Source")

# 변환: 단어 분리 및 카운트
class Tokenizer(FlatMapFunction):
    def flat_map(self, value):
        for word in value.lower().split():
            yield (word, 1)

class Adder(MapFunction):
    def map(self, value):
        return value[0], value[1] + 1

word_counts = (text_stream
    .flat_map(Tokenizer(), output_type=Types.TUPLE([Types.STRING(), Types.INT()]))
    .key_by(lambda x: x[0])  # 단어별로 키 지정
    .map(Adder(), output_type=Types.TUPLE([Types.STRING(), Types.INT()]))
)

# 결과 출력 (표준 출력)
word_counts.print()

# 작업 실행
print("소켓 스트림 처리 시작... (데이터를 localhost:9999로 전송하세요)")
print("예: nc -lk 9999")
env.execute("Socket WordCount")

🚀 실행 방법

1. 터미널 A에서 Flink 클러스터 실행: ./bin/start-cluster.sh
2. 터미널 B에서 netcat 서버 시작: nc -lk 9999
3. 터미널 C에서 위 Python 스크립트 실행: python wordcount.py
4. 터미널 B에 단어 입력 → 터미널 C에서 실시간 카운트 출력 확인

5. 핵심 설정 레퍼런스

실무에서 자주 조정하는 Flink 설정값들입니다. flink-conf.yaml 또는 코드에서 설정 가능합니다.

설정 키 기본값 설명 실무 권장값
taskmanager.numberOfTaskSlots 1 각 TaskManager의 태스크 슬롯 수 (CPU 코어당 1개 권장) 물리적 코어 수
parallelism.default 1 작업 기본 병렬도 TaskManager 수 × taskSlots
state.backend (없음) 상태 백엔드 유형 (jobmanager, filesystem, rocksdb) rocksdb (대용량 상태 시)
execution.checkpointing.interval (없음) 체크포인트 간격 (ms) 1m (60000) - 처리량과 복구 시간 절충
execution.checkpointing.timeout 10m 체크포인트 완료 대기 시간 5m (300000)
pipeline.object-reuse false 객체 재사용으로 GC 오버헤드 감소 (주의 필요) true (성능 최적화 시)
table.exec.state.ttl 0 (무제한) 상태 데이터의 TTL (Time-To-Live) 7d (604800000) - 상태 무한 증가 방지

6. 실무 팁 & 트러블슈팅

⚠️ 자주 겪는 문제 1: 백프레셔(Backpressure) 발생

증상: 웹 UI에서 작업 그래프가 빨간색, 처리 지연 증가, 체크포인트 타임아웃.
원인: 싱크 처리 속도가 소스 수신 속도보다 느림, 키 분포 편향(Data Skew), 리소스 부족.
해결: 1. 병렬도 증가 (env.set_parallelism()) 2. 상태 백엔드를 RocksDB로 변경 (대용량 상태 시) 3. 윈도우 크기 조정 또는 데이터 리파티셔닝 4. 싱크 측 병목 확인 (예: DB 쓰기 속도)

💡 실무 팁 1: 상태 TTL 설정으로 메모리 관리

키별 상태가 무한정 커지는 것을 방지하려면 항상 TTL을 설정하세요. 특히 세션 윈도우나 유저 활동 추적 시 필수입니다.

Python
from pyflink.common import Time
from pyflink.common.state import StateTtlConfig

ttl_config = (StateTtlConfig
    .new_builder(Time.days(7))  # 7일 후 상태 자동 삭제
    .set_update_type(StateTtlConfig.UpdateType.OnReadAndWrite)
    .set_state_visibility(StateTtlConfig.StateVisibility.NeverReturnExpired)
    .build())

# 상태 디스크립터에 TTL 적용
state_descriptor = ValueStateDescriptor("user_state", Types.STRING())
state_descriptor.enable_time_to_live(ttl_config)

⚠️ 자주 겪는 문제 2: 체크포인트 실패

증상: Checkpoint expired before completing 에러, 작업 실패.
원인: 네트워크 지연, 상태가 너무 커서 저장 시간 초과, 저장소 성능 문제.
해결: 1. 체크포인트 간격 증가 (execution.checkpointing.interval) 2. 타임아웃 시간 증가 (execution.checkpointing.timeout) 3. 증분 체크포인트 활성화 (state.backend.incremental: true for RocksDB) 4. HDFS 대신 로컬 SSD 사용 검토

💡 실무 팁 2: 키 선택 전략으로 데이터 편향 해결

특정 키에 데이터가 몰리는 경우(예: 인기 상품 ID), 리파티셔닝 전에 salting 기법을 적용하세요.

Python
from pyflink.datastream import StreamExecutionEnvironment

env = StreamExecutionEnvironment.get_execution_environment()

# 원본 스트림
data_stream = env.from_collection([...])

# Salting: 키에 랜덤 접미사 추가
def add_salt(element):
    import random
    key, value = element
    salt = random.randint(0, 9)  # 0-9 사이 소금
    return (f"{key}_{salt}", value)

salted_stream = data_stream.map(add_salt)

# Salting된 키로 리파티셔닝 후 집계
result = (salted_stream
    .key_by(lambda x: x[0])  # salted_key로 분산
    .reduce(lambda a, b: (a[0], a[1] + b[1]))  # 집계
)

# 최종 결과에서 salt 제거 (필요시)
final_result = result.map(lambda x: (x[0].split('_')[0], x[1]))
