1. GitHub Actions란?

GitHub Actions는 GitHub 리포지토리에서 직접 소프트웨어 워크플로우를 자동화할 수 있는 플랫폼이다. 코드 빌드, 테스트, 배포부터 이슈 관리, 컨테이너 배포, 인프라 프로비저닝까지 광범위한 작업을 자동화하는 데 사용된다.

🚀 자동화

코드 푸시, PR 생성, 이슈 오픈 등 GitHub 이벤트를 트리거로 CI/CD 파이프라인을 구동한다. 수동 작업을 제거하고 개발 생산성을 극대화한다.

🔧 통합

GitHub 생태계와 네이티브 통합. Secrets 관리, 환경 변수, 컨테이너 레지스트리(ghcr.io)와 완벽 호환된다. 외부 도구와의 연계도 Marketplace를 통해 쉽게 가능.

☁️ 확장성

Self-hosted Runner로 온프레미스 또는 특정 클라우드 환경에서도 실행 가능. 매트릭스 빌드로 멀티 OS/버전 테스트를 병렬 처리할 수 있다.

2. 왜 써야 하는가? (기존 대안과 비교)

도구 장점 단점 GitHub Actions 대비
Jenkins 높은 자유도, 풍부한 플러그인, 온프레미스 중심 설치/관리 부담, 설정 복잡성, UI/UX 구식 관리 부담 없음, YAML 기반 단순 설정, GitHub과 통합 최적화
GitLab CI 통합 플랫폼, 강력한 컨테이너 지원 GitLab에 종속적, SaaS 요금제 제한 GitHub 사용자에게 친숙, 무료 요금제 관대, Marketplace 생태계 풍부
CircleCI 빠른 실행 속도, 직관적인 UI 비용 부담, 복잡한 설정 시 가독성 저하 GitHub 내 무료 실행 분 충분, YAML 구조 직관적, Secrets 관리 통합
Azure DevOps MS 생태계 통합, 엔터프라이즈 기능 무겁고 복잡한 UI, GitHub 연동 추가 설정 필요 경량화된 워크플로우, 개발자 친화적, GitHub 리포지토리와 직접 연동

결론: 이미 GitHub을 사용 중이라면 설정/통합 비용이 가장 낮고, 무료 요금제로도 소규모 팀의 CI/CD 요구를 충분히 충족시킬 수 있다. 특히 오픈소스 프로젝트에는 무제한 무료 실행 시간이 제공된다.

3. 핵심 개념 (아키텍처 다이어그램)

워크플로우는 이벤트에 의해 트리거되며, 작업(Job)과 단계(Step)로 구성된다. 각 작업은 가상 환경(Runner)에서 실행된다.

+-----------------------------------------------------------------------------------+ | GitHub Repository | | | | +---------------------+ +----------------------+ +----------------+ | | | Event Trigger | ----> | Workflow File | ----> | Runner | | | | (push, pr, issue) | | (.github/workflows/*)| | (GitHub-hosted)| | | +---------------------+ +----------------------+ | or Self-hosted | | | +----------------+ | | | | | +---------------------+ +----------------------+ \|/ | | | GitHub Secrets | | Actions Marketplace| +------------+ | | | & Variables | | (Reusable Actions) | | Jobs & | | | +---------------------+ +----------------------+ | Steps | | | +------------+ | | | | | +----------------+ | | | Artifacts | | | | & Logs | | | +----------------+ | +-----------------------------------------------------------------------------------+
  • 워크플로우 (Workflow): 자동화된 프로세스를 정의하는 YAML 파일. 리포지토리의 .github/workflows/ 디렉토리에 저장.
  • 작업 (Job): 하나의 러너에서 실행되는 단계들의 집합. 기본적으로 작업들은 병렬 실행되지만, 의존성을 정의해 순차 실행 가능.
  • 단계 (Step): 작업 내에서 실행되는 개별 명령어 또는 액션.
  • 액션 (Action): 재사용 가능한 자동화 단위. 공식 Marketplace에서 찾거나 직접 작성 가능.
  • 러너 (Runner): 워크플로우를 실행하는 머신. GitHub 호스트형(ubuntu-latest 등) 또는 Self-hosted.

4. 퀵스타트: 설치 → 설정 → 첫 예제

4.1. 사전 준비

GitHub 계정과 리포지토리가 필요하다. 별도 설치할 것은 없다.

CLI: 로컬에서 GitHub CLI로 빠른 설정 (선택사항)
# GitHub CLI 설치 (macOS)
brew install gh

# GitHub 인증
gh auth login

# 현재 디렉토리를 새 리포지토리로 생성 및 푸시
gh repo create my-actions-demo --private --source=. --remote=origin --push

4.2. 첫 워크플로우 파일 생성

리포지토리 루트에 .github/workflows/ci.yml 파일을 생성한다.

YAML: 기본 Python 애플리케이션 CI 파이프라인
name: Python CI

on:
  push:
    branches: [ main, develop ]
  pull_request:
    branches: [ main ]

jobs:
  test:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        python-version: ["3.9", "3.10", "3.11"]

    steps:
      - name: Checkout repository
        uses: actions/checkout@v4

      - name: Set up Python ${{ matrix.python-version }}
        uses: actions/setup-python@v5
        with:
          python-version: ${{ matrix.python-version }}

      - name: Install dependencies
        run: |
          python -m pip install --upgrade pip
          if [ -f requirements.txt ]; then pip install -r requirements.txt; fi
          pip install pytest

      - name: Run tests with pytest
        run: |
          pytest -v --cov=./ --cov-report=xml

      - name: Upload coverage to Codecov
        uses: codecov/codecov-action@v3
        with:
          file: ./coverage.xml
          fail_ci_if_error: true

4.3. Secrets 설정 (예: Docker Hub)

리포지토리 Settings → Secrets and variables → Actions에서 추가.

YAML: Secrets 사용 예시 (Docker 빌드 및 푸시)
- name: Log in to Docker Hub
  uses: docker/login-action@v2
  with:
    username: ${{ secrets.DOCKER_USERNAME }}
    password: ${{ secrets.DOCKER_PASSWORD }}

- name: Build and push
  uses: docker/build-push-action@v4
  with:
    push: true
    tags: user/app:latest

5. 핵심 설정 레퍼런스

키워드 설명 예시
on 워크플로우를 트리거할 이벤트 정의 on: [push, pull_request] 또는 schedule: cron: '0 2 * * *'
jobs.<job_id>.runs-on 작업을 실행할 러너 환경 지정 ubuntu-latest, windows-2022, self-hosted
jobs.<job_id>.strategy.matrix 매트릭스 전략으로 여러 환경 병렬 실행 Python 버전, Node.js 버전, OS 조합 등
steps.uses 재사용 액션 호출 (Marketplace 또는 로컬) actions/checkout@v4, actions/setup-python@v5
steps.with 액션에 전달할 입력 매개변수 python-version: '3.10', node-version: '18'
steps.run 셸 명령어 실행 (기본 bash, Windows는 PowerShell) run: python script.py 또는 멀티라인 |
env 환경 변수 설정 (워크플로우/작업/단계 수준) env: { DATABASE_URL: ${{ secrets.DB_URL }} }
if 조건부 실행 (컨텍스트 변수 활용) if: github.ref == 'refs/heads/main'

6. 실무 팁 & 트러블슈팅

💡 팁 1: 캐시 활용으로 빌드 시간 단축

의존성 설치 시간을 줄이려면 actions/cache를 사용하라.

- name: Cache pip packages
  uses: actions/cache@v3
  with:
    path: ~/.cache/pip
    key: ${{ runner.os }}-pip-${{ hashFiles('**/requirements.txt') }}
    restore-keys: |
      ${{ runner.os }}-pip-

💡 팁 2: Self-hosted Runner 보안

프라이빗 네트워크에 Runner를 설치할 때는 레이블을 명시적으로 지정하고, 필요한 작업에만 사용하도록 제한하라.

jobs:
  deploy-on-prem:
    runs-on: [self-hosted, on-prem-linux]
    steps:
      - run: ./deploy.sh

⚠️ 문제 1: "Permission denied" when cloning

서브모듈이 있는 리포지토리에서 발생. checkout 액션에 토큰을 전달하라.

- uses: actions/checkout@v4
  with:
    token: ${{ secrets.GITHUB_TOKEN }}
    submodules: recursive

⚠️ 문제 2: Job 시간 초과 (Timeout)

GitHub 호스트형 Runner는 작업당 최대 6시간 실행 제한이 있다. 장시간 작업은 Self-hosted Runner로 이동하거나 작업을 분할하라.

💡 팁 3: 디버깅 로그 활성화

리포지토리 Secrets에 ACTIONS_STEP_DEBUG를 true로 설정하면 상세 디버그 로그를 볼 수 있다.

💡 팁 4: Composite Action으로 재사용성 높이기

여러 워크플로우에서 공통으로 사용하는 단계들을 Composite Action으로 묶어 관리하라.

# action.yml
name: 'Setup Python with caching'
description: 'Sets up Python and caches pip packages'
runs:
  using: "composite"
  steps:
    - uses: actions/setup-python@v5
    - uses: actions/cache@v3
      ...
