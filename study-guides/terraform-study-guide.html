1. Terraform란?

Terraform은 HashiCorp가 개발한 오픈소스 Infrastructure as Code(IaC) 도구입니다. 선언적 언어(HCL)를 사용하여 클라우드 및 온프레미스 리소스를 정의, 프로비저닝, 관리합니다.

선언적 구성

원하는 인프라 상태(What)를 코드로 선언합니다. Terraform이 이를 구현하는 방법(How)을 자동으로 결정합니다.

상태 관리

실제 인프라 상태를 상태 파일(.tfstate)에 저장하여 추적하고, 변경 시 필요한 작업만 계산합니다.

멀티 클라우드

AWS, Azure, GCP, Kubernetes, VMware 등 수백 개의 Provider를 통해 하이브리드 환경을 통합 관리합니다.

2. 왜 써야 하는가?

기존 인프라 관리 방식과 Terraform의 IaC 접근법을 비교합니다.

방식 장점 단점 Terraform 비교
GUI/콘솔 수동 관리 직관적, 즉시 적용 반복 불가, 문서화 없음, 오류 많음, 버전 관리 불가 코드화로 재사용, 문서화, 버전 관리 가능
CLI 스크립트 (Bash/Python) 자동화 가능, 재사용성 상태 추적 불가, 롤백 복잡, 의존성 관리 어려움 상태 기반으로 변경만 적용, 롤백 간소화
Cloud Native 템플릿 (ARM, CF) 클라우드 통합, 선언적 특정 클라우드 종속, 멀티 클라우드 불가, 모듈화 제한 멀티 클라우드 지원, 모듈 재사용성 높음
기존 IaC 도구 (Ansible, Puppet) 구성 관리 강함 프로비저닝(생성)에 약함, 상태 관리 부재 생성 + 구성 통합, 상태 파일로 정확한 관리
핵심 가치: Terraform은 인프라의 생성, 변경, 삭제를 코드로 관리하여 반복성, 문서화, 협업, 안정성을 극대화합니다.

3. 핵심 개념

Terraform의 핵심 구성 요소와 작동 흐름을 ASCII 다이어그램으로 표현합니다.

+-------------------+ +-------------------+ +-------------------+ | Terraform Code | | Terraform CLI | | Remote State | | (.tf files) |----->| (terraform) |----->| (Backend) | | | | | | | | - Provider | | - init | | - .tfstate 파일 | | - Resource | | - plan | | - 상태 저장소 | | - Variable | | - apply | | - 동기화 | | - Module | | - destroy | | | +-------------------+ +-------------------+ +-------------------+ ^ ^ | | v v +-------------------+ +-------------------+ +-------------------+ | Configuration | | Execution Plan | | Actual Infra | | 언어: HCL | | (Graph) | | (Cloud/On-prem) | | | | | | | | - 선언적 구문 | | - 의존성 그래프 | | - AWS EC2 | | - 변수/출력 | | - 변경 계산 | | - Azure VM | | - 함수 사용 | | - 작업 순서 결정 | | - GKE Cluster | +-------------------+ +-------------------+ +-------------------+
  • Provider: AWS, Azure, GCP 등 실제 인프라 플랫폼과 연결하는 플러그인.
  • Resource: 관리할 인프라 객체 (예: aws_instance, kubernetes_pod).
  • Module: 재사용 가능한 코드 단위로 인프라 패키징.
  • State (.tfstate): 실제 인프라와 코드의 매핑 상태를 저장한 JSON 파일.
  • Backend: 상태 파일을 저장하는 위치 (로컬, S3, Terraform Cloud).

4. 퀵스타트

실습 환경 구축부터 첫 리소스 생성까지 단계별 가이드.

Step 1: 설치

CLI (MacOS/Linux 예시)
# Terraform 다운로드 및 설치 (최신 버전)
wget https://releases.hashicorp.com/terraform/1.6.0/terraform_1.6.0_linux_amd64.zip
unzip terraform_1.6.0_linux_amd64.zip
sudo mv terraform /usr/local/bin/

# 설치 확인
terraform version

Step 2: 프로젝트 설정

main.tf
# AWS Provider 정의
terraform {
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
  }
}

# Provider 구성 (환경 변수나 AWS CLI 설정 활용)
provider "aws" {
  region = "ap-northeast-2"  # 서울 리전
}

# 변수 정의
variable "instance_type" {
  description = "EC2 인스턴스 타입"
  default     = "t3.micro"
}

Step 3: 첫 리소스 생성

ec2.tf
# EC2 인스턴스 생성
resource "aws_instance" "web_server" {
  ami           = "ami-0c9c942bd7bf113a2"  # Amazon Linux 2023
  instance_type = var.instance_type

  tags = {
    Name = "Terraform-Quickstart-Instance"
  }
}

# 출력값 정의 (생성 후 확인용)
output "instance_ip" {
  value = aws_instance.web_server.public_ip
}

Step 4: 실행

CLI 실행 순서
# 초기화 (Provider 다운로드)
terraform init

# 실행 계획 확인 (무엇이 생성/변경될지)
terraform plan

# 실제 적용 (리소스 생성)
terraform apply -auto-approve

# 출력값 확인
echo "생성된 인스턴스 IP: $(terraform output -raw instance_ip)"

# 삭제 (인프라 제거)
terraform destroy -auto-approve
주의: `apply`와 `destroy`는 실제 인프라를 생성/삭제합니다. 테스트 환경에서 먼저 실행하세요.

5. 핵심 설정 레퍼런스

실무에서 자주 사용되는 Terraform 구성 요소의 주요 설정을 표로 정리합니다.

구성 요소 키/설정 설명 예시
Provider region, alias, profile 클라우드 지역, 다중 계정, 프로파일 지정 region = "us-east-1"
Resource depends_on, count, for_each 의존성 명시, 다수 생성, 각각 생성 count = 3
Variable type, default, sensitive 데이터 타입, 기본값, 민감 정보 숨김 type = string
Output value, description, sensitive 출력 값, 설명, 민감 정보 숨김 value = aws_vpc.id
Backend s3, workspace_key_prefix S3 백엔드, 상태 파일 경로 패턴 bucket = "my-terraform-state"
Module source, version 모듈 위치 (로컬, Git, Registry), 버전 source = "terraform-aws-modules/vpc/aws"
Data Source filter, id 기존 리소스 조회 조건, 식별자 filter { name = "vpc-name" }

6. 실무 팁 & 트러블슈팅

자주 겪는 문제 1: 상태 파일 충돌

문제: 팀원이 동시에 apply 시 상태 파일이 충돌하여 변경 사항 손실.

해결: 원격 백엔드(S3 + DynamoDB) 사용. Terraform Cloud 또는 AWS S3에 상태를 저장하고 DynamoDB로 Locking 적용.

terraform {
  backend "s3" {
    bucket         = "team-terraform-state"
    key            = "project/terraform.tfstate"
    region         = "ap-northeast-2"
    dynamodb_table = "terraform-locks"
  }
}

자주 겪는 문제 2: Provider 버전 호환

문제: Provider 버전 업데이트 후 리소스 속성 오류 발생.

해결: required_providers 블록에 버전 제한 명시. 주요 업데이트 전 plan으로 검증.

required_providers {
  aws = {
    source  = "hashicorp/aws"
    version = "~> 5.0"  # 5.0.x만 허용, 5.1 자동 업데이트 방지
  }
}

자주 겪는 문제 3: 대규모 리소스 삭제 실패

문제: destroy 실행 시 의존성 문제로 특정 리소스 삭제 실패.

해결: -target 옵션으로 특정 리소스 먼저 삭제 후 전체 destroy. 상태 파일에서 리소스 제거 후 import 재시도.

# 특정 리소스만 대상으로 삭제
terraform destroy -target=aws_instance.web_server

# 상태 파일에서 리소스 제거 후 실제 인프라와 재연결
terraform state rm aws_instance.web_server
terraform import aws_instance.web_server i-abc123def
실무 팁:
  • 모듈화: 재사용 가능한 모듈을 만들어 프로젝트 구조를 단순화.
  • 환경 변수: 민감 정보(키, 토큰)는 환경 변수로 관리.
  • 코드 검증: terraform fmt, validate를 CI/CD 파이프라인에 통합.
  • 변수 파일: dev.tfvars, prod.tfvars로 환경별 변수 분리.
