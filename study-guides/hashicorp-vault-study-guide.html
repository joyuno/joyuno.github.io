1. HashiCorp Vault란?

HashiCorp Vault는 비밀 정보(Secrets) 관리, 데이터 암호화, 그리고 Identity-Based Access 제어를 위한 중앙화된 도구입니다. 애플리케이션, 서비스, 인프라 구성 요소들이 안전하게 비밀번호, API 키, 인증서 등에 접근할 수 있도록 해주는 보안 플랫폼입니다.

Secrets Management

정적 비밀번호부터 동적 생성 비밀번호(예: DB 계정)까지 전 생애주기를 중앙에서 관리. 접근 정책(Policy)으로 세밀한 권한 제어 가능.

Data Encryption as a Service

애플리케이션에서 직접 암호화 키를 관리할 필요 없이, Vault API를 호출해 데이터를 암호화/복호화. 키 회전(Key Rotation) 자동화.

Identity-Based Access

Kubernetes, AWS, GitHub 등 다양한 인증 방법(Auth Method)을 통해 인증된 클라이언트에게 자동으로 비밀 정보 접근 권한 부여.

2. 왜 써야 하는가?

기존 비밀 정보 관리 방식(환경 변수, 설정 파일, 다른 클라우드 서비스)의 한계를 해결하고, 보안성과 운영 효율성을 동시에 높입니다.

방식장점단점Vault의 해결책
환경 변수 / .env 파일간단함, 빠른 적용정적 관리, 권한 제어 어려움, 유출 시 대응 불가동적 비밀 생성, 접근 로깅, 자동 만료/회전
클라우드 Secrets Manager (AWS 등)관리형 서비스, 통합 쉬움벤더 종속, 멀티클라우드/하이브리드 환경 비효율오픈소스, 하이브리드/멀티클라우드 지원, 일관된 인터페이스
설정 파일에 하드코딩없음보안 최악, 버전 관리 시스템에 유출, 회전 불가비밀 정보를 코드/저장소에서 완전 분리
자체 구축 솔루션커스터마이징 가능유지보수 부담, 보안 전문성 요구, 확장성 문제검증된 오픈소스 엔터프라이즈 솔루션 제공
핵심 가치: Vault는 단순히 비밀을 저장하는 곳이 아닙니다. 비밀 정보의 수명주기 관리, 암호화 서비스, 그리고 인증된 엔티티만 접근하도록 보장하는 정책 기반 접근 제어를 통합적으로 제공합니다.

3. 핵심 개념

아키텍처 개요

+-------------------+ 인증(Auth) +----------------------+ | 클라이언트 | -------------------> | Vault 서버 | | (App, CLI, CI/CD)| | (Storage Backend) | +-------------------+ | +----------------+ | | | | Secret Engine | | | 비밀 읽기/쓰기 | | - KV | | | (인증 후 Token) | | - Database | | v | | - PKI | | +-------------------+ | | - Transit | | | 비밀 정보/ | | +----------------+ | | 암호화 데이터 | +-----------|----------+ +-------------------+ | [영구 저장] (Consul, Raft, S3...)

주요 컴포넌트

  • Storage Backend: Vault의 상태(암호화된 비밀, 구성)를 영구 저장. (Raft(내장), Consul, AWS S3 등)
  • Secret Engine: 비밀을 생성/저장/관리하는 기능 단위. (KV, Database, PKI, Transit, AWS 등)
  • Auth Method: 사용자/서비스 인증 방식. (Token, Userpass, AppRole, Kubernetes, AWS IAM 등)
  • Policy: 인증 후 부여되는 권한을 정의한 규칙(HCL). (예: "경로 'secret/data/team/*'에 대한 읽기 권한")
  • Token: 인증 후 발급되는 임시 접근 키. TTL(Time-To-Live)로 수명 관리.

4. 퀵스타트

Step 1: 개발 모드로 Vault 서버 실행

가장 빠른 시작 방법입니다. (주의: 데이터가 메모리에만 저장되며, 프로덕션 절대 불가)

# 1. 바이너리 다운로드 (Mac/Linux 예시)
wget https://releases.hashicorp.com/vault/1.16.0/vault_1.16.0_linux_amd64.zip
unzip vault_1.16.0_linux_amd64.zip
sudo mv vault /usr/local/bin/

# 2. 개발 서버 실행
vault server -dev -dev-root-token-id="root-token-12345"

# 3. 새 터미널에서 환경 변수 설정
export VAULT_ADDR='http://127.0.0.1:8200'
export VAULT_TOKEN='root-token-12345'

Step 2: 첫 비밀 저장과 읽기 (KV Secret Engine)

# KV v2 시크릿 엔진 활성화 (경로: secret/)
vault secrets enable -path=secret kv-v2

# 비밀 쓰기 (경로: secret/myapp/config)
vault kv put secret/myapp/config username="appuser" password="s3cr3tP@ss"

# 비밀 읽기
vault kv get secret/myapp/config

# 특정 필드만 읽기 (JSON 출력)
vault kv get -field=password secret/myapp/config

Step 3: Python 애플리케이션에서 비밀 읽기

# pip install hvac
import hvac

# Vault 클라이언트 초기화 (개발 서버용)
client = hvac.Client(
    url='http://127.0.0.1:8200',
    token='root-token-12345'  # 실제 환경에선 AppRole 등 안전한 방법 사용
)

# 비밀 읽기
secret_response = client.secrets.kv.v2.read_secret_version(
    path='myapp/config',
    mount_point='secret'
)

secret_data = secret_response['data']['data']
print(f"Username: {secret_data['username']}")
print(f"Password: {secret_data['password']}")
주의: 개발 모드(-dev)는 절대 프로덕션에서 사용하지 마세요. 루트 토큰이 평문으로 노출되며, 서버 재시작 시 모든 데이터가 사라집니다.

6. 실무 팁 & 트러블슈팅

문제: "permission denied" 오류

원인: 클라이언트 토큰에 해당 경로에 대한 Policy 권한이 없음.

해결:

# 1. 현재 토큰의 정책 확인
vault token lookup

# 2. 해당 경로에 대한 정책 작성/수정
# 예: policy.hcl
path "secret/data/myapp/*" {
  capabilities = ["read"]
}

# 3. 정책 적용
vault policy write myapp-read policy.hcl

# 4. 토큰에 정책 연결 (인증 방법에 따라 다름)
# AppRole의 경우 Role에 정책 부여
vault write auth/approle/role/myrole policies="myapp-read"

문제: Vault 서버 재시작 후 "seal" 상태

원인: Vault는 기본적으로 봉인(Seal) 상태로 시작. 복호화 키 필요.

해결:

# 1. Unseal Key(복구 키)로 봉인 해제
# 보통 3/5, 4/7 같은 Shamir 분산 기법 사용
vault operator unseal [Unseal Key 1]
vault operator unseal [Unseal Key 2]
vault operator unseal [Unseal Key 3]
# 필요한 임계값 도달 시 자동 unseal

# 2. 자동 Unseal 설정 (Cloud KMS 등)을 프로덕션에 추천

팁: Kubernetes에서 안전하게 사용하기

방법: ServiceAccount 토큰을 이용한 Kubernetes 인증 방식 사용.

# 1. Vault에 Kubernetes 인증 방법 활성화
vault auth enable kubernetes

# 2. Vault가 Kubernetes API와 통신할 수 있도록 구성
vault write auth/kubernetes/config \
    token_reviewer_jwt="$(cat /var/run/secrets/kubernetes.io/serviceaccount/token)" \
    kubernetes_host="https://$KUBERNETES_PORT_443_TCP_ADDR:443" \
    kubernetes_ca_cert=@/var/run/secrets/kubernetes.io/serviceaccount/ca.crt

# 3. Kubernetes ServiceAccount와 Vault Policy 매핑
vault write auth/kubernetes/role/myapp-role \
    bound_service_account_names=myapp-sa \
    bound_service_account_namespaces=default \
    policies=myapp-read \
    ttl=1h

팁: Database 동적 비밀 생성

시나리오: 애플리케이션마다 독립된 DB 계정, 자동 회전.

# 1. Database Secret Engine 활성화
vault secrets enable database

# 2. PostgreSQL 연결 구성
vault write database/config/my-postgresql \
    plugin_name=postgresql-database-plugin \
    connection_url="postgresql://{{username}}:{{password}}@localhost:5432/postgres" \
    allowed_roles="myapp-db-role" \
    username="vaultadmin" \
    password="adminpassword"

# 3. 역할 생성 (비밀 생성 규칙)
vault write database/roles/myapp-db-role \
    db_name=my-postgresql \
    creation_statements="CREATE ROLE \"{{name}}\" WITH LOGIN PASSWORD '{{password}}' VALID UNTIL '{{expiration}}'; GRANT SELECT ON ALL TABLES IN SCHEMA public TO \"{{name}}\";" \
    default_ttl="1h" \
    max_ttl="24h"

# 4. 동적 비밀 생성 (매번 새 계정)
vault read database/creds/myapp-db-role
