1. Kubeflow란?

Kubeflow는 Kubernetes 위에서 머신러닝 워크플로우(ML Workflow)를 배포, 관리, 확장하기 위한 오픈소스 플랫폼입니다. ML 시스템의 수명 주기(개발, 학습, 배포, 모니터링)를 Kubernetes의 이점(이식성, 확장성, 리소스 효율)을 활용해 관리합니다.

ML 워크플로우 오케스트레이션

복잡한 ML 파이프라인(데이터 전처리 → 학습 → 평가 → 배포)을 재현 가능하고 재사용 가능한 컴포넌트로 정의하고 실행합니다. Argo Workflows를 기반으로 합니다.

실험 관리 및 추적

MLflow, Kubeflow Metadata, Katib를 통해 하이퍼파라미터, 메트릭, 아티팩트를 추적하고 실험을 비교합니다.

서빙 및 프로덕션화

KServe(KFServing), Seldon Core를 통한 표준화된 모델 서빙, A/B 테스트, 카나리 배포, 자동 스케일링을 지원합니다.

핵심 가치: Kubeflow는 "Kubernetes for ML"을 실현합니다. 각 ML 단계를 컨테이너화하고, K8s 네이티브 리소스로 관리함으로써 클라우드/온프레미스 환경에 구애받지 않는 통합 플랫폼을 제공합니다.

2. 왜 써야 하는가? (기존 대안과 비교)

솔루션 장점 단점 / Kubeflow와의 차이 적합한 경우
Kubeflow 엔드투엔드 통합, K8s 네이티브, 확장성, 커뮤니티 활성 학습 곡선 가파름, 운영 복잡성 K8s 환경의 프로덕션 ML 시스템 전체 관리
MLflow 실험 추적 우수, 사용 편의성, 경량 오케스트레이션/서빙 기능 약함, 스케일링 제한 실험 관리와 모델 레지스트리 중심 프로젝트
Airflow 뛰어난 스케줄링, 다양한 연동, 성숙한 생태계 ML 특화 기능 부족, 컨테이너/리소스 관리 직접 구현 필요 ETL/배치 중심 워크플로, ML이 일부인 경우
Custom Script + K8s 유연성 최대, 특정 요구사항 정확히 구현 모든 인프라와 도구를 직접 개발/유지보수해야 함 매우 특수한 요구사항, 대규모 플랫폼 팀 보유

결론: 이미 Kubernetes를 사용 중이고, ML 워크플로의 프로덕션화와 표준화를 목표로 한다면 Kubeflow는 가장 합리적인 선택입니다.

3. 핵심 개념

Kubeflow는 여러 컴포넌트의 집합체입니다. 주요 컴포넌트와 그 관계를 이해해야 합니다.

+----------------------------------------------------------------------------------------+ | User Interfaces (UI/CLI/API) | | (Kubeflow Dashboard, KFP Client, SDK) | +----------------+----------------------+-----------------------+------------------------+ | | | | | | Pipelines | Training | Serving | Hyperparameter | | (KFP) | (TFJob, PyTorch)| (KServe) | Tuning (Katib) | | | Job, MPIJob) | | | +----------------+----------------------+-----------------------+------------------------+ | | | Kubernetes Native CRDs & Operators | | (Pod, Service, Deployment, PersistentVolume, etc.) | | | +----------------------------------------------------------------------------------------+ | | | Kubernetes Cluster | | (On-prem, Cloud GKE/EKS/AKS, Managed K8s) | | | +----------------------------------------------------------------------------------------+

주요 컴포넌트 설명

  • Kubeflow Pipelines (KFP): 파이프라인 정의, 실행, 관리 엔진. DSL/SDK로 파이프라인 작성.
  • Training Operators (TFJob, PyTorchJob 등): 분산 ML 학습 작업을 관리하는 K8s 커스텀 리소스.
  • KServe: 표준화된 고성능 모델 서빙 레이어. 자동 스케일링, 카나리 배포 지원.
  • Katib: 하이퍼파라미터 튜닝, 신경망 아키텍처 탐색(NAS) 시스템.
  • Central Dashboard: 통합 웹 UI.
  • Kubernetes: 모든 컴포넌트가 K8s 리소스(CRD)로 구현됨.

4. 퀵스타트 (Minikube 기반 로컬 설치)

전제 조건

# kubectl, minikube, kustomize 설치 확인
kubectl version --client
minikube version
kustomize version

단계 1: Minikube 클러스터 시작 및 리소스 할당

minikube start --cpus=6 --memory=12g --disk-size=50g
minikube addons enable ingress
minikube addons enable metrics-server

단계 2: Kubeflow 설치 (Manifest 방식)

# Kubeflow manifests 다운로드
export KUBEFLOW_VERSION=v1.8.0
git clone -b ${KUBEFLOW_VERSION} https://github.com/kubeflow/manifests.git
cd manifests

# 단일 명령어 설치 (모든 컴포넌트)
while ! kustomize build example | kubectl apply -f -; do echo "Retrying..."; sleep 10; done
주의: 네트워크 환경에 따라 설치에 20-30분 소요될 수 있습니다. `kubectl get pods -n kubeflow`로 모든 Pod가 Running 상태가 될 때까지 기다리세요.

단계 3: 대시보드 접근

# 포트 포워딩
kubectl port-forward svc/istio-ingressgateway -n istio-system 8080:80

# 브라우저에서 http://localhost:8080 접속
# 기본 자격 증명 (설치 방식에 따라 다름) 예: user@example.com / 12341234

단계 4: 첫 번째 파이프라인 실행 (Python SDK 예제)

first_pipeline.py

import kfp
from kfp import dsl
from kfp.components import create_component_from_func

# 1. 간단한 컴포넌트 정의 (컨테이너화된 함수)
@create_component_from_func
def preprocess_op(data_path: str) -> str:
    import pandas as pd
    import json
    # 간단한 전처리 로직 (예시)
    df = pd.read_csv(data_path)
    processed_path = '/tmp/processed_data.csv'
    df.to_csv(processed_path, index=False)
    return processed_path

@create_component_from_func
def train_op(processed_data_path: str, model_path: str):
    from sklearn.ensemble import RandomForestClassifier
    import pandas as pd
    import joblib
    df = pd.read_csv(processed_data_path)
    X = df.drop('target', axis=1)
    y = df['target']
    model = RandomForestClassifier(n_estimators=10)
    model.fit(X, y)
    joblib.dump(model, model_path)

# 2. 파이프라인 정의
@dsl.pipeline(
    name='First ML Pipeline',
    description='A simple training pipeline'
)
def my_pipeline(data_url: str = 'gs://my-bucket/data.csv'):
    preprocess_task = preprocess_op(data_path=data_url)
    train_task = train_op(
        processed_data_path=preprocess_task.output,
        model_path='/tmp/model.joblib'
    )

# 3. 클라이언트 생성 및 실행
client = kfp.Client(host='http://localhost:8080/pipeline')
client.create_run_from_pipeline_func(
    my_pipeline,
    arguments={},
    experiment_name='Quickstart'
)
# 실행
python first_pipeline.py

5. 핵심 설정 레퍼런스

설정 항목 권장값 / 예시 설명 관련 컴포넌트
파이프라인 실행 Timeout ttl_seconds_after_finished: 604800 완료된 실행 기록 보관 기간 (초). 기본값은 무제한. KFP
Pod 기본 리소스 요청/제한 limits: {cpu: 2, memory: 4Gi} 파이프라인 컴포넌트 Pod에 대한 기본 리소스 할당. 과도한 리소스 소비 방지. KFP, Training
Artifact 저장소 minio 또는 S3/GCS 엔드포인트 파이프라인 실행 중 생성된 아티팩트(모델, 메트릭) 저장 위치. KFP
모델 서빙 스케일링 minReplicas: 1, maxReplicas: 5 KServe InferenceService의 자동 스케일링 범위. KServe
Katib Experiment Config maxTrialCount: 20, parallelTrialCount: 3 하이퍼파라미터 튜닝 시 최대 시도 횟수와 병렬 실행 수. Katib
PersistentVolume (PVC) StorageClass standard 또는 fast-ssd 데이터셋, 모델 체크포인트 저장용 영구 볼륨 클래스. 전체

6. 실무 팁 & 트러블슈팅

문제: ImagePullBackOff / ErrImagePull

원인: 프라이빗 레지스트리 이미지 또는 네트워크 문제.

해결:

# 1. 이미지 풀 시크릿 생성
kubectl create secret docker-registry my-registry-key \
  --docker-server= \
  --docker-username= \
  --docker-password= \
  -n kubeflow

# 2. Pod/Deployment의 imagePullSecrets에 추가

문제: 파이프라인 실행이 Pending 상태

원인: 리소스 부족 (CPU/Memory/GPU) 또는 PVC 바인딩 실패.

해결:

# 리소스 이벤트 확인
kubectl describe pod  -n kubeflow

# 노드 리소스 확인
kubectl describe nodes

# PVC 상태 확인
kubectl get pvc -n kubeflow

팁: 파이프라인 컴포넌트 캐싱 활용

동일한 입력으로 반복 실행 시 캐싱을 통해 비용/시간 절약.

@dsl.pipeline(
    name='cached-pipeline',
)
def my_pipeline():
    task = my_component(...)
    # 캐싱 비활성화가 필요한 경우
    task.execution_options.caching_strategy.max_cache_staleness = "P0D"

팁: 환경 변수 및 시크릿 관리

민감한 정보(API 키, DB 패스워드)는 Kubernetes Secret으로 관리.

# 시크릿 생성
kubectl create secret generic ml-secrets \
  --from-literal=API_KEY='xxx' -n kubeflow

# 파이프라인 컴포넌트에서 환경 변수로 사용
def my_op():
    import os
    key = os.environ['API_KEY']
    # ...
디버깅 기본 순서: 1) `kubectl get pods -n kubeflow`로 상태 확인 → 2) 문제 Pod의 `kubectl describe pod` → 3) `kubectl logs -c ` 로그 확인 → 4) 관련 K8s 리소스(Service, Ingress, PVC) 상태 점검.
