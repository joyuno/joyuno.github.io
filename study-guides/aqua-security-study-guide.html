1. Aqua Security란?

Aqua Security는 클라우드 네이티브 애플리케이션 보호 플랫폼(CNAPP) 및 컨테이너 워크로드 보호 플랫폼(CWPP)의 선도 제공자입니다. 개발부터 런타임까지의 전체 소프트웨어 수명 주기(SDLC)에 걸쳐 취약점 관리, 규정 준수, 위협 감지, 런타임 보호를 제공하는 통합 보안 솔루션입니다.

취약점 관리 & 이미지 스캔

컨테이너 이미지, IaC 템플릿, 레지스트리, 호스트 OS의 취약점 및 구성 오류를 정적 분석(SAST) 및 동적 분석(DAST)을 통해 조기에 발견합니다.

런타임 보호 & 행동 분석

컨테이너, 서버리스, 호스트의 런타임 동작을 지속적으로 모니터링하여 이상 활동, 파일 무결성 위반, 네트워크 공격 패턴을 실시간으로 감지하고 차단합니다.

규정 준수 & 자동화

CIS 벤치마크, GDPR, HIPAA, PCI-DSS, NIST 등 주요 규정 및 표준에 대한 자동화된 평가와 보고서 생성을 통해 지속적인 컴플라이언스를 보장합니다.

2. 왜 써야 하는가? (기존 대안과 비교)

솔루션 주요 초점 Aqua 대비 장단점 적합한 시나리오
Aqua Security 통합 CNAPP (전체 수명 주기) 장점: 통합성, 자동화, 런타임 차단, Kubernetes 네이티브
단점: 상대적 비용, 학습 곡선		 엔터프라이즈 K8s, 멀티클라우드, 규제 산업
Sysdig Secure 런타임 포렌식, 감시 Sysdig Capture 강점, 포렌식에 특화. Aqua는 예방/차단에 더 중점. 사고 대응, 심층 감시가 주 요구사항일 때
Prisma Cloud (Palo Alto) 클라우드 보안 포스트어, CSPM 클라우드 인프라 보안(CSPM)이 매우 강함. Aqua는 워크로드 보호(CWPP)에 더 특화. 멀티클라우드 인프라 보안이 최우선일 때
Open Source (Trivy, Falco) 특정 기능 (스캔/런타임) 비용 효율적, 유연함. Aqua는 통합 관리, 지원, 자동화 제공. 소규모 팀, DIY 가능, 제한된 예산
Native (EKS/GKE Security) 특정 클라우드 제공자 통합 간단한 통합, 벤더 Lock-in. Aqua는 멀티클라우드/하이브리드 일관성 제공. 단일 클라우드 벤더에 깊이 종속된 환경

결론: Aqua는 특히 Kubernetes 기반의 복잡한 클라우드 네이티브 환경에서 통합성과 자동화된 런타임 보호를 필요로 하는 조직에 적합합니다. 오픈소스 도구들의 조합으로 유사 기능을 구축하는 것보다 관리 부담과 보안 격차를 줄여줍니다.

3. 핵심 개념과 아키텍처

Aqua 플랫폼은 주로 Aqua Console (관리 평면)과 Enforcer/Scanner (데이터 평면)로 구성됩니다. 다음 ASCII 다이어그램은 전형적인 Kubernetes 환경에서의 구성 요소 상호작용을 보여줍니다.

+----------------------------------------------------------------------------------------+ | Aqua Cloud / Self-Hosted Console | | [ 취약점 DB ] [ 정책 엔진 ] [ 대시보드 ] [ 리포트 ] [ API ] | +------------------------------------------+-------------------------------------------+ | (REST API / gRPC) +------------------------------------------v-------------------------------------------+ | Kubernetes Cluster (Target) | | +------------------+ +------------------+ +------------------+ | | | Aqua Enforcer | | Aqua Scanner | | Aqua KubeEnforcer| | | | (DaemonSet/Depl.)| | (Job/CronJob) | | (Admission Ctrl) | | | | - Runtime Protect| | - Image Scanning | | - Image Assurance| | | | - File Integrity | +------------------+ | - Policy Block | | | | - Network Firewall| +------------------+ | | +------------------+ | | | | | v | | +------------------+ +------------------+ +------------------+ | | | 워크로드 Pod | | 워크로드 Pod | | 워크로드 Pod | | | | (App Container) | | (App Container) | | (App Container) | | | +------------------+ +------------------+ +------------------+ | | | | +------------------+ +------------------+ | | | Container Registry|<--- Scan --->| CI/CD Pipeline | | | | (ECR, ACR, Harbor)| | (Jenkins, GitLab)| | | +------------------+ +------------------+ | +-------------------------------------------------------------------------------------+

주요 구성 요소 설명

  • Aqua Console: 중앙 관리 인터페이스. 정책 정의, 모니터링, 보고.
  • Aqua Enforcer (DaemonSet): 각 노드에 배포되어 런타임 보호(프로세스, 파일, 네트워크 모니터링) 및 정책 실행.
  • Aqua Scanner: 이미지, 레지스트리, 인프라를 스캔하여 취약점 및 미스컨피규레이션 탐지.
  • Aqua KubeEnforcer (Admission Controller): Kubernetes API 서버에 통합되어 Pod 생성 시 정책 기반 승인/거부 결정.
  • Aqua Gateway: Self-Hosted 설치 시 구성 요소 간 통신 중계 및 오프라인 업데이트 캐시.

4. 퀵스타트: 설치부터 첫 실행까지

Kubernetes (EKS) 환경에 Aqua Enforcer를 설치하고 기본 이미지 스캔을 수행하는 과정입니다.

  1. 전제 조건 확인 및 CLI 도구 설치

    Aqua CLI 도구인 aquactl을 설치합니다.

    Shell
    # macOS / Linux
curl -sL https://download.aquasec.com/aquactl/stable/aquactl-$(uname -s)-$(uname -m) -o aquactl
chmod +x aquactl
sudo mv aquactl /usr/local/bin/

# 버전 확인
aquactl version

  2. Aqua Console에 연결 및 토큰 생성

    Aqua Cloud 콘솔 또는 Self-Hosted 콘솔에 로그인하여 Kubernetes용 배포 토큰을 생성합니다. (콘솔 UI: Integrations → Kubernetes)

  3. Helm을 이용한 Enforcer 설치

    가장 일반적인 설치 방법은 Helm을 사용하는 것입니다.

    Shell
    # Helm repo 추가
helm repo add aqua-helm https://aquasecurity.github.io/helm-charts/
helm repo update

# 네임스페이스 생성
kubectl create namespace aqua

# Enforcer 설치 (토큰과 콘솔 주소는 실제 값으로 변경)
helm install aqua-enforcer aqua-helm/enforcer \
  --namespace aqua \
  --set global.platform=eks \
  --set global.enforcerToken=<YOUR_ENFORCER_TOKEN> \
  --set global.gateway.address=<AQUA_CONSOLE_URL> \
  --set global.gateway.port=3622

  4. 기본 정책 검증 및 이미지 스캔 실행

    설치가 완료되면 Aqua CLI를 사용해 로컬 이미지를 스캔하거나, KubeEnforcer가 배포되었다면 Pod 생성 시 자동 스캔이 트리거됩니다.

    Shell - 로컬 이미지 스캔
    # Docker 이미지 스캔 예제 (Aqua Scanner 사용)
docker pull nginx:latest
aquactl scan image nginx:latest --registry docker.io --user <CONSOLE_USER> --password <CONSOLE_PASSWORD>

# 결과는 취약점 목록과 심각도별 요약을 JSON 또는 테이블 형식으로 출력합니다.
    Python - Aqua API를 통한 스캔 결과 조회
    import requests
import json

AQUA_URL = "https://your-aqua-console.com"
API_KEY = "your-api-key"
headers = {"Authorization": f"Bearer {API_KEY}"}

# 최근 이미지 스캔 결과 가져오기
response = requests.get(f"{AQUA_URL}/api/v1/images", headers=headers)
images = response.json()

for image in images['result']:
    if image['vulns_found'] > 0:
        print(f"Image: {image['registry']}/{image['repo_name']}:{image['tag']}")
        print(f"  Critical Vulns: {image['crit_vulns']}, High Vulns: {image['high_vulns']}")

5. 핵심 설정 레퍼런스

설정 영역 키/파라미터 기본값/예시 설명
Runtime Protection enforcer.runtime.enabled true 런타임 행동 모니터링 활성화
enforcer.runtime.forensic true 사고 발생 시 포렌식 데이터 수집 활성화
enforcer.networkProtection.enabled false 네트워크 방화벽 및 DDoS 보호 활성화 (성능 영향 고려)
Image Scanning scanner.scanOnPush true 레지스트리에 이미지 Push 시 자동 스캔
scanner.ignoreUnfixed false 수정 패치가 아직 없는 취약점도 보고
scanner.severityThreshold HIGH 이 심각도 이상일 경우 CI/CD 파이프라인 실패 (CRITICAL, HIGH, MEDIUM, LOW)
Admission Control (KubeEnforcer) kubeEnforcer.auditFailure true 정책 위반 시 감사 로그만 기록 (차단하지 않음)
kubeEnforcer.denyUnscanned false 스캔되지 않은 이미지의 Pod 실행 차단
kubeEnforcer.policies.image_assurance 예: 블랙리스트 CVE, 허용 레지스트리 Pod 생성 시 평가할 세부 보안 정책
Integrations global.integrations.jira.enabled false Jira 티켓 자동 생성 연동
global.integrations.slack.webhook "" 심각한 알림을 Slack 채널로 전송

6. 실무 팁 & 트러블슈팅

문제: Enforcer Pod가 CrashLoopBackOff 상태

증상: kubectl get pods -n aqua에서 Enforcer Pod가 반복적으로 재시작됨.

원인 및 해결:

  1. 토큰 또는 게이트웨이 주소 오류: kubectl logs -n aqua <enforcer-pod> --previous로 로그 확인. 토큰이 만료되었거나 Console 주소/포트가 잘못되었을 수 있습니다. 콘솔에서 새 토큰 발급.
  2. 호스트 커널 모듈 불일치: Enforcer는 호스트 커널 모듈(eBPF)을 로드합니다. 호스트 OS(특히 Amazon Linux 2)와 Enforcer 버전이 호환되는지 확인. uname -r로 커널 버전 확인.
  3. 리소스 부족: Enforcer에 충분한 CPU/Memory 할당 확인. DaemonSet 리소스 리미트 증가.

팁: CI/CD 파이프라인에 통합하기

GitLab CI, GitHub Actions, Jenkins에서 Aqua Scanner를 실행하여 "Shift Left" 보안을 구현하세요.

GitHub Actions 예제 (.github/workflows/aqua-scan.yml)
name: Aqua Security Scan
on: [push]
jobs:
  scan:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - name: Build Docker image
        run: docker build -t myapp:${{ github.sha }} .
      - name: Run Aqua CLI Scan
        run: |
          curl -sL https://download.aquasec.com/aquactl/stable/aquactl-linux-amd64 -o aquactl
          chmod +x aquactl
          ./aquactl scan image myapp:${{ github.sha }} \
            --registry local \
            --user ${{ secrets.AQUA_USER }} \
            --password ${{ secrets.AQUA_PASSWORD }} \
            --threshold HIGH
        # --threshold HIGH 옵션: HIGH 이상 취약점 발견 시 스텝 실패

문제: 너무 많은 False Positive 알림

해결:

  • 정책 조정: Console에서 Runtime Policy를 세부적으로 조정. 예: 특정 경로(/tmp)의 파일 변경은 허용, 특정 프로세스(예: apt-get) 실행은 허용.
  • Exception Rule 생성: 특정 이미지, 레이블, 네임스페이스에 대해 알림/차단 규칙을 예외 처리.
  • 학습 모드 활용: 새 워크로드 배포 시 처음 몇 시간/일은 "Audit" 모드로 실행하여 정상적인 행동 패턴을 학습시킨 후 "Enforce" 모드로 전환.

문제: 스캔 속도가 너무 느림

해결:

  • Scanner 리소스 증가: Scanner 배포의 CPU/Memory 리미트를 높입니다.
  • 병렬 스캔 설정: Helm 값 scanner.replicaCount 및 scanner.concurrentScans를 증가시킵니다.
  • 오프라인 데이터베이스 캐시: Self-Hosted 환경에서는 Aqua Gateway를 통해 취약점 DB를 로컬에 캐싱하여 네트워크 지연을 줄입니다.
