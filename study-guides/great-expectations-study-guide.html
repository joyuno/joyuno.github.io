1. Great Expectations란?

Great Expectations(GX)는 데이터 품질 검증(Data Validation)을 위한 오픈소스 Python 프레임워크입니다. 데이터 파이프라인의 각 단계에서 데이터가 기대한 대로 정확하고 완전하며 일관성을 유지하는지 테스트할 수 있습니다.

📋 Expectation 정의

데이터에 대한 검증 규칙을 선언적으로 정의합니다. (예: "이 컬럼은 NULL이 아니어야 함", "값은 0에서 100 사이여야 함")

🔍 데이터 검증 실행

Expectation을 실제 데이터(DataFrame, SQL 테이블 등)에 대해 실행하고 결과를 생성합니다.

📊 결과 문서화

검증 결과를 HTML 리포트, Data Docs로 자동 생성하여 팀과 공유할 수 있습니다.

핵심 가치

데이터에 대한 신뢰(Trust)를 구축하고, 데이터 문제를 프로덕션 전에 조기에 발견하여 비용을 절감합니다.

2. 왜 써야 하는가?

기존 데이터 품질 검증 방식과의 비교

방식 장점 단점 GX 대비
수동 스크립트 유연함, 빠른 구현 유지보수 어려움, 문서화 부재, 재사용성 낮음 표준화된 프레임워크, 자동 문서화
단위 테스트(pytest) 익숙한 도구, CI/CD 통합 용이 데이터 특화 기능 부족, 대용량 데이터 비효율 데이터 최적화, 풍부한 Expectation 라이브러리
SQL CHECK 제약조건 데이터베이스 내장, 빠름 제한된 로직, 이관 어려움, 복잡한 조건 표현 불가 복잡한 비즈니스 로직, 다중 데이터 소스 지원
Great Expectations 표준화, 자동 문서화, 풍부한 커뮤니티, 확장성 학습 곡선, 초기 설정 필요 -

3. 핵심 개념

GX의 주요 구성 요소와 데이터 흐름

Data Source (Pandas, Spark, SQL) │ ▼ +-----------------------------+ │ Datasource │ │ (데이터 연결 구성) │ +-----------------------------+ │ ▼ +-----------------------------+ │ Batch Request │ │ (검증할 데이터 샘플 지정)│ +-----------------------------+ │ ▼ +-----------------------------+ │ Expectation Suite │←---(정의)--- Expectation │ (검증 규칙 집합) │ (단일 규칙) +-----------------------------+ │ ▼ +-----------------------------+ │ Checkpoint │ │ (실행 주기/트리거 정의) │ +-----------------------------+ │ ▼ +-----------------------------+ │ Validation Result │ │ (실행 결과) │ +-----------------------------+ │ ▼ +-----------------------------+ │ Data Docs (HTML) │ │ (자동 생성 리포트) │ +-----------------------------+

주요 용어

  • Expectation: "컬럼 A는 고유해야 한다"와 같은 단일 검증 규칙.
  • Expectation Suite: 여러 Expectation의 집합. 하나의 데이터 자산(테이블)에 대한 모든 검증 규칙.
  • Validator: Expectation을 데이터에 실행하는 객체.
  • Checkpoint: Datasource, Expectation Suite, Action(예: 리포트 생성)을 묶어 실행하는 단위.
  • Data Docs: 검증 결과를 시각화한 HTML 리포트.

4. 퀵스타트

단계 1: 설치

CLI / Terminal
# 기본 설치
pip install great_expectations

# 추가 기능 (예: Spark, Snowflake)
pip install "great_expectations[spark, snowflake]"

단계 2: 프로젝트 초기화

CLI / Terminal
# GX 프로젝트 구조 생성
great_expectations init

이 명령어는 great_expectations/ 디렉토리를 생성하며, 설정 파일과 기본 폴더 구조를 만듭니다.

단계 3: 첫 Expectation 생성 및 검증 (Python)

Python Script (example.py)
import great_expectations as gx
import pandas as pd

# 1. 컨텍스트 생성 (GX 진입점)
context = gx.get_context()

# 2. 샘플 데이터 생성
df = pd.DataFrame({
    "user_id": [1, 2, 3, 4, 5],
    "email": ["a@test.com", "b@test.com", "c@test.com", "d@test.com", "e@test.com"],
    "age": [25, 31, 45, 22, 38],
    "signup_date": pd.date_range("2024-01-01", periods=5)
})

# 3. Datasource 추가 (Pandas)
datasource = context.sources.add_pandas(name="my_pandas_datasource")

# 4. Data Asset (테이블) 정의
data_asset = datasource.add_dataframe_asset(name="users", dataframe=df)

# 5. Batch Request 생성 (검증할 데이터 지정)
batch_request = data_asset.build_batch_request()

# 6. Expectation Suite 생성 및 규칙 추가
expectation_suite_name = "users_suite"
suite = context.add_expectation_suite(expectation_suite_name=expectation_suite_name)

# Validator를 통해 Expectation 정의
validator = context.get_validator(
    batch_request=batch_request,
    expectation_suite_name=expectation_suite_name
)

# Expectation 규칙 추가
validator.expect_column_values_to_not_be_null(column="user_id")
validator.expect_column_values_to_be_between(column="age", min_value=18, max_value=100)
validator.expect_column_values_to_be_unique(column="email")
validator.expect_column_values_to_match_regex(column="email", regex=r"^[^@]+@[^@]+\.[^@]+$")

# Suite 저장
validator.save_expectation_suite(discard_failed_expectations=False)

# 7. Checkpoint 생성 및 실행
checkpoint = context.add_or_update_checkpoint(
    name="my_first_checkpoint",
    validations=[
        {
            "batch_request": batch_request,
            "expectation_suite_name": expectation_suite_name,
        }
    ],
)
checkpoint_result = checkpoint.run()

# 8. 결과 확인
print(f"성공: {checkpoint_result['success']}")
print(f"결과 리포트: great_expectations/uncommitted/data_docs/local_site/index.html")

실행 후 확인

CLI에서 great_expectations docs build를 실행하거나, Python에서 context.build_data_docs()를 호출하면 Data Docs가 생성됩니다. 브라우저로 great_expectations/uncommitted/data_docs/local_site/index.html 파일을 열어 시각화된 리포트를 확인하세요.

5. 핵심 설정 레퍼런스

구성 요소 설정 파일 주요 설정 항목 설명
Data Context great_expectations.yml plugins_directory, stores, data_docs_sites, checkpoint_store_name GX 프로젝트의 루트 설정. 저장소(Store)와 데이터 문서 사이트를 정의.
Datasource great_expectations/datasources/ name, class_name, connection_string, credentials 데이터베이스, 데이터레이크, 파일시스템 등 데이터 소스 연결 정보.
Expectation Suite great_expectations/expectations/<suite_name>.json expectation_type, kwargs, meta 검증 규칙의 집합. JSON 형식으로 저장됨.
Checkpoint great_expectations/checkpoints/<checkpoint_name>.yml name, validations, action_list 어떤 데이터를, 어떤 규칙으로, 실행 후 어떤 액션(이메일, 리포트)을 취할지 정의.
Data Docs great_expectations.yml 내 data_docs_sites site_name, class_name, store_backend, site_index_builder HTML 리포트 생성 위치와 방식을 설정 (로컬 파일, S3, GCS 등).

6. 실무 팁 & 트러블슈팅

🚀 성능 최적화

문제: 대용량 데이터 검증 속도가 느림.
해결:

  • 샘플링 사용: batch_request에 sampling_method 적용.
  • Spark/Dask 활용: 분산 처리 백엔드로 전환.
  • Expectation 최소화: 비용 높은 Expectation(예: expect_column_pair_values_to_be_equal)을 피하거나 샘플로만 실행.

🔗 CI/CD 통합

팁: 파이프라인 단계별 Checkpoint를 설정하고, 실패 시 CI 작업을 중단하도록 구성.
예시 (GitHub Actions): 

- name: Run Data Tests
  run: python -m pytest tests/data_validation/ --tb=short
GX 결과를 pytest와 연동하여 사용 가능.

💾 저장소 관리

문제: Expectation Suite, Validation Result가 로컬에만 저장됨.
해결: great_expectations.yml에서 Store 백엔드를 원격 저장소(S3, GCS, Azure Blob)로 변경하여 팀원과 공유 및 버전 관리.

🐛 일반적인 오류

"ModuleNotFoundError: No module named 'great_expectations'"
→ 가상환경 활성화 및 패키지 재설치 확인.

"BatchRequest is not valid for this context"
→ Datasource 또는 Data Asset 이름이 정확한지 context.datasources로 확인.

Data Docs가 생성되지 않음
→ context.build_data_docs() 명시적 호출 또는 checkpoint.run() 후 great_expectations docs build CLI 실행.

주의사항

프로덕션 환경에서는 절대 great_expectations/uncommitted/ 디렉토리를 버전 관리에 포함시키지 마세요. 이 디렉토리는 로컬 캐시와 임시 파일을 포함합니다. .gitignore에 추가 필수.
