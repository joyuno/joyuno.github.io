1. DNSSEC란?

DNSSEC(Domain Name System Security Extensions)는 DNS 응답의 무결성과 인증을 제공하는 보안 프로토콜 확장입니다. 전통적인 DNS는 스푸핑, 캐시 포이즈닝 등 중간자 공격에 취약했으나, DNSSEC은 디지털 서명을 통해 응답의 진위를 검증할 수 있게 합니다.

데이터 무결성

DNS 레코드가 전송 중 변조되지 않았음을 보장합니다. RRSIG(Resource Record Signature) 레코드를 통해 검증됩니다.

데이터 출처 인증

응답이 실제 권한 있는 네임서버에서 왔음을 증명합니다. DNSKEY와 DS(Delegation Signer) 레코드 체인을 통해 구현됩니다.

재생 공격 방지

서명된 레코드에 포함된 만료 시간(Inception/Expiration)을 통해 오래된 응답의 재사용을 차단합니다.

핵심: DNSSEC은 암호화(기밀성)를 제공하지 않습니다. 오직 인증과 무결성만을 담보합니다.

2. 왜 써야 하는가?

기존 DNS 대비 DNSSEC의 보안적 가치와 운영상 고려사항을 비교합니다.

항목 전통적 DNS DNSSEC 실무적 영향
응답 인증 없음 (UDP 스푸핑 가능) 강력한 공개키 기반 인증 피싱/스푸핑 공격 차단
데이터 무결성 보장되지 않음 디지털 서명으로 보장 캐시 포이즈닝 방지
성능 오버헤드 낮음 중간 (서명 검증, 추가 레코드) 리졸버 CPU 사용량 증가
설정 복잡도 단순 고도화 (키 관리, 롤오버) 자동화 도구 필요 (e.g., OpenDNSSEC)
호환성 범용 리졸버/서버 지원 필요 구형 장비에서 문제 가능

3. 핵심 개념

DNSSEC의 동작을 이해하기 위한 키 레코드와 아키텍처 흐름입니다.

Client (Stub Resolver) | v Recursive Resolver (with DNSSEC Validation) | |--- Query: example.com A |--- RD=1, DO=1 (DNSSEC OK bit set) | v Root Server | |--- Respond with .com NS + DS (Delegation Signer) |--- RRSIG for DS (signed by Root KSK) | v .com TLD Server | |--- Respond with example.com NS + DS |--- RRSIG for DS (signed by .com KSK) | v example.com Authoritative Server | |--- Respond with: | * A record (192.0.2.1) | * RRSIG for A (signed by ZSK) | * DNSKEY (ZSK public key) | * RRSIG for DNSKEY (signed by KSK) | v Resolver Validation Chain: 1. Fetch DNSKEY (ZSK) from answer. 2. Use ZSK to verify RRSIG of A record. 3. Fetch parent's DS hash of child's KSK. 4. Verify KSK's hash matches DS. 5. Chain up to Root Trust Anchor.
  • KSK (Key Signing Key): ZSK를 서명하는 장기간 마스터 키. DS 레코드로 부모 도메인에 등록.
  • ZSK (Zone Signing Key): 실제 RRset(A, MX 등)을 서명하는 단기 운영 키.
  • DS (Delegation Signer): 자식 영역의 KSK 공개키 해시를 부모 영역에 등록하는 레코드. 신뢰 체인 형성.
  • RRSIG (Resource Record Signature): RRset에 대한 디지털 서명 레코드.
  • Trust Anchor: 검증의 시작점 (일반적으로 Root KSK). 리졸버에 하드코딩 또는 RFC 5011로 업데이트.

4. 퀵스타트 (BIND9 기반)

Step 1: BIND9 설치 및 기본 영역 생성

# Ubuntu/Debian
sudo apt update
sudo apt install bind9 bind9utils dnsutils -y

# /etc/bind/named.conf.local 에 영역 추가
zone "lab.example" {
    type master;
    file "/etc/bind/db.lab.example";
    allow-transfer { none; };
    allow-query { any; };
    dnssec-policy default;  # BIND 9.16+ DNSSEC 자동 정책
};

Step 2: 영역 파일 작성 및 서명

# /etc/bind/db.lab.example
$TTL 3600
@ IN SOA ns1.lab.example. admin.lab.example. (
    2026030101 ; Serial
    7200       ; Refresh
    3600       ; Retry
    1209600    ; Expire
    3600 )     ; Negative TTL

@        IN NS   ns1.lab.example.
ns1      IN A    10.0.1.10
www      IN A    10.0.1.100
api      IN A    10.0.1.101
_dmarc   IN TXT  "v=DMARC1; p=none;"

# 키 생성 및 영역 서명 (수동 방식 예시)
sudo dnssec-keygen -a RSASHA256 -b 2048 -n ZONE lab.example  # ZSK 생성
sudo dnssec-keygen -a RSASHA256 -b 2048 -f KSK -n ZONE lab.example  # KSK 생성
sudo dnssec-signzone -S -z -o lab.example -N INCREMENT /etc/bind/db.lab.example

Step 3: 리졸버 검증 테스트 (Python 예제)

import dns.resolver
import dns.dnssec
import dns.message
import dns.query

def validate_dnssec(domain, rdtype='A'):
    resolver = dns.resolver.Resolver()
    resolver.use_edns(0, 0, flags=dns.flags.DO)  # DNSSEC OK bit 설정

    try:
        answer = resolver.resolve(domain, rdtype, raise_on_no_answer=False)
        print(f"[*] Query: {domain} {rdtype}")
        print(f"[*] Response: {answer.rrset}")

        # AD (Authentic Data) 플래그 확인
        if answer.response.flags & dns.flags.AD:
            print("[+] DNSSEC 검증 성공 (AD flag set)")
        else:
            print("[-] DNSSEC 검증 실패 또는 미서명 영역")
            # 실제 검증 로직은 dns.dnssec.validate()로 구현 가능

    except dns.resolver.NXDOMAIN:
        print("[-] NXDOMAIN")
    except Exception as e:
        print(f"[!] 오류: {e}")

if __name__ == "__main__":
    validate_dnssec("www.lab.example")

Step 4: CLI 검증 명령어

# dig로 DNSSEC 응답 확인
dig +dnssec www.lab.example A

# 응답에 'ad' 플래그와 RRSIG, DNSKEY 레코드 확인
# ad flag 존재 시 검증 성공

# DNSKEY 레코드 확인
dig DNSKEY lab.example +multiline

# 특정 레코드의 RRSIG 확인
dig RRSIG www.lab.example A +multiline

# 트러블슈팅: +cd 플래그로 검증 비활성화
dig +cd +dnssec www.lab.example

6. 실무 팁 & 트러블슈팅

주의: 키 롤오버는 사전 계획과 테스트 없이 수행 시 도메인 해결 불가를 초래할 수 있습니다.

자주 겪는 문제

  • SERVFAIL (DNSSEC 검증 실패):
    • 원인: 부모 DS 레코드와 자식 DNSKEY 불일치, 서명 만료, 신뢰 앵커 문제.
    • 해결: dig +cd로 검증 비활성화 시 응답 확인. delv 도구로 상세 검증 경로 추적.
  • 부모 등록기관에 DS 전파 지연:
  • 원인: DS 레코드 제출 후 TTL 및 캐시로 인한 지연 (최대 48시간).
  • 해결: 롤오버 시 새 키 활성화 전 미리 DS를 제출하고 이전 키를 유지하는 Pre-Publish 방식 사용.
  • EDNS0 버퍼 크기 문제:
  • 원인: DNSSEC 응답은 크기가 커서 전통적인 512바이트 UDP 응답 초과.
  • 해결: 리졸버와 파이어월이 EDNS0(Extension Mechanisms for DNS) 및 큰 UDP 패킷을 허용하는지 확인.
  • Python dnspython으로 AD 플래그 확인 불가:
  • 원인: 대부분의 스텁 리졸버는 AD 플래그를 지우거나 전달하지 않음.
  • 해결: 재귀 리졸버(예: 8.8.8.8)에 직접 질의하거나, dns.dnssec.validate()를 사용해 직접 검증 로직 구현.
