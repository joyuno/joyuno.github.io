1. DNSSEC란?

DNSSEC(Domain Name System Security Extensions)은 DNS 응답에 암호학적 서명을 추가하여 데이터의 무결성과 출처 인증을 보장하는 프로토콜입니다. 기존 DNS는 응답을 위조하거나 변조할 수 있는 취약점이 있지만, DNSSEC은 공개키 기반 서명으로 이를 방어합니다.

3대 핵심 능력

🔏 데이터 원본 인증

DNS 응답이 진짜 권한 있는 서버에서 왔는지 검증합니다. (Origin Authentication)

🧩 데이터 무결성

전송 중 데이터가 변경되지 않았음을 보장합니다. (Data Integrity)

⛓️ 신뢰 체인

루트 존부터 최종 도메인까지 계층적으로 서명을 연결하여 신뢰를 형성합니다. (Chain of Trust)

🛡️ 무결성 🔑 인증 🔗 체인

2. 왜 써야 하는가?

DNSSEC이 없다면 DNS 응답은 중간자 공격, 캐시 포이즈닝, 응답 위조에 취약합니다. 아래 비교표를 통해 기존 DNS와 DNSSEC의 차이를 확인하세요.

공격 / 위협 일반 DNS DNSSEC
DNS 캐시 포이즈닝 ❌ 취약 ✅ 서명 검증으로 차단
중간자 공격 (MiTM) ❌ 취약 ✅ 무결성 보장
응답 변조 / 위조 ❌ 취약 ✅ 서명 불일치 시 거부
데이터 출처 확인 ❌ 불가능 ✅ 권한 서버 인증
운영 복잡성 🟢 낮음 🟡 중간 (키 관리 필요)
💡 보안 전문가의 조언: DNSSEC은 완벽한 개인정보 보호를 제공하지는 않지만(암호화 없음), 데이터 무결성과 인증이 중요한 환경에서는 필수입니다. 특히 TLS 인증서 발급, 소프트웨어 업데이트, 금융 트랜잭션 도메인에서 강력히 권장됩니다.

3. 핵심 개념

DNSSEC은 DNS 계층 구조에 서명 레코드(RRSIG)와 공개키 레코드(DNSKEY)를 추가하여 동작합니다. 아래 다이어그램은 리졸버가 DNSSEC으로 검증된 응답을 받는 과정을 보여줍니다.

┌─────────────────────────────────────────────────────┐ │ DNSSEC 아키텍처 │ ├─────────────────────────────────────────────────────┤ │ │ │ 사용자 ──→ 리졸버 ──→ 루트 존 (DS, DNSKEY) │ │ │ │ │ ▼ │ │ TLD 존 (DS, DNSKEY) │ │ │ │ │ ▼ │ │ 권한 있는 네임서버 (RRSIG, DNSKEY) │ │ │ │ │ ▼ │ │ 서명된 응답 반환 │ │ │ │ * RRSIG : 리소스 레코드 서명 │ │ * DNSKEY: 존 서명 키 │ │ * DS : 상위 존에 위임된 서명 다이제스트 │ └─────────────────────────────────────────────────────┘

핵심 레코드 타입:

  • DNSKEY — 존의 공개키
  • RRSIG — 각 레코드 세트의 디지털 서명
  • DS — 상위 존에 등록된 하위 존의 키 해시 (신뢰 앵커)
  • NSEC / NSEC3 — 존재하지 않는 레코드에 대한 인증 (부정 응답 서명)
⚠️ 주의: DNSSEC은 데이터 무결성만 보장하며 기밀성(암호화)은 제공하지 않습니다. 개인정보 보호가 필요하면 DNS-over-TLS(DOT) 또는 DNS-over-HTTPS(DOH)와 함께 사용해야 합니다.

4. 퀵스타트

Python과 CLI를 이용해 DNSSEC을 빠르게 체험해봅니다. 아래 단계를 따라주세요.

① 설치

Python (dnspython)

pip install dnspython

CLI 도구 (dig, dnssec-keygen 등) — BIND 유틸리티

# Ubuntu/Debian
sudo apt-get install bind9-utils

# macOS (Homebrew)
brew install bind

② Python으로 DNSSEC 검증 예제

import dns.resolver
import dns.dnssec

domain = "example.com"
try:
    answer = dns.resolver.resolve(domain, "A", raise_on_no_answer=False)
    # RRSIG 확인
    if answer.response and answer.response.answer:
        for rrset in answer.response.answer:
            if rrset.rdtype == dns.rdatatype.RRSIG:
                print(f"✅ RRSIG 발견: {rrset}")
            else:
                print(f"📦 레코드: {rrset}")
    print(f"✓ {domain} resolved successfully")
except dns.dnssec.ValidationFailure as e:
    print(f"❌ DNSSEC 검증 실패: {e}")
except Exception as e:
    print(f"⚠️ 오류: {e}")

③ CLI 명령어 (dig +dnssec)

# DNSSEC 서명 확인
dig example.com +dnssec

# DNSKEY 레코드 조회
dig example.com DNSKEY

# DS 레코드 조회 (상위 존 위임)
dig example.com DS
💡 팁: dig +dnssec +multiline 옵션을 사용하면 RRSIG와 DNSKEY를 사람이 읽기 쉬운 형태로 볼 수 있습니다.

5. 핵심 설정 레퍼런스

BIND(naemd) 기준 DNSSEC 설정 및 운영에 필요한 주요 명령어와 설정값을 표로 정리했습니다.

명령어 / 설정 설명 예시
dnssec-keygen ZSK(Zone Signing Key) 또는 KSK(Key Signing Key) 생성 dnssec-keygen -a ECDSAP256SHA256 -b 256 -n ZONE example.com
dnssec-signzone 존 파일에 서명하여 서명된 존 파일 생성 dnssec-signzone -o example.com -k Kexample.com.+013+12345.key db.example.com
dnssec-verify 서명된 존의 유효성 검증 dnssec-verify -o example.com db.example.com.signed
named.conf BIND 설정에서 DNSSEC 활성화 dnssec-enable yes; dnssec-validation yes;
auto-dnssec 존 자동 서명 (maintain / offline) auto-dnssec maintain; in zone config
rndc reconfig 설정 변경 후 적용 rndc reconfig

6. 실무 팁 & 트러블슈팅

DNSSEC 운영 중 자주 마주치는 문제와 해결 방법입니다.

🔴 문제 1: SERVFAIL 응답 (DNSSEC 검증 실패)
원인: 신뢰 앵커가 잘못되었거나, 상위 존 DS 레코드 불일치, 키 만료.
해결: dig +dnssec +cd (체인 끊기)로 원본 응답 확인 후, dnssec-verify로 존 점검. 신뢰 앵커 업데이트(rndc secroots).
🔴 문제 2: 키 만료 (RRSIG expiration)
원인: 서명 유효기간이 지남.
해결: dnssec-signzone -t로 재서명하거나 auto-dnssec maintain로 자동 갱신. 키 롤오버 일정 수립.
🔴 문제 3: 체인 오브 트러스트 단절 (Broken chain)
원인: 부모 존 DS 레코드가 자식 존 DNSKEY와 일치하지 않음.
해결: DS 레코드를 부모 존에 올바르게 등록했는지 확인. dig +trace +dnssec로 체인 추적.
✅ 실무 팁
  • KSK와 ZSK를 분리하고, KSK는 오프라인 보관.
  • 키 롤오버 전에 dnssec-settime로 사전 게시.
  • 모니터링: dnssec-check 또는 Zonemaster로 주기적 검증.
  • 로그: BIND에서 category dnssec; 로그 활성화.
