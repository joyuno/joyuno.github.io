1. Volatility란?

Volatility는 오픈 소스 메모리 포렌식 프레임워크로, RAM 덤프 파일에서 디지털 아티팩트를 추출하고 분석하는 데 사용됩니다. 휘발성 메모리의 상태를 정지된 시점에서 조사하여 실행 중인 프로세스, 네트워크 연결, 레지스트리 핸들, 암호화된 키, 루트킷 흔적 등을 발견할 수 있습니다.

메모리 아키텍처 추상화

Windows, Linux, macOS, Android 등 다양한 OS의 메모리 구조를 이해하고, 커널 데이터 구조를 파싱하여 일관된 인터페이스로 제공합니다. 프로필(Profile) 시스템을 통해 OS별 특성을 처리합니다.

플러그인 기반 확장성

200개 이상의 공식 플러그인과 사용자 정의 플러그인을 통해 분석 기능을 무한히 확장할 수 있습니다. Python으로 쉽게 새로운 분석 모듈을 개발하여 통합할 수 있습니다.

자동화 및 API 제공

CLI 도구뿐만 아니라 풍부한 Python API를 제공하여 대규모 인시던트 대응 자동화 파이프라인에 통합할 수 있습니다. AI 엔지니어는 API를 활용해 분석 로직을 학습 데이터로 활용할 수 있습니다.

2. 왜 써야 하는가?

디스크 포렌식만으로는 잡히지 않는 휘발성 증거를 확보할 수 있어 현대 보안 위협(파일리스 멀웨어, 메모리 상주 공격)에 필수적입니다. 기존 대안과의 비교는 다음과 같습니다.

도구/접근법 장점 단점 Volatility 대비
커맨드라인 네이티브 도구
(tasklist, netstat, ps)		 실시간 정보, 시스템에 내장 공격자에 의해 변조 가능, 히스토리 없음, 깊은 분석 불가 변조되지 않은 정지된 상태의 메모리 분석 가능
상용 메모리 분석 도구
(Rekall, Belkasoft)		 GUI, 통합 보고서 비용, 확장성 제한, 스크립팅 제약 무료 오픈소스, 풀 스크립팅 및 API 지원
수동 메모리 파싱 완전한 제어, 맞춤형 분석 시간 소모적, 오류 가능성 높음, 전문성 요구 검증된 프레임워크와 풍부한 플러그인으로 생산성 향상
EDR/시그니처 기반 검사 실시간 차단, 알려진 위협 탐지 알려지지 않은(Zero-day) 공격 탐지 불가, 증거 보존 미흡 시그니처 없이도 행위 기반 이상징후 탐지 가능

AI 엔지니어에게 주는 가치

Volatility는 메모리 스냅샷을 구조화된 데이터(프로세스 목록, DLL, 핸들 테이블 등)로 변환하는 강력한 엔진입니다. 이 데이터는 악성 행위 패턴 학습, 이상 탐지 모델 훈련, 자동화된 인시던트 트라이지 파이프라인 구축에 직접 활용될 수 있습니다.

3. 핵심 개념

Volatility의 동작 원리는 크게 세 단계로 나눌 수 있습니다: 메모리 이미지 로딩 → 적절한 프로필 식별 및 적용 → 플러그인 실행을 통한 계층적 분석.

+-----------------------+ | Memory Image File | | (raw, crashdump, etc)| +----------+------------+ | v +----------+------------+ | Address Space Layer | | (물리 메모리 가상화) | +----------+------------+ | v +----------+------------+ +-------------------+ | OS Profile Layer +------> Symbol Tables | | (Windows10x64, etc) | | (PDB, DWARF) | +----------+------------+ +-------------------+ | v +----------+------------+ | Plugin Subsystem | | (pslist, netscan, | | malfind, handles) | +----------+------------+ | v +----------+------------+ | Structured Output | | (JSON, text, CSV) | +-----------------------+

주요 용어

  • 프로필(Profile): 특정 OS 버전(예: Win10x64_18362)의 메모리 구조를 정의한 컬렉션. 커널 심볼과 타입 정보를 포함합니다.
  • 주소 공간(Address Space): 물리 메모리, 페이징 파일, 하이버네이트 파일 등 다양한 메모리 소스를 일관된 방식으로 접근할 수 있게 하는 추상화 계층.
  • 플러그인(Plugin): 특정 분석 작업(예: 프로세스 리스트 출력)을 수행하는 모듈. volatility3에서는 "클래스"로 구현됩니다.
  • 심볼 테이블(Symbol Table): 커널 데이터 구조체의 오프셋과 레이아웃 정보. 정확한 분석의 핵심입니다.

4. 퀵스타트

단계 1: 설치 (Python 환경)

pip install volatility3

# 추가 유용한 라이브러리
pip install yara-python pandas jupyter  # 분석 및 자동화용

단계 2: 첫 번째 명령어 실행 (CLI)

메모리 이미지(memory.dmp)가 있다고 가정하고 Windows 프로세스 목록을 확인합니다.

# 기본 사용법: volatility -f [메모리파일] [플러그인]
volatility -f /path/to/memory.dmp windows.pslist

# 더 자세한 정보 (PID, PPID, 생성 시간 등)
volatility -f memory.dmp windows.psscan

# 출력을 JSON으로 받아 자동화 파이프라인에 연결
volatility -f memory.dmp windows.pslist --output json > processes.json

단계 3: Python API를 이용한 스크립팅

#!/usr/bin/env python3
"""Volatility3 Python API 예제: 프로세스 및 DLL 목록 추출"""

import volatility3.framework
from volatility3.framework import contexts, automagic, constants
from volatility3.plugins.windows import pslist, dlllist
import json
import sys

def analyze_memory(image_path):
    # 1. 컨텍스트 생성
    ctx = contexts.Context()
    ctx.config['automagic.LayerStacker.single_location'] = f"file://{image_path}"

    # 2. 자동 마법 구성 실행 (프로필 자동 감지)
    automagics = automagic.choose_automagic(automagic.available(ctx))
    for amagic in automagics:
        amagic(ctx)

    # 3. 플러그인 실행: 프로세스 목록
    pslist_plugin = pslist.PsList(ctx)
    process_tree = []
    for proc in pslist_plugin.generate_configuration():
        process_info = {
            "pid": proc.UniqueProcessId,
            "name": proc.ImageFileName.cast("string", max_length=255),
            "ppid": proc.InheritedFromUniqueProcessId,
            "create_time": proc.get_create_time()
        }
        process_tree.append(process_info)

    # 4. 플러그인 실행: 각 프로세스의 DLL 목록
    dlllist_plugin = dlllist.DllList(ctx)
    dll_map = {}
    # (실제 구현에서는 프로세스별로 dlllist.generate_configuration 호출)

    # 결과 출력
    result = {
        "image_file": image_path,
        "process_count": len(process_tree),
        "processes": process_tree
    }
    print(json.dumps(result, indent=2, default=str))

if __name__ == "__main__":
    if len(sys.argv) != 2:
        print(f"Usage: {sys.argv[0]} <memory_image_path>")
        sys.exit(1)
    analyze_memory(sys.argv[1])

주의: 프로필 자동 감지 실패 시

자동 감지가 실패하면 --profile 옵션으로 명시적으로 지정해야 합니다. 사용 가능한 프로필은 volatility --info로 확인할 수 있습니다. Linux의 경우 linux.banner 플러그인으로 커널 버전을 먼저 확인하는 것이 좋습니다.

5. 핵심 설정 레퍼런스

설정 옵션 기본값 설명 사용 예시 (CLI)
-f, --filename 없음 분석할 메모리 이미지 파일 경로 -f /evidence/mem.dmp
--profile 자동 감지 OS 프로필 강제 지정 (Win10x64, LinuxUbuntu2004x64 등) --profile=Win10x64_19041
-o, --output text 출력 형식 (text, json, csv, pretty) --output json
--output-file stdout 출력을 파일로 저장 --output-file=result.json
-v, --verbose 0 로그 상세도 (0=에러, 1=경고, 2=정보, 3=디버그) -v 2
--plugins 설치된 경로 사용자 정의 플러그인 경로 추가 --plugins=/my_plugins
--cache ~/.cache/volatility3 심볼 캐시 디렉토리 경로 --cache=/tmp/vol_cache

Python API 주요 설정 경로: ctx.config['PLUGIN_NAME.OPTION'] 형식으로 접근합니다. 예: ctx.config['automagic.LayerStacker.single_location'] = "file:///path/to/image"

6. 실무 팁 & 트러블슈팅

문제: "No suitable address space mapping found"

원인: 메모리 이미지 형식을 인식하지 못함. 또는 이미지가 손상되었을 수 있습니다.

해결:

  1. file 명령어로 이미지 형식 확인.
  2. --layer-format 옵션으로 명시적 지정 (예: --layer-format=dmp).
  3. DumpIt, WinPMEM 등 신뢰된 도구로 메모리 덤프를 다시 획득.

문제: 프로필을 찾을 수 없음

원인: 분석 대상 OS에 대한 심볼 테이블(프로필)이 로컬 캐시에 없습니다.

해결:

  1. 인터넷이 연결된 상태에서 명령어를 실행하면 자동 다운로드 시도.
  2. 수동 다운로드: volatility --os <OS> --output-dir ./symbols
  3. Windows의 경우, 해당 OS 버전의 ntoskrnl.pdb 파일을 찾아 지정.

문제: 플러그인 실행 시 비정상 종료

원인: 메모리 이미지와 프로필 불일치, 또는 플러그인 내부 버그.

해결:

  1. linux.banner 또는 windows.info로 정확한 OS 버전 재확인.
  2. -v 3 (디버그 모드)로 실행하여 스택 트레이스 확인.
  3. 문제가 되는 특정 플러그인을 제외하고 다른 플러그인으로 우회 분석.

고급 실무 팁

  • 대규모 자동화: Python API를 사용하여 수백 개의 메모리 이미지를 배치 처리하고, 결과를 중앙 ELK 스택이나 SIEM으로 전송하세요.
  • AI/ML 통합: windows.pslist, windows.netscan, windows.handles의 출력을 정규화하여 프로세스 행위 그래프를 구성하고, 이상 탐지 모델의 입력 피처로 사용하세요.
  • 성능 최적화: 자주 사용하는 프로필은 로컬 캐시에 유지하고, 대용량 이미지 분석 시 --quick 모드(있는 경우)를 고려하세요.
  • 커스텀 플러그인 개발: 신규 멀웨어 패밀리의 아티팩트를 추적하려면 공식 문서의 플러그인 개발 가이드를 따라 Python 클래스를 작성하세요.
